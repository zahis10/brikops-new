## 3. Startup Guard Behavior

`backend/config.py` enforces the following on import (before any route loads):

| Variable | Requirement | On Missing |
|----------|-------------|------------|
| `APP_ID` | Non-empty | `sys.exit(1)` with clear error |
| `APP_MODE` | Must be `dev` or `prod` | `sys.exit(1)` with clear error |
| `JWT_SECRET` | Min 32 characters | `sys.exit(1)` with clear error |
| `JWT_SECRET_VERSION` | Non-empty | `sys.exit(1)` with clear error |
| `MONGO_URL` | Non-empty | `sys.exit(1)` with clear error |
| `DB_NAME` | Non-empty | `sys.exit(1)` with clear error |

On successful startup, a sanitized log is emitted showing APP_ID, APP_MODE, DB_NAME — **never** showing secret values (only last 4 chars of JWT_SECRET).

## 4. JWT Issuer Enforcement

| Feature | Status |
|---------|--------|
| `iss` claim set to `APP_ID` on token creation | Enforced |
| `iss` validated on every token verification | Enforced |
| Tokens with wrong/missing `iss` | **Rejected (401)** |
| `secret_version` claim included in token | Enforced |
| Tokens with old/unknown `secret_version` | **Rejected (401)** |
| Algorithm restricted to `HS256` only | Enforced |
| `exp`, `iat`, `iss` claims required | Enforced |
| Clock skew tolerance | 60 seconds max |

### Secret Rotation Procedure

To rotate JWT secrets safely:

1. Set `JWT_SECRET` to new value (min 32 chars)
2. Increment `JWT_SECRET_VERSION` (e.g., `v1` → `v2`)
3. Restart the application
4. All existing tokens with old version are automatically rejected
5. Users must re-login to get new tokens with the updated secret version
