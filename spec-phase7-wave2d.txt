# #391 — Phase 7 Wave 2d: Logo color + Capgo autoUpdate + native-change guard

## What & Why
After Wave 2c (the diagnostic build tag) shipped successfully, two things
are now clear from Zahi's iPhone + app.brikops.com screenshot:

1. **Logo is white where it should be orange.** On `app.brikops.com` the
   D2 daily splash renders the BrikMark "B" in white (`#fff`) instead of
   the brand orange (`#F59E0B`). The accent + "Ops" text are already
   orange — only the main mark is wrong.

2. **Capgo OTA is silently disabled on the installed iOS app.** Root
   cause confirmed: `frontend/capacitor.config.json` has never contained
   a `plugins.CapacitorUpdater` block. In Capgo v7, without an explicit
   `autoUpdate: true`, the plugin does NOT check for updates on app
   resume. Result: Zahi's iPhone is stuck on bundle 1.0.11, ignoring
   every bundle we've uploaded since (1.0.12–1.0.23).

This wave does three things in a single commit cycle:

- (A) Fix the logo color.
- (B) Add the missing Capgo config so future OTA deploys actually reach
      the iPhone automatically.
- (C) Remove the Wave 2c diagnostic `W2c · <sha>` tag from the splash —
      its job is done.

Plus a safety net to prevent this class of bug from silently recurring:

- (D) Add a preflight warning in `deploy.sh` that detects native-layer
      changes (capacitor.config.json, Podfile, build.gradle, ios/,
      android/) and reminds the operator that `./deploy.sh --prod` alone
      is not enough — they must also run `./ship.sh` for native changes
      to reach devices.
- (E) Tighten the "2 modes" section in `CLAUDE.md` to make this rule
      crystal clear.

## Done looks like
- After `./deploy.sh --prod`, `app.brikops.com` shows the BrikMark "B"
  in orange `#F59E0B` on both initial paint and on return visits.
- The `W2c · <sha>` tag no longer appears at the bottom of the splash.
- `capacitor.config.json` contains a valid `plugins.CapacitorUpdater`
  block with `autoUpdate: true` and `channel: "production"`.
- Running `./deploy.sh` (dry-run, no args) with native-layer changes
  staged/committed prints a BIG yellow warning block:
  ```
  ⚠️  NATIVE CHANGES DETECTED
      Files touched: capacitor.config.json
      ./deploy.sh --prod alone will NOT deliver these to iPhone/Android.
      After this deploy, you MUST also run ./ship.sh on the Mac to
      rebuild the native app and push it to TestFlight / Play Store.
  ```
  and continues execution normally (it does not block the deploy —
  it only warns).
- `CLAUDE.md` "2 מצבים בלבד" section has an expanded bullet list of
  files that require mode 2 (`ship.sh`) — `capacitor.config.json`,
  `package.json` with Capacitor plugin changes, anything under `ios/`
  or `android/`, Podfile, build.gradle, versionCode/versionName.
- No visual regression anywhere else. Brick animation, BrikWordmark,
  blueprint grid, radial glow all unchanged.

## Out of scope
- No changes to `D3ABrickAssembly.jsx` visuals or `BrickAssembly` internals.
- No changes to `BrikLogo.jsx` component (we only change the COLOR PROP
  passed to BrikMark from D2DailySplash.jsx).
- No new native dependencies. No pod install. No version bumps.
- No changes to `.github/workflows/deploy-frontend-capgo.yml`.
- No changes to Android config (the `CapacitorUpdater` block applies to
  both platforms via `capacitor.config.json`; Android behavior is a
  free side-benefit).
- Do NOT add a Settings → About version indicator in this wave. That is
  a future wave.
- Do NOT run `./ship.sh` from Replit. That is Zahi's Mac action after
  this wave deploys.

### Task 4 — Add native-change guard to deploy.sh

Edit `deploy.sh`. After the existing `check_files()` function populates
`frontend_changed` / `backend_changed`, add a new detection for native
changes. Use the SAME `check_files` infrastructure already in place.

Approximate location: right after line 57 (`check_files < <(git diff ...)`
blocks). Add a new local variable `native_changed=0` alongside the
existing ones, and extend the `check_files` pattern matching to set it
to 1 when any of these paths are touched:

- `frontend/capacitor.config.json`
- `frontend/ios/**`
- `frontend/android/**`
- `frontend/package.json` (only when the diff mentions `@capacitor`
  or `@capgo` — a simple `git diff | grep -q "@capacitor\|@capgo"`
  suffices)

Then, in the "Deploy Status" echo block (around line 114–130, where
the script currently prints "What will deploy:"), add right after the
existing echos:

```bash
if [[ $native_changed -eq 1 ]]; then
  echo
  echo "⚠️  ═══════════════════════════════════════════════════"
  echo "   NATIVE CHANGES DETECTED"
  echo "   Files touched: capacitor.config.json / ios / android / capacitor plugins"
  echo "   ./deploy.sh --prod alone will NOT deliver these to the"
  echo "   installed iPhone/Android apps. After this deploy, you"
  echo "   MUST also run ./ship.sh on the Mac to rebuild the native"
  echo "   app and push to TestFlight / Play Store."
  echo "   ═══════════════════════════════════════════════════"
  echo
fi
```

Keep it as a WARNING, not a block. The script must still proceed with
the deploy if the operator confirms. Do NOT add an extra confirmation
prompt — the existing `read -rp "Deploy to production? (y/N): "` at
line 187 is sufficient.

Also add the same warning block to the final "DEPLOY SUMMARY" section
(around line 214+), right after the frontend/backend status lines, so
the operator sees it one last time after the push completes.

Verify by running (in Replit shell after the edit is committed but
before deploy):
