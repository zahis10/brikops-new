# #393 — Login UX: No Auto-Keyboard on Phone + OTP Autofill Popup

## What & Why
שתי תקלות UX בזרימת ההתחברות, שתיהן JS בלבד:

1. **מקלדת נפתחת אוטומטית באייפון בכניסה לעמוד Login.** באנדרואיד לא קורה
   בזכות מדיניות אבטחה של Chrome WebView. באייפון (Capacitor WKWebView) אין
   חסימה כזאת — כל `autoFocus` שמופעל ב-mount פותח את המקלדת, מכסה את הלוגו
   ואת הכפתורים החלופיים (WhatsApp, Apple, Google).

2. **ה-OTP שמגיע ב-SMS לא מפעיל חלון autofill לחיצה-אחת.** המשתמש נאלץ
   להקליד ידנית את 6 הספרות במקום לקבל "From Messages: 123456" מעל המקלדת
   (iOS) או פופ-אפ Web OTP (אנדרואיד Chrome / דפדפן).

שורשי הבעיות אומתו:

1. `LoginPage.js:548` ו-`LoginPage.js:823` — `autoFocus` על שני ה-phone
   inputs. ב-iOS זה פותח את המקלדת המספרית מיד עם render.
2. כל שלושת ה-OTP inputs (605, 777, 874) כבר עם `autoComplete="one-time-code"`
   ו-`inputMode="numeric"` — התשתית הבסיסית קיימת. אבל **אף אחד לא קורא
   ל-`navigator.credentials.get({ otp: { transport: ['sms'] } })`** שזה ה-API
   המפורש שפותח את החלונית ב-Web OTP (עובד על Android Chrome + דפדפן).

ה-SMS template בבקאנד (`backend/contractor_ops/otp_service.py:210`) כבר בנוי
נכון עבור iOS 14+ domain-bound autofill:

```
BrikOps: קוד האימות שלך הוא 123456. בתוקף 10 דקות.

@app.brikops.com #123456
```

השורה האחרונה (`@domain #code`) היא הפורמט שגם iOS וגם Web OTP API מצפים לו.
**אין צורך לגעת בבקאנד.**

### Task 2 — יצירת hook `useOtpAutofill`
קובץ חדש: `frontend/src/hooks/useOtpAutofill.js`

```js
import { useEffect } from 'react';

/**
 * Web OTP API autofill hook.
 * On mount: if navigator.credentials supports OTPCredential (Android Chrome,
 * desktop Chrome, and some Android WebViews), requests a one-time code from
 * an incoming SMS that ends with `@<origin> #<code>`. When the user taps the
 * popup, the code is written via setCode.
 *
 * Silently no-ops on iOS Safari/WebKit (autocomplete="one-time-code" handles
 * those via the "From Messages" keyboard chip).
 *
 * Automatically aborts when the component unmounts.
 */
export default function useOtpAutofill(setCode) {
  useEffect(() => {
    if (typeof window === 'undefined') return;
    if (!('OTPCredential' in window)) return;
    if (!navigator.credentials || !navigator.credentials.get) return;

    const ac = new AbortController();

    navigator.credentials
      .get({ otp: { transport: ['sms'] }, signal: ac.signal })
      .then((cred) => {
        if (cred && typeof cred.code === 'string' && cred.code.length > 0) {
          const digits = cred.code.replace(/\D/g, '').slice(0, 6);
          if (digits.length === 6) {
            setCode(digits);
          }
        }
      })
      .catch((err) => {
        // AbortError = unmount; NotAllowedError = user dismissed; ignore both
        if (err && err.name !== 'AbortError') {
          // eslint-disable-next-line no-console
          console.debug('[useOtpAutofill] Web OTP not available:', err.name);
        }
      });

    return () => ac.abort();
  }, [setCode]);
}
```

**דגשים:**
- ה-hook מקבל `setCode` בלבד — פונקציה ש-`setOtpCode(code)` או
  `setSocialOtp(code)` נכנסות אליה. אל תוסיף עוד פרמטרים.
- אסור לקרוא ל-`submit` אוטומטית אחרי מילוי. המשתמש לוחץ בעצמו.
  (הסיבה: UX של רוב האפליקציות מראה למשתמש מה נכתב לפני שליחה.)
- השתמש ב-`useEffect` **בלי dependencies חדשים** מעבר ל-`setCode`. אם
  `setCode` משתנה בין renders (בגלל שהוא לא stable) — ה-hook ינתק ויתחבר
  מחדש. זה תקין.

  ו-864).

  המשתמש לא נמצא כרגע במסך OTP, ה-API פשוט ימתין. כשה-SMS מגיע, הפופ-אפ

  הערך לא ייראה עד שיגיע למסך. לא מסוכן.

**לא לעשות:**
- לא להוסיף `useOtpAutofill` בתוך sub-components — זה יוצר instances כפולים
- לא לקרוא ל-hook בתוך `useEffect` אחר — hooks חייבים לרוץ ב-top level של
  הרכיב
- לא לעטוף בתנאי `if (phoneStep === 'otp') useOtpAutofill(...)` — זה מפר את
  כלל ה-Rules of Hooks
