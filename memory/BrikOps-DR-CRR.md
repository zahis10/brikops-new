## S3 Cross-Region Replication (CRR) — the full picture

### The setup

```
┌──────────────────────────────────┐      async replication      ┌────────────────────────────────────────────┐
│  brikops-prod-files              │  ━━━━━━━━━━━━━━━━━━━━━━━▶  │  brikops-prod-files-backup-ireland         │
│  eu-central-1 (Frankfurt)        │      15-60s typical         │  eu-west-1 (Ireland)                       │
│  PRIMARY — app reads/writes here │                             │  BACKUP — app does NOT read from here      │
│  Standard storage class          │                             │  Standard-IA (~45% cheaper)                │
│  Versioning: ON                  │                             │  Versioning: ON                            │
└──────────────────────────────────┘                             └────────────────────────────────────────────┘
```

### Replication rule config (in source bucket → Management → Replication rules)

- **Rule name:** `brikops-prod-files-backup-ireland`
- **Status:** Enabled
- **Priority:** 0
- **Scope:** Apply to all objects in the bucket
- **Destination:** `brikops-prod-files-backup-ireland` in eu-west-1
- **IAM role:** auto-generated by AWS (`s3crr_role_for_brikops-prod-files_to_brikops-prod-files-backup-ireland` or similar)
- **Storage class on destination:** **Standard-IA**
- **Encryption (KMS):** OFF
- **Delete marker replication:** **OFF** ← CRITICAL SAFETY FEATURE
- **Replica modification sync:** default
- **Replication Time Control (RTC):** OFF (we don't need 15-min SLA; saves cost)

### Why "Delete marker replication: OFF" is the single most important setting

If enabled:
- User/hacker/bug deletes file in Frankfurt → Frankfurt gets delete marker → **Ireland ALSO gets delete marker** → Ireland "forgets" the file too.
- Ransomware scenario: attacker with `s3:DeleteObject` permission destroys both copies.

With it OFF (our setup):
- User deletes in Frankfurt → Frankfurt gets delete marker → **Ireland ignores it** → Ireland keeps the file visible.
- Even a full adversary with production S3 delete access cannot touch the Ireland backup.

### Historical backfill — how we caught up the 727 pre-existing files

CRR only replicates **new** uploads after the rule is enabled. To backfill:

1. S3 → Batch Operations → Create job
2. Source: current bucket
3. Operation: **Replicate**
4. Priority: 10
5. Completion report: unchecked (skip audit log for first run)
6. Permissions: create new IAM role
7. Job started "New" → confirmed → "Active" → 5 min runtime → "Complete"
8. Result: **727/727 objects replicated, 0 failed**
9. Job ID: `b530fb45-7354-48cf-9f24-3d462acd0cca`

## Failover procedure (how to activate Ireland as primary)

**When to use:** Frankfurt region has outage > 30 min, or bucket corruption in Frankfurt confirmed.

**Do NOT use for:** single-file deletes (use Show versions), bulk deletes (use s3-bulk-restore.sh), Atlas issues (unrelated).

### Steps

1. **Confirm Frankfurt is actually down** — check `https://health.aws.amazon.com/`. Don't failover on a transient blip.
2. **Replit → Secrets / Environment:**
   - `AWS_BUCKET` → `brikops-prod-files-backup-ireland`
   - `AWS_REGION` → `eu-west-1`
3. **Deploy:** `./deploy.sh --prod` from Replit (OTA push — no native build needed for backend env vars).
4. **Verify:** open BrikOps on phone → open a task with photos → photos load from Ireland.
5. **Communicate:** notify users of degraded mode (reads from backup region may be slightly slower; writes will succeed in Ireland).

### When Frankfurt recovers

1. CRR does NOT auto-reverse. Ireland → Frankfurt replication is not configured.
2. To restore primary: set up reverse replication OR sync manually with `aws s3 sync s3://brikops-prod-files-backup-ireland s3://brikops-prod-files` (can take hours).
3. Once synced, reverse env vars back.

**Open decision for future Zahi:** do we want bi-directional replication? Currently no — Ireland is strictly one-way backup. Reconsider when scale justifies complexity.

## What runs automatically (no human touch needed)

- Every new S3 upload → replicated to Ireland in 15-60 sec
- Every file modification → new version created in both regions
- Atlas continuous backup → snapshots every 6h / daily / weekly
- Atlas cross-region copy → eu-west-1

## What requires human action

- PITR target time selection
- Failover decision (Frankfurt → Ireland)
- Customer communication during incidents
- Quarterly sanity check: Ireland object count matches Frankfurt
- Adding a new bucket: needs its own replication rule (not inherited)

## Key lesson from 2026-04-22 setup session

When creating the Ireland bucket, the region selector defaults to whatever region you're currently in (Frankfurt). **Always verify the region dropdown at the top-right is set to Ireland (eu-west-1) BEFORE clicking Create bucket.** A backup bucket in the same region as the primary = no disaster protection at all.
