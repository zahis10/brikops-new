DESIGN NOTES
================================================================

1. The spec said "1 file only" — we touched 2. Why?
   The spec's thumbnail JSX uses `<img src={tenant.id_photo_url}>`,
   but in this codebase `id_photo_url` is the persisted S3
   stored_ref (same shape as meter `photo_url`). It's not a
   directly-loadable URL — only `display_url`, generated server-side
   via `generate_url()`, is loadable into an `<img>` tag.

   The existing meter form (HandoverMeterForm.js line 11) uses
   `meters[type]?.display_url` for `<img src>` — and the GET
   protocol endpoint at lines 1122-1129 explicitly populates
   `meters.{type}.display_url` from `meters.{type}.photo_url` via
   `generate_url()`. Without our T002 backend addition, the new
   thumbnail would only work in the seconds right after upload
   (when the FE has the fresh response in memory). After page
   reload it would break — silently in S3 mode (404 on the bucket
   key) and only "happen to work" in local mode by URL-path
   coincidence. Court-grade evidence cannot have a
   silently-broken thumbnail.

   T002 is 14 lines total (7 lines × 2 endpoints), mirroring the
   meter loop exactly. `generate_url` was already imported (line
   9). Zero new dependencies.

2. New field `id_photo_display_url` lives only in transit
   It's added to the GET response payload (T002a), to React state
   (T001b), and to the upload handler's setState (T001c). It is
   NEVER persisted to MongoDB — the upload endpoint at line 2492
   only $set's `id_photo_url` (the stored_ref), never the
   display_url. This matches the meter pattern's `photo_url`
   (persisted) vs `display_url` (transient) separation.

3. PUT with stale display_url in body — non-issue
   When the FE sends a PUT with `tenants` that came from a fresh
   GET, the body may include `id_photo_display_url`. The PUT
   handler at line 1165 does `update_fields[field] = body[field]`
   for `tenants`, so the stale display_url would land in the
   document. This is fine because:
     (a) presigned URLs have a TTL — even if persisted, they're
         worthless after expiry
     (b) the PUT response loop (T002b) overwrites it fresh on the
         way back out
     (c) the stored_ref `id_photo_url` is what the PDF service
         actually reads; display_url is purely a thumbnail aid
   Server-side stripping is out of scope for this task. The
   underlying S3 object remains ACL-protected regardless.

4. No new PII audit logging
   `[HANDOVER-PII]` is for mutations (upload, delete). GET is a
   read on a protocol the user already has access to. No new
   PII surface.

5. UI structure faithful to spec
   - 2 outlined pill buttons side-by-side (purple Camera, slate
     Upload)
   - 2 separate hidden inputs with unique IDs per tenant index
   - Camera input has `capture="environment"` (opens camera on
     mobile, ignored on desktop)
   - Upload input has no capture (gallery on mobile, file dialog
     on desktop)
   - Thumbnail w-24 h-18, X delete in -top-1.5 -right-1.5 corner
   - Labels: "צלם"/"העלה" → "החלף בצילום"/"החלף מהגלריה"
     when photo exists; "מעלה..." + spinner on the camera button
     during upload
   - Signed-protocol with photo: thumbnail visible, no X, label
     "(נעול — פרוטוקול חתום)"
   - Signed-protocol without photo: nothing renders
