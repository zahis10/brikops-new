DESIGN NOTES
================================================================

1. tenants_data construction had to MOVE
   The original `tenants_data = [t for t in tenants if t.get("name")]`
   was built at line 370 — BEFORE `image_keys` was assembled and
   resolved into `images`. To attach `id_photo_b64` we need both the
   tenant data AND the resolved images dict, so the build was moved
   to line 707, just before the context dict (where `images` is fully
   populated). The same name-filter is preserved — no behavioral
   change for tenants without a name. The placeholder comment at the
   original line 370 explains the relocation.

2. Image-key naming uses ORIGINAL tenant index (not filtered)
   In step B2 we enumerate `tenants` (the unfiltered list from
   protocol). In step B4 we ALSO enumerate `tenants` with the same
   indices, then filter-by-name. That keeps the `images.get(
   f"tenant_{idx}_id")` lookup consistent regardless of which earlier
   tenants in the list lack a name.

3. `.tenant-id-photos` kept as `table-cell` (NOT switched to flex)
   The spec asked to change `.tenant-id-photos` from `display:
   table-cell; width: 110px` → `display: flex`. Doing so would have
   broken the existing tenant-row table layout (the tenant card uses
   `.tenant` rows where `.tenant-avatar`, `.tenant-info`, and
   `.tenant-id-photos` are all `table-cell`s aligning vertically).
   We kept the table-cell wrapper and only upgraded the inner
   `.id-photo` box (90×56, inline-flex for centered label, has-image
   variant). The plan's "architectural notes" section explicitly
   documented this fall-back option.

4. Court-grade compression tier
   Defect photos: 1100/q85 (existing). ID photos: 1200/q80 (new).
   Meter photos: 800/q80 (existing). Signatures: 300/q85 (existing).
   ID is wider than defects (more pixels for OCR/verification) but
   slightly lower JPEG quality (smaller files, still court-readable).
   Constants chosen to match the spec exactly.

5. PII audit prefix `[HANDOVER-PII]` (NOT `[HANDOVER]`)
   Distinct prefix so audit-log extraction tooling can grep on it
   for regulatory exports. Both upload AND delete log a line.

6. Frontend API uses axios + multipart pattern
   The original spec used a `fetch + getToken() + API_BASE` pattern
   that doesn't exist in this codebase. Adapted to the established
   axios + `getAuthHeader()` + `'Content-Type': 'multipart/form-data'`
   pattern used by `signRole`, `uploadMeterPhoto`, `uploadOrgLogo`,
   etc. The plan's "architectural notes" called this out.

7. `onUpdated?.()` called on success
   Both upload and delete handlers call `onUpdated?.()` after toast
   so the parent can refetch protocol if it wants. Matches how the
   existing `handleSave` works.
