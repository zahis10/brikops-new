THREE VULNERABILITIES CLOSED
----------------------------

(1) CRIT-A — Role escalation at POST /api/auth/register
    Before: any unauthenticated client could POST a `role` field of their
            choosing (e.g. "project_manager", "admin", "management_team")
            and the server would persist whatever was sent. The schema
            default of `Role.viewer` was a weak hint, not an enforcement.
    After:  the server now hardcodes `resolved_role = "viewer"` for every
            new account regardless of what the client sends. If a client
            ever requests a role other than "viewer", a WARNING is logged
            with the requested role and email so we can spot abuse.
            Track upgrade for legitimate users still happens through the
            existing `register-with-phone` + PM-approval flow, which is
            untouched.

(2) CRIT-B — IDOR at POST /api/tasks/{task_id}/reopen
    Before: the route was protected by the global Role check
            (`require_roles('project_manager', 'management_team')`), but
            this is a PLATFORM role check — it does NOT verify that the
            caller has any role on THE SPECIFIC PROJECT the task belongs
            to. A user with the global PM role on project A could reopen
            a closed task on project B that they have no business touching.
    After:  after the existing 404/status checks and BEFORE any DB mutation,
            the route now resolves the caller's role on `task.project_id`
            via `_get_project_role(...)` and rejects with 403 if the role
            is not `project_manager` or `management_team` on that project.
            Hebrew error message: 'אין לך הרשאה לפתוח מחדש משימה זו'.

(3) CRIT-C — IDOR at POST /api/tasks/{task_id}/attachments
    Before: the route only checked the caller's GLOBAL role != 'viewer'.
            Any non-viewer in the system could upload attachments to any
            task — even tasks on projects they had no membership in and
            were not assignees of. This was inconsistent with the
            `/updates` endpoint right above it (which already enforced
            project membership OR assignee).
    After:  the route now mirrors the `/updates` guard exactly — resolves
            project membership via `_get_project_membership(...)`, checks
            `is_assignee = task.assignee_id == user.id`, and rejects with
            403 if membership is 'none' AND user is not the assignee.
            Hebrew error message: 'אין לך הרשאה להעלות קבצים למשימה זו'.

WHAT THIS DOES NOT TOUCH
------------------------
  • No frontend code (`git diff --stat frontend/` is empty — verified).
  • No schemas, no migrations, no env vars, no secrets, no GitHub workflows,
    no Capgo workflow, no deploy.sh, no Mongo indexes.
  • No changes to legitimate user flows: /api/auth/register-with-phone (the
    real registration path) and PM-approval flows are untouched.
  • No changes to any GET endpoint or to any of the 8 other mutation
    endpoints in tasks_router.py — all 11 already had project-scoped guards
    (see Coverage Audit below); only /reopen and /attachments were missing
    them.

@router.post("/tasks/{task_id}/reopen", response_model=Task)
async def reopen_task(task_id: str, user: dict = Depends(require_roles('project_manager', 'management_team'))):
    db = get_db()
    task = await db.tasks.find_one({'id': task_id}, {'_id': 0})
    if not task:
        raise HTTPException(status_code=404, detail='Task not found')
    if task['status'] not in ('closed', 'approved'):
        raise HTTPException(status_code=400, detail='Only closed tasks can be reopened')
    ts = _now()
    prev_status = task['status']
    await db.tasks.update_one({'id': task_id}, {
        '$set': {'status': 'reopened', 'updated_at': ts},
        '$unset': {'force_closed_by': '', 'force_closed_reason': '', 'force_closed_at': '', 'force_closed_type': ''},
    })
    await db.task_status_history.insert_one({
        'id': str(uuid.uuid4()), 'task_id': task_id,
        'old_status': prev_status, 'new_status': 'reopened',
        'changed_by': user['id'], 'note': 'Task reopened', 'created_at': ts,
    })
    await db.task_updates.insert_one({
        'id': str(uuid.uuid4()), 'task_id': task_id, 'user_id': user['id'],
        'user_name': user.get('name', ''), 'content': 'Task reopened',
        'update_type': 'status_change', 'old_status': prev_status,
        'new_status': 'reopened', 'created_at': ts,
    })
    await _audit('task', task_id, 'reopen', user['id'], {})
    updated = await db.tasks.find_one({'id': task_id}, {'_id': 0})
    return Task(**updated)
