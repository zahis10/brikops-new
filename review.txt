ROOT CAUSE (verified on code)
--------------------------------------------------------------------------------
services/object_storage.py save_bytes() returns different shapes per backend:
    • S3 mode    → "s3://{key}"          e.g. s3://safety/<pid>/<uuid>.jpg
    • local mode → "/api/uploads/{key}"
The D3 SSRF gate — _PHOTO_REF_RE (work_diary_router.py) and the duplicated
_safe_ref built inside work_diary_pdf.py — only allowed an OPTIONAL
"/api/uploads/" prefix. So every REAL s3:// ref failed shape validation →
_validate_photo_refs raised 422 → PATCH photo_refs rejected. Photos could never
be saved once the backend ran on S3 (staging/prod).

Why D3 missed it: the D3 probes ran FILES_STORAGE_BACKEND=local (standing rule),
so only the local "/api/uploads/" shape was ever exercised — the s3:// shape
was never tested.

NEW STANDING RULE (recorded in memory): stored-ref shape validations must carry
direct regex assertions for BOTH backend shapes (local + s3), not just local.

THE CHANGE (2 files — the two regexes kept IDENTICAL)
--------------------------------------------------------------------------------
1. backend/contractor_ops/work_diary_router.py   (_PHOTO_REF_RE)
2. backend/services/work_diary_pdf.py            (_safe_ref)

  old: r"^(?:/api/uploads/)?safety/([A-Za-z0-9_-]+)/[A-Za-z0-9][A-Za-z0-9.+_-]*$"
  new: r"^(?:/api/uploads/|s3://)?safety/([A-Za-z0-9_-]+)/[A-Za-z0-9][A-Za-z0-9.+_-]*$"

Nothing else touched: MAX_DIARY_PHOTOS=12 cap, group(1)==project_id match, and
the ".." re-check all stay exactly as they were.

3. backend/tests/probe_diary_d3_http.py — new assertions (V1j-m, V2fix2a-e, V3fix2).

SECURITY POSTURE (UNCHANGED — architect-confirmed)
--------------------------------------------------------------------------------
• http:// and https:// still CANNOT match: the prefix alternation allows only
  the two literal prefixes ("/api/uploads/" | "s3://") or nothing, then requires
  the literal "safety/"; the body permits no ":" and no "/" beyond the fixed
  segments — so a full URL scheme can never satisfy it.
• "s3://<bucket>/safety/..." (extra bucket segment) is still REJECTED — after
  "s3://" the next chars must be literally "safety/". Our refs are "s3://" + key
  with NO bucket segment, so this is correct.
• Cross-project refs still blocked by the unchanged group(1)==project_id check.
• Path traversal still blocked by the unchanged ".." check.
The SSRF gate that D3 added is fully preserved; this only widens the allowed
prefix set to include our own S3 output shape.
