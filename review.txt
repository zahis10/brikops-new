### HIGH-A — Stored XSS via contractor proof upload
Files: tasks_router.py · upload_safety.py · storage_service.py · tests/

The contractor-proof endpoint accepted ANY file type and the S3 key
extension was derived from the user-supplied filename. An attacker could
upload a .html file with text/html content-type, then trick the manager
UI into rendering it as inline content (XSS).

Fix:
  - upload_safety.py — added ALLOWED_PROOF_EXTENSIONS / ALLOWED_PROOF_TYPES
    (images + PDF, per Zahi locked-decision: PDFs are first-class proofs).
  - tasks_router.py — `validate_upload(...)` is now called inside the
    per-file loop before storage; rejects with 422.
  - storage_service.py — extension is now derived from a MIME whitelist
    (EXT_BY_MIME), NEVER from the filename. Unknown MIMEs become `.bin`.
  - tests/test_contractor_hardening.py — added 2 tests: PDF accepted (200),
    HTML rejected (422 + status not mutated).

Verification:
  - PDF upload: validate_upload(cert.pdf, application/pdf) → OK
  - HTML upload: → HTTPException(422, 'סוג קובץ לא נתמך')
  - SVG upload: → HTTPException(422, 'סוג קובץ לא נתמך')

### HIGH-B — Brute-force lockout on /login + /login-phone
Files: auth_lockout.py (NEW) · auth_router.py · onboarding_router.py · server.py

Both password-login endpoints had no rate-limiting. Failure messages were
also distinct in Hebrew per failure mode, allowing username/phone
enumeration ('מספר טלפון לא רשום' vs 'סיסמה שגויה' vs 'לא הוגדרה סיסמה').

Fix — new shared module backend/contractor_ops/auth_lockout.py:
  - 5 failures within 15-min window → lockout for 15 min
  - Per-(identifier, IP) tuple — different IPs scoped independently
  - 24h TTL auto-cleanup via Mongo TTL index on last_failure
  - Compound unique index on (identifier, ip)
  - All failure paths in BOTH routers collapse to one English 401:
    "Authentication failed". Locked-out responses add a Retry-After header.
  - Per locked-decision #7: the 400 'לא הוגדרה סיסמה. יש להתחבר עם OTP.'
    on /login-phone is also folded into the generic 401 — the login UI
    already exposes an OTP button, so the hint was redundant and leaked
    account existence.
  - Counter increments even when user does not exist (prevents
    enumeration via "which emails enter lockout vs which never do").

Verification (live against running dev server):
  - Attempts 1-5 same email/IP: HTTP 401 (no Retry-After)
  - Attempt 5: count=5, lockout_until set in DB
  - Attempts 6+: HTTP 401 with `retry-after: 899` (~15 min)
  - Same email from 8.8.8.8 while locked from 127.0.0.1: NOT locked (scoped)
  - Mongo collection has 2 indexes: auth_failed_identifier_ip_unique +
    auth_failed_ttl (TTL=86400)

Architect-review fix (sliding window — caught at code review):
  Original code only set first_failure on insert and never reset it.
  An attacker who paced probes >24h apart could drive count arbitrarily
  high while keeping first_failure stale, never tripping the threshold.
  Fixed: when existing first_failure ≤ window_start, count is reset to 1
  and lockout_until is unset BEFORE evaluating the threshold.
  Re-verified: seeded count=4 with first_failure 16min old; next failure
  reset count to 1 (not progressed to lockout). Full 1→6 cycle still fires.

### HIGH-C — PayPlus webhook HMAC enforcement
File: billing_router.py:~1453-1473

HMAC verification only fired when `User-Agent: PayPlus` was present
(trivially bypassable). On hash mismatch the handler returned 200 OK
silently (allowed attacker probing).

Fix:
  - 'hash' header is now MANDATORY for ALL requests, regardless of UA.
  - Missing hash → 401 "Missing webhook signature"
  - Hash mismatch → 401 "Invalid webhook signature"
  - hmac.compare_digest used for constant-time comparison

Verification:
  - curl without hash header → HTTP 401
  - curl with bad hash      → HTTP 401

Known low-risk edge: malformed JSON returns 400 before hash check (no
state mutation, no DB write, no info leak — purely about request shape).
Logged as follow-up for symmetry with the GI fix.

### HIGH-D — GreenInvoice webhook auth + startup guard
File: billing_router.py:~1238-1253 (top of handler) + server.py:~360-370

Original token check was wrapped in `if GI_WEBHOOK_SECRET:` so an UNSET
secret silently disabled auth. Worse, the token check was placed AFTER
several early-return branches (no-doc-id → 200, duplicate doc-id → 200),
so unauthenticated callers could trigger DB writes by sending an empty
or known-duplicate doc_id.

Fix:
  - Token check moved to TOP of handler, before any DB I/O or branching.
  - hmac.compare_digest used for constant-time comparison.
  - Missing OR mismatched X-Webhook-Token / ?token= → 401.
  - Startup guard in server.py: if GI_BASE_URL is set but
    GI_WEBHOOK_SECRET is NOT, raise RuntimeError and refuse to boot.
    Mirrors the existing S5b WhatsApp/META_APP_SECRET guard.

Verification:
  - missing token + missing doc_id (was 200): → HTTP 401
  - missing token + empty body (was 200):     → HTTP 401
  - VALID token + missing doc_id (legitimate): → HTTP 200 (no_doc_id)
  - bad token:                                  → HTTP 401

### MED-D — Paywall middleware case-sensitivity
Files: server.py:~1371 + ~77 + ~298

FastAPI's HTTPBearer dependency is case-insensitive on the Bearer scheme,
but the paywall middleware used `.startswith('Bearer ')` (case-sensitive).
A request with `Authorization: bearer X` (lowercase b) was authenticated
by the dependency but slipped past the paywall check.

Fix:
  - paywall_middleware: `auth_header.lower().startswith('bearer ')`
  - _extract_jwt_payload: same change (architect-review consistency)
  - debug_db_ping handler: same change (architect-review consistency)

Verification: `Authorization: bearer fake` → HTTP 401 (paywall now enforced)
