FIX 1 — Orphan pages 5+6 (Option A: split rooms, keep break)
================================================================

PROBLEM
-------
On v3.5 staging:
  page 4: defects summary table
  page 5: EMPTY — only header strip
  page 6: ALMOST EMPTY — only "ROOM INSPECTIONS · בדיקת חדרים"
  page 7: first room (כניסה לדירה)

Three things conspired:
  • `<div class="page-break">` forced a fresh page for the section
  • the v3.5 `keep-with-next` made the page-label glue itself to
    the first room
  • `<div class="no-break">` around every room said "do not split
    this room across pages"

The first room is too tall to fit alone on the freshly-opened
page → WeasyPrint pushes the whole thing out → page 5 ends up
empty. Then on page 6 the same thing happens with the next page
(label moves with it), leaving page 6 with just the label.

OPTION A FIX (per Zahi's revised spec)
--------------------------------------
Keep the section's forced page-break (the section still opens
cleanly on its own page with the header strip on top — that's
desirable), but allow rooms to split across pages, with the
room title glued to the start of its items table on whichever
page that table begins.

Three changes:

  1a. Replace the `no-break` wrapper around each room with a new
      `room-block` class, in the inspection_sections loop:

        -  <div class="no-break">
        +  <div class="room-block">

      (only one occurrence in the body — the room-block wrapper
       inside `{% for sec in inspection_sections … %}` at the
       new line 1333)

  1b. Add CSS rules:

        +  .room-block {
        +    page-break-inside: auto;
        +  }
        +  table.items tbody tr {
        +    page-break-inside: avoid;
        +  }

      `.room-head` already had `page-break-after: avoid` from a
      prior version (line 476), so the room title stays glued
      to the start of its items table. Combined with the new
      `tbody tr` rule, individual table rows never split.

  1c. Remove `keep-with-next` from the section's page-label —
      no longer needed and was making things worse:

        -  <div class="page-label keep-with-next">ROOM INSPECTIONS · בדיקת חדרים</div>
        +  <div class="page-label">ROOM INSPECTIONS · בדיקת חדרים</div>

      The `.keep-with-next` CSS rule itself is left in place
      (harmless; might be useful again later).

Result:
  • ROOM INSPECTIONS still starts on a fresh page with header
    strip on top (page-break preserved).
  • The first room flows immediately under the page-label on
    that same page.
  • A long room may split between table rows across pages, with
    the room title staying with the start of the table on its
    first page; rows stay intact.
  • No more orphan / empty pages.

FIX 2 — Defect photos still too dominant
================================================================

PROBLEM
-------
v3.5 set 1-photo and 2-photo cells to 50% width. Zahi feedback:
"תמונות עדיין ענקיות מדי לדעתי". One photo at 50% still feels
billboard-sized; two photos at 50% each take the entire row
width.

FIX
---
  -.photo-row.count-1 .cell { width: 50%; }
  -.photo-row.count-2 .cell { width: 50%; }
  +.photo-row.count-1 .cell { width: 40%; }
  +.photo-row.count-2 .cell { width: 40%; }
   .photo-row.count-3 .cell { width: 33.33%; }

  • 1 photo at 40% — comfortable card-sized image.
  • 2 photos at 40% each (= 80% total) — fits with margin to spare.
  • 3 photos unchanged at 33% each.

The underlying photo source is still 1100 px wide JPEG q=85
(unchanged from v3.5), so reducing the cell width actually
INCREASES effective DPI in print: ~1100 px / ~6 cm cell ≈ ~190
DPI (was ~150 DPI at 50% / ~7.5 cm). Sharper, not blurrier.

FIX 3 — Page header: contractor logo + delivery date (macro)
================================================================

PROBLEM
-------
  • The interior page header strip showed only the BrikOps
    platform logo. The contractor's company logo
    (`logo_b64` in the context, loaded from `company_logo_url`
    by the service at handover_pdf_service.py:387) was never
    rendered.
  • The delivery date hid as plain text in the right strip,
    sharing space with the reference number — easy to miss.
  • Seven copies of the same `<div class="ph">…</div>` block
    lived inline at 7 page-break sections (TENANTS, TABLE OF
    CONTENTS · DEFECTS SUMMARY, ROOM INSPECTIONS, HANDED-OVER
    ITEMS · METER READINGS, LEGAL DECLARATIONS, TENANT NOTES,
    SIGNATURES). Any future header tweak meant 7 edits.

FIX
---
DRY refactor + content additions, no new dependencies:

  3a. New Jinja macro at the top of `<body>`:

        {% macro page_header() %}
        <div class="ph">
          <div class="ph-left">
            {% if brikops_logo_b64 %}<img class="ph-logo" src="{{ brikops_logo_b64 }}" alt=""/>{% endif %}
            {% if logo_b64 %}<img class="ph-logo ph-company-logo" src="{{ logo_b64 }}" alt=""/>{% endif %}
            <span class="ph-title">פרוטוקול מסירה{% if building_name or unit_name %} · <b>{{ building_name }}{% if unit_name %} · {{ unit_name }}{% endif %}</b>{% endif %}</span>
          </div>
          <div class="ph-right">
            {% if signed_date %}<span class="ph-date">מסירה: <b>{{ signed_date }}</b></span>{% else %}{{ generation_date }}{% endif %}
            {% if display_number %} · {{ display_number }}{% endif %}
          </div>
        </div>
        {% endmacro %}

      Three behaviour changes vs. the old inline block:
        - Adds the contractor logo (`logo_b64`) next to the
          BrikOps logo when present (silently skipped otherwise).
        - Right side switches from raw `generation_date` to
          "מסירה: <signed_date>" in orange-bold; falls back to
          `generation_date` only when not yet signed (preserves
          the previous behaviour for unsigned protocols).
        - Reference number (`display_number`) remains in the
          same right-side position, after a separator.

  3b. New CSS (placed in the existing legal/sig CSS block):

        +.ph-company-logo {
        +  height: 22px;
        +  margin-right: 8px;
        +  vertical-align: middle;
        +}
        +.ph-date {
        +  color: var(--ink);
        +}
        +.ph-date b {
        +  color: var(--orange);
        +  font-weight: 700;
        +}

      `--orange: #F08A3E` is the existing palette token at
      template line 22.

  3c. All 7 inline `<div class="ph">…</div>` blocks (which were
      byte-identical) replaced with `{{ page_header() }}`:

        $ grep -c "{{ page_header() }}" template
        → 7

        $ grep -c '<div class="ph">' template
        → 1   (only the one inside the macro definition itself)

      Net code change: -7 × 7 lines of duplicated header markup,
      +13 lines for the single macro definition. Future header
      changes are now one-line edits.
