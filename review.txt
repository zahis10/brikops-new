ARCHITECT REVIEW — 1 BLOCKING FINDING, FIXED + RE-VERIFIED
--------------------------------------------------------------------------------
Finding (severe, real): photo_refs were accepted as arbitrary strings; the PDF
image loader HTTP-fetches any ref that generate_url passes through as an
http(s) URL. A writer could PATCH photo_refs=["http://169.254.169.254/..."] and
make the SERVER fetch internal targets on export (SSRF), plus no server-side
count cap (resource exhaustion).
FIX (two layers, both in this batch):
  a. Write-time gate — _validate_photo_refs() on PATCH: each ref must match
     ^(/api/uploads/)?safety/{THIS project_id}/{uuid}.{ext}$ (no schemes, no
     "..", no slashes in filename), max 12 refs → otherwise 422 Hebrew detail.
  b. Export-time defense-in-depth — generate_work_diary_pdf skips (and logs)
     any ref not matching the same shape BEFORE it reaches the loader.
New negative probes V1f-V1i (below) prove: http:// ref → 422, cross-project
ref → 422, 13 refs → 422, entry untouched after rejections.
All other areas passed: RBAC read-widen correct, PDF endpoint/headers/audit
correct, fail-soft photo handling correct, paywall tuple correct.
