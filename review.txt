KNOWN LIMITATION (documented, not fixed in this batch)
================================================================================
The `change_task_status` endpoint (Vector 2 — POST /tasks/{id}/status
with target_status='closed') has NEVER created a Bell notification —
only a WhatsApp. After this batch removes the WhatsApp:
  - Contractor gets ZERO notification when PM manually transitions a
    defect to 'closed' via this endpoint.
  - That's a silent close.

Why this is acceptable for this batch:
  1. The "manual status change to closed" path is rarely used in
     practice. The canonical PM-close flow is `manager_decision`
     (after contractor uploaded proof), which still creates a Bell.
  2. If a PM is manually changing status (without going through the
     proof-review flow), they're typically doing admin cleanup and
     have already informed the contractor via other channels.
  3. Adding a Bell to change_task_status would require extending its
     existing dispatch logic (which currently has no Bell at all) —
     larger scope and risk for an edge case.

If observed as a problem in practice: separate follow-up batch to add
a Bell to change_task_status for status='closed' transitions.

================================================================================
RISK
================================================================================
Very low. Removing two WA dispatch points; no change to message
content, no change to data model, no change to other notification
paths.

Risk vectors:
- ⚠ False expectation — a contractor who used to rely on the WA on
  close to "confirm" the PM saw the work will no longer get it. But
  they can still see the status in the app, and they got a WA when
  the PM rejected previously. If a real user reports this is a loss,
  we can add it back behind a different template (text-only, no photo).
- ⚠ Edge case — if some downstream system (analytics, webhook
  consumer, etc.) listens for 'manager_approved' or 'status_closed'
  notification events, they'll stop firing. Verified no other
  consumers in this codebase. If we have external listeners, this
  spec needs an addendum.

Bundle size impact: zero (backend only).
Backend code size impact: ~3 LOC net change (one branch added, one
dict key removed, plus comments).
