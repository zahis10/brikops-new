KEY IMPLEMENTATION DECISIONS
--------------------------------------------------------------------------------
• Deep-link (?date= outside loaded month) — per your approved correction:
  month picker jumps to date.slice(0,7), list loads that month, entry matched
  by diary_date; no match → "לא נמצא יומן לתאריך זה" empty state with a back
  button. URL state via cloned URLSearchParams, single-key set/delete.
• Autosave engine: pendingRef accumulates per-key partials, 800ms debounce,
  flush() before sign and before refresh-derived. After architect review
  (below) hardened twice: (1) failed non-409 PATCH restores its payload into
  pendingRef (newer edits win per key) so nothing is silently lost; (2)
  inFlightRef promise lock — flush awaits any in-flight PATCH first, so
  sign/refresh can never race a timer-fired autosave. 409 on PATCH → fetch
  server truth → editor flips read-only (entry was signed elsewhere).
• 409 on create ("יומן היום" double-tap or created on another device) → not an
  error: reload month + open the existing entry.
• Any edit to a derived row marks it source:"manual" (survives
  refresh-derived, verified by probe).
• Sign guard mirrors backend: no_work=false + empty description → blocked
  client-side with the same Hebrew message the API returns (422 text).
• no_work entry is created as DRAFT with the auto description (D1 semantics),
  sign flow identical to a normal day.

ARCHITECT REVIEW
--------------------------------------------------------------------------------
Round 1 flagged TWO material autosave issues, both FIXED and re-built:
  1. flush() cleared pendingRef before the PATCH succeeded → failed non-409
     save silently dropped queued edits. FIX: restore failed payload into
     pendingRef (newer edits win per key) → retried on next flush.
  2. flush() didn't await an already-in-flight PATCH → sign/refresh could
     race a timer-fired autosave. FIX: inFlightRef promise lock; flush always
     awaits the in-flight request before sending or returning.
