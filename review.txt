--------------------------------------------------------------------------------
ROOT CAUSE
--------------------------------------------------------------------------------
A blob: object URL is valid ONLY inside the document that created it. 4c stores one
in meters[type].display_url for the offline preview (display-only, correct), but
handleSave persisted the WHOLE meters object verbatim — so display_url:"blob:…"
leaked into handover_form_updates AND onto the server. After an app restart the 4b
form-hydrate (setMeters(rec.value)) reinstated the DEAD blob URL → broken image
(device tests 1+2; test 1's water meter broke via test 2's form save — one save
snapshots both meters). The server could also serve it: get_protocol regenerates
display_url ONLY when photo_url is truthy (~1138), so a not-yet-drained meter
returned the dead string verbatim.
Pre-existing latent bug, also fixed: even ONLINE, handleSave always sent a stale
display_url (presigned URL) to the server.

COMMIT MESSAGE (also written to .local/.commit_message)
--------------------------------------------------------------------------------
fix(handover): never persist session-local display_url — broken meter previews

display_url is derived display state (often a blob: object URL valid only in
the creating document). The meters form persisted it into the offline outbox
and the server, so after an app restart the stored URL was dead → broken
previews (device tests 1+2). handleSave now strips display_url from every
persisted payload (outbox + server, online included — closing a pre-existing
latent bug of storing stale presigned URLs); the 4b form-hydrate merges the
queued value while preserving the live preview; the form drain strips legacy
queued records. Server data self-heals: get_protocol regenerates display_url
from photo_url on every GET.
