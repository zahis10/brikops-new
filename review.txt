================================================================
WHY Rubik-Bold.ttf STAYS ON DISK (two isolated PDF paths)
================================================================

The codebase has TWO independent PDF generators:

  1. HANDOVER PDF — backend/services/handover_pdf_service.py
     Engine: WeasyPrint (HTML/CSS → PDF)
     Template: backend/templates/handover_protocol_pdf.html
     Fonts: declared via @font-face inside the template.
     After this PR: only Rubik-Regular.ttf (variable, 300-900) +
                    Caveat-Regular.ttf. NO reference to Rubik-Bold.

  2. SAFETY PDF — backend/services/safety_pdf.py
     Engine: ReportLab (programmatic Canvas + Platypus)
     Fonts: registered programmatically via reportlab.pdfbase.
       Line 38: bold = os.path.join(_FONTS_DIR, "Rubik-Bold.ttf")
       Line 44: pdfmetrics.registerFont(TTFont("Rubik-Bold", bold))
       Lines 221, 225: ParagraphStyle(... fontName="Rubik-Bold" ...)
     Still requires the static Rubik-Bold.ttf at runtime.

These two paths share the fonts/ directory but otherwise do not
overlap. The handover fix (variable font for all weights) does
not affect safety_pdf, and keeping the static Bold on disk does
not affect handover (the handover template no longer references
it). Both stay green.

If the safety PDF is ever migrated to WeasyPrint or re-fonted
later, Rubik-Bold.ttf can be revisited — but that is OUT OF SCOPE
for this PR.

COMMIT A — Variable Rubik for all weights (Hebrew gibberish fix #3)
================================================================

ROOT CAUSE (re-diagnosed after follow-up #2 didn't resolve it)
--------------------------------------------------------------
Removing text-transform: uppercase removed half the symptom but
not the cause. Real root cause: the static Rubik-Bold.ttf shipped
in backend/fonts/ has NO Hebrew glyphs:

  backend/fonts/Rubik-Bold.ttf:    47KB,  hebrew=0/27,    static
  backend/fonts/Rubik-Regular.ttf: 359KB, hebrew=27/27,   variable
                                   (wght axis range 300-900)

WeasyPrint behavior for "Handover Protocol · פרוטוקול מסירה"
(the .cover-eyebrow at font-weight: 600):
  1. Picks Rubik @ 700 (closer to 600 than to 400).
  2. For Hebrew → Rubik-Bold has no Hebrew → fallback to
     Rubik-Regular variable instanced at weight 700.
  3. For Latin lowercase → "should" use Rubik-Bold, but in
     practice the variable-instance subset for the same paragraph
     gets cached / picked, and that subset (built for Hebrew +
     uppercase + digits at weight 700) has no Latin lowercase.
  4. lowercase Latin fails to map → renders as Latin Extended
     visual lookalikes ("AOĻŠŇŜŴÒÜ! ÚŜÛŜŅŜŜ" pattern).

Independent confirmation (pdfminer extraction from staging PDF):
  Hebrew chars  → IRKPGT+Rubik (variable, ~400)  — correct
  Latin chars   → IEUYIT+Rubik-Bold              — gibberish
And fontTools subset log on staging PDF:
  Glyph names: ['.notdef', 'A', 'B', 'C', 'D', 'E', 'F', 'G',
                'H', 'I', 'L', 'M', 'N', 'O', 'P', 'R', 'S', 'T',
                'U', 'V', 'Y', 'hyphen', 'one', 'periodcentered',
                'space', 'two', 'zero']
  → no lowercase → bug confirmed

WHAT CHANGES IN BEHAVIOR
------------------------
WeasyPrint now sees a single @font-face that spans 300-900 and
points to the variable file. For any requested weight (400, 600,
700, 800), it instances the variable font at that weight. The
single font source means every weight subset shares the same
glyph table — Latin lower/upper, Hebrew, digits, punctuation are
all available. No more font-fallback to DejaVu / no more partial
subset / no more Latin-Extended gibberish on Hebrew or lowercase.

COMMIT B — Background-image for defect photos (object-fit fix)
================================================================

ROOT CAUSE (revealed by follow-up #2 diagnostic)
------------------------------------------------
The diagnostic block added in follow-up #2 fired with great news:

  [PDF] Photo diag: section='כניסה לדירה' item='משקוף' ...
    status=defective item.photos_count=1 defect.proof_urls_count=1
    photo_keys=['photo_0'] fetched_truthy=1/1   ← TRUTHY!

Data layer: completely fine. Photos arrive in template context.

`pdfimages -list latest.pdf` confirms the JPEGs are EMBEDDED
in the PDF (3 images at 600x390). They simply don't render
inside the .photo cells.

The .photo CSS used WeasyPrint's weakest spot:
  .photo     { position: relative; padding-bottom: 75%; }
  .photo img { position: absolute; inset: 0; object-fit: cover; }

WeasyPrint embeds the <img> in the PDF object table but its
`object-fit: cover` + absolute-inside-percentage-padding combo
fails to actually paint the image into the layout box.

THE FIX — switch defect photos to background-image
--------------------------------------------------
WeasyPrint's `background-size: cover` is robust. Move the image
from <img> child to a CSS background-image on the .photo div.

File: backend/templates/handover_protocol_pdf.html

--- Change B.1 — rewrite .photo CSS rule (was lines 979-993) ---

BEFORE:
  .photo {
    display: block;
    position: relative;
    padding-bottom: 75%;
    border-radius: 8px;
    overflow: hidden;
    background: var(--paper);
    border: 1px solid var(--slate-2);
  }
  .photo img {
    position: absolute; inset: 0;
    width: 100%; height: 100%;
    object-fit: cover;
    display: block;
  }

AFTER:
  .photo {
    display: block;
    width: 100%;
    height: 0;
    padding-bottom: 75%;
    border-radius: 8px;
    overflow: hidden;
    background-color: var(--paper);
    background-size: cover;
    background-position: center;
    background-repeat: no-repeat;
    border: 1px solid var(--slate-2);
  }

`.photo img` block is REMOVED entirely (no longer needed because
the new .photo carries its own image via background).

--- Change B.2 — rewrite the defect photo-row Jinja loop ---
Location: line 1377 (was 1385).

BEFORE:
  {% for i in range(3) %}<div class="cell">{%- set p = _ph[i] if i < _ph|length else None -%}{% if p %}<div class="photo"><img src="{{ p }}" alt=""></div>{% else %}<div class="photo-empty"></div>{% endif %}</div>{% endfor %}

AFTER:
  {% for i in range(3) %}<div class="cell">{%- set p = _ph[i] if i < _ph|length else None -%}{% if p %}<div class="photo" style="background-image: url('{{ p }}');"></div>{% else %}<div class="photo-empty"></div>{% endif %}</div>{% endfor %}

DIFF:
  -  ...{% if p %}<div class="photo"><img src="{{ p }}" alt=""></div>{% else %}...
  +  ...{% if p %}<div class="photo" style="background-image: url('{{ p }}');"></div>{% else %}...

Security note: `{{ p }}` is a base64 data URL we built ourselves
in handover_pdf_service.py — safe to inject into a CSS inline
style. Jinja autoescape doesn't apply inside style attributes
to data URLs in this context (the data URL contains no quotes).
