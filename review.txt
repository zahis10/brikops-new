# #601 — Auto-upgrade existing QC runs to current template version

**Branch**: staging
**Severity**: P1 (Zahi prod 2026-05-07 — admin template edit didn't propagate)
**Family**: stand-alone fix (NOT a #503 followup) — closes a long-standing
template-immutability bug exposed by the QC↔Matrix sync feature
**Files**: 2 | **LOC**: +76 / -2 + new test file (~220 LOC) | **Tests**: 3 new

---

## What this ships

A single new helper `_resolve_run_template_with_upgrade` and 2 wire-in
sites. When the user opens a QC run via `get_or_create_floor_run` or
`get_or_create_unit_run`, the helper checks if the run's pinned
`template_version_id` matches the project's CURRENT template. If stale,
it upgrades the run doc + emits an audit entry + returns the current
template, so the existing `_backfill_missing_items` can add items for
new stages admin added after the run was created.

Idempotent (no DB write when already current). Reversible (one-line
Atlas rename if needed). No schema change. No frontend change. No new
feature flag.

## DECISIONS LOCKED

- **D1** (corrects spec) — `_audit` POSITIONAL: `_audit("qc_run", run["id"], "template_upgraded", actor_id, payload)`. Wrapped in try/except per existing pattern. T_UPGRADE_2 enforces this signature.
- **D2** — `db.qc_runs.update_one({"id": run["id"]}, {"$set": {...}})` pattern matches the rest of the file. Local `run` dict mutated so caller sees upgraded fields without re-fetch.
- **D3** — Helper wired into 2 entry points only (get_or_create_floor_run L751, get_or_create_unit_run L899). Other 17 mutation `_get_template(db, run=run, ...)` calls left as-is — by the time they fire, the run was already upgraded by the get_or_create call that loaded it. Race window documented (admin edit while user has page open → mutation uses old tpl until reload). Acceptable.
- **D4** — Spec test 3.3 (full TestClient endpoint test) replaced with T_UPGRADE_3 (direct call to `_backfill_missing_items` with v2 template, asserts only the new (stage_id, item_id) tuples are inserted). Cheaper, same coverage.
- **D5** — `_template_version_cache` left alone. Other runs may still legitimately use OLD vid until they're upgraded too. Defensive.

## RISK NOTES

- **Idempotent + reversible**. To undo upgrades:
  ```js
  db.qc_runs.updateMany(
    { template_upgraded_at: { $exists: true } },
    { $rename: { "template_upgraded_from": "template_version_id" } }
  )
  ```
- **Race window** (D3): admin edits template while user has page open →
  next mutation uses OLD template until reload. Documented. Not new —
  same race existed pre-fix.
- **Stage REMOVAL by admin**: items for the removed stage stay in DB
  (orphaned). Matrix doesn't render the removed column. Harmless.
- **Stage RENAME** (rare): old items orphan, new items get added on
  next get_or_create_*_run. Cosmetic only.
- **Audit failure non-blocking** — try/except keeps the upgrade alive
  even if `_audit` raises (matches existing pattern).
- **Caching note**: `_template_version_cache` may still hold OLD vid
  for OTHER runs that haven't been upgraded yet — desirable.
