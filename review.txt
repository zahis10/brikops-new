STEP 0 — FINDINGS
--------------------------------------------------------------------------------
- Share flows before this batch downloaded/shared a naked QR PNG (fileDownload
  helpers). Recipients received an image with no context and the guard-scan
  flow depended on the saved image.
- Project-name source for S3: SafetyHomePage already loads the project object;
  both SafetyWorkerCard and GuestPassSection are rendered from SafetyHomePage,
  which now passes projectName={project?.name || ''} to each. No new fetch.
- Guest list API (_pass_list_item in guest.py) exposed qr_display_url but no
  gate link; raw token was (correctly) never exposed — kept that way.
- Public gate page (GatePage.js) showed GREEN state text only; no QR on
  screen, so the link alone was not a complete "boarding pass".
- server.py has a global /api/ middleware forcing Cache-Control
  no-cache/no-store on every API response — it overrode the qr.png caching
  headers and needed a scoped exemption (see S5a note below).

S5a — Public QR endpoint (gate_public.py)
  NEW GET /api/gate/{token}/qr.png (unauthenticated, like the gate page):
  per-IP throttle (XFF-aware), 20-128 token length pre-check, active
  worker_entry_tokens OR guest_entry_passes lookup (id-only projection),
  neutral 404 "קוד לא תקף" on any miss, StreamingResponse of the QR PNG
  encoding _gate_url(token) — ZERO information gain (the bytes encode only
  the URL the caller already has). Headers: X-Robots-Tag "noindex, nofollow",
  Cache-Control "private, max-age=3600".
  server.py note: the global /api/ no-store middleware now skips responses
  where the endpoint set an explicit Cache-Control. Only qr.png does; every
  other API response keeps the no-store default (verified by regressions).

S5b — QR on green pass page (GatePage.js)
  GateQrCard (180px img from `${API}/gate/${token}/qr.png`, caption
  "הצג לסורק בכניסה", hides itself on img error) rendered on worker GREEN
  and guest GREEN only. Red/expired/invalid states unchanged.
