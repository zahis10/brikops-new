EARLIER PASS — SPEC COMPLIANCE FIXES (still in place)
────────────────────────────────────────────────────────────────────
The first pass was approved by Zahi with the two explicit deviations
below accepted; subsequent code-review passes added: per-spec sigchip
triplet, flex-wrap layout, per-zero severity hiding, payload-driven
unsigned signature state, tiered compression on local /api/uploads/,
JetBrains Mono fallback stack, and cleanup of stray files + verbose
comments.

  ✓ #1 Legal sigchip format
        Now renders the exact triplet:
            <name> · ת״ז <id_number> · <signed_date>
        — uses global `signed_date` (not per-signer `signed_at`)
        — ת״ז comes from `s.id_number` if present, else looked up
          from the matching tenant by name; falls back to "—"
          (so the dot-separated triplet structure always renders)

  ✓ #2 Legal-sigline + sigchip flex-wrap layout
        .legal-sigline → display:flex; flex-wrap:wrap; align-items:center
        .sigchip       → display:inline-flex; flex-wrap:wrap;
                         align-items:center; (margin-based gaps preserved)
        WeasyPrint 60+ supports display:flex + flex-wrap; chips wrap
        cleanly when many signers exist.

  ✓ #4 Severity-bar per-zero hiding
        Each of the 3 bars (קריטי / רגיל / קוסמטי) now renders
        ONLY when its count > 0. The header itself is hidden when
        all three are zero. Visible bars share width equally
        (50%/50% with two, 100% with one) so the row always feels
        intentional, not "missing a column".

  ✓ #5 Contractor unsigned-state by image_b64 absence
        sig_card macro now flags a card as `.unsigned` when neither
        `sig.image_b64` (canvas) nor `sig.typed_name` (typed) is
        present — no longer driven by a possibly-stale `sig.signed`
        boolean. Validated: a card with `signed=True, type='canvas',
        image_b64=None, typed_name=''` correctly renders dashed/gray.

  ✓ #6 Tiered compression on local /api/uploads/ path
        `_local_image_to_base64` now accepts `max_width` + `jpeg_quality`
        and always pipes data through `_resize_image`. The local
        branch in `_fetch_local_or_remote_image` passes the same
        per-tier params it computes for remote fetches. Result:
        defect/meter/signature uploads are compressed identically
        whether served from S3 or `/api/uploads/`. This was a
        real WhatsApp-size leak before — now closed.
