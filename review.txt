#502 — Execution Matrix .xlsx Export
=====================================

WHAT & WHY
----------
PMs need to send the matrix view to stakeholders / archive snapshots /
work offline. Current options (screenshot, manual rebuild) are unusable
at >50 units. This adds a one-click .xlsx download of the currently-
filtered matrix view, RTL with Hebrew status labels and notes as Excel
comments.

- yarn build: clean (no errors, only the existing chunk-size warnings)

CHANGES
-------
backend/contractor_ops/execution_matrix_router.py
  PART 0 — hoisted `_summarize_cell` from nested-in-get_matrix to
    module level (L189-207). Same shape, same behavior. Both /matrix
    and /export.xlsx now consume it. Get_matrix call site at L309
    unchanged in semantics, just calls the module-level fn.
  PART 2 — new POST /{project_id}/export.xlsx endpoint (L623-728).
    RBAC via _check_matrix_view (PM/super_admin/owner/management_team;
    contractor 403). Hard cap of 2000 units → 400 with Hebrew detail.
    Filters honored via payload {unit_ids, stage_ids} (empty=all).
    RFC 5987 Content-Disposition with Hebrew filename* + ASCII fallback.
    Filename project-name sanitized (re.sub r'[/\\:*?"<>|]' → '_').

backend/contractor_ops/execution_matrix_export.py  (NEW, 144 LOC)
  Pure builder — no DB / FastAPI. Single sheet "מטריצת ביצוע":
    - RTL view, frozen E2 (header row + 3 location columns).
    - Status cells: Hebrew label from STATUS_LABELS + light fill from
      STATUS_FILLS (mirrors frontend STATUS_CONFIG ramp).
    - Tag cells: text_value as plain text.
    - Notes attached as openpyxl Comment with last_actor_name as author
      (fallback "BrikOps").
  _clean_note: strip + control-char regex r'[\x00-\x08\x0b-\x1f]' +
  500-char cap. Preemptive guard against IllegalCharacterError when PMs
  paste from Word/Email/WhatsApp (Fix #2 from approval).

  T1 build_matrix_xlsx structure (sheet name, RTL, frozen, header row).

  T4 endpoint RBAC (contractor 403, PM 200 with StreamingResponse).
  T5 filename sanitization (slash, colon → underscore in RFC 5987 encoded

frontend/src/services/matrixService.js
  exportXlsx(projectId, {unit_ids, stage_ids}) → {blob, filename}.

frontend/src/pages/ExecutionMatrixPage.js
  Download button before Settings in header (always visible — no canEdit

ALL 3 PRE-T002 FIXES APPLIED
----------------------------
1. CRITICAL — _summarize_cell hoisted to module level. Both endpoints
   share the same canonical shape. No NameError risk.
2. PART 1 preemptive control-char strip in _clean_note. PM-pasted notes
   with NULL/VT/etc no longer crash openpyxl.
3. PART 4 preemptive blob-error parsing. Backend HTTPException details
   surface to user instead of generic message.

DO NOT
------
- Do NOT modify _summarize_cell signature or output shape — both
  endpoints depend on identical fields (especially last_actor_name).
- Do NOT remove the 2000-unit cap without raising openpyxl memory limits
  in the deploy env.
- Do NOT switch responseType from 'blob' without removing the blob-error
  parsing in handleExport.

OUT OF SCOPE
------------
- Server-side persistent export jobs (immediate StreamingResponse only).
- PDF export (separate ticket).
- Per-user export rate limiting (no abuse pattern observed yet).
- Multi-sheet Excel (single sheet matches the matrix view 1:1).

FAILURE PATHS COVERED
---------------------
1. Contractor tries to export → 403 (T4a).
2. Project >2000 units → 400 with Hebrew detail surfaced via blob-parse
   to alert (T5b + handleExport catch).
3. Note has control chars → stripped, no IllegalCharacterError (T3).
4. Hebrew project name with /:?* → sanitized in filename* (T5a).
5. Network/blob-decode failure → generic Hebrew alert fallback.
