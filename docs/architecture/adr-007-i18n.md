# ADR-007 — i18n (cross-cutting)

**Status:** Accepted
**Date:** 2026-04-29
**Scope:** `frontend/src/i18n/index.js`, `frontend/src/i18n/he.json`
(686 lines, source of truth), `frontend/src/i18n/en.json` /
`ar.json` / `zh.json` (frozen pre-launch), every component that
displays user-facing text.

## Why this exists

BrikOps is a Hebrew-first product for Israeli construction projects.
Every screen is RTL, every label is Hebrew, every user is on a
WhatsApp-flavored Hebrew text dialect. The i18n module wraps this in
a way that lets us:

1. **Centralize Hebrew text** so editors and PMs can change wording
   without touching JSX.
2. **Keep hooks for future locales** (en/ar/zh) without paying
   maintenance cost today.
3. **Catch missing translations early** — dev mode warns when a key
   is missing.

The "internationalization" here is mostly **localization to one
locale**, with the multi-locale infrastructure pre-wired for the day
we expand to Arabic-speaking contractors or English-speaking auditors.

## Architectural decisions

- **Decision: a thin module-level `t()` function, not React Context.**
  Why: i18n is read-only, stable per session, and called from
  everywhere — utility files, services, modal helpers, not just
  components. Putting it in Context would force every consumer to be
  inside a React tree (services aren't) AND would re-render the whole
  tree on language change (we don't have a language switcher
  pre-launch). The module-level singleton in
  `frontend/src/i18n/index.js` matches the actual usage.

- **Decision: section-based key namespacing.** Every key is
  `t('section', 'key')` — e.g. `t('statuses', 'closed')`,
  `t('roles', 'project_manager')`. Why: groups related keys for the
  editor, prevents collisions, and lets us add typed shortcut helpers
  per section (`tStatus`, `tCategory`, `tPriority`, `tRole`,
  `tSubRole`, `tTrade`).

- **Decision: dev-mode console.warn on missing keys.** When
  `process.env.NODE_ENV === 'development'` and the current locale
  doesn't have the key, the helper warns with a fallback note. Keeps
  missing translations visible during development without breaking
  production renders.

- **Decision: he is the fallback for every locale.** If a key is
  missing in the current locale, fall back to `he` (not en). Why:
  Hebrew is the source of truth and the most-curated locale; falling
  back to English would introduce mixed-language UI.

- **Decision: en/ar/zh JSON files are frozen pre-launch.** They exist
  (438-439 lines each) but are not updated as new keys are added to
  he.json (now 686 lines). Why: maintaining four locales pre-launch
  triples the translation cost without user benefit — the only users
  are 41 Israeli internal testers. Post-launch decision: either drop
  the frozen files or fund a translation pass.

- **Decision: PM/admin UI is forced Hebrew.** Even if `setLanguage()`
  is called with another locale, key paths under
  `pages/Admin*` and management dialogs render Hebrew because every
  string in those code paths is inline (not via `t()`). Why: super-
  admin UI is internal-only and never gets internationalized.

- **Decision: RTL is via `dir="rtl"` on the page root container, not
  via CSS.** Why: Tailwind's RTL utilities (`rtl:`, `ltr:`) compose
  cleanly with `dir`; CSS-only RTL would fight the Tailwind defaults.
  Inputs that should stay LTR (phone numbers, emails, English IDs)
  opt out with `dir="ltr"` on the field.

## Conventions used here

- **Add a new user-facing string?** Add to `he.json` first under the
  right section, then call via `t('section', 'key')`. Other locales
  intentionally drift.
- **Use the typed shortcut where one exists.** `tStatus('closed')`
  beats `t('statuses', 'closed')` — same effect, less boilerplate,
  catches typos faster.
- **Inline Hebrew for one-off labels** is acceptable for UI strings
  that are stable (page titles, button labels in admin tools). When
  the label appears in 3+ places, move to `he.json`.
- **Never** `setLanguage()` from user code in production builds — the
  language switcher isn't shipped pre-launch and switching mid-session
  causes mixed-language renders for any string that was already
  rendered.

## What's in he.json (sections)

Inspecting `frontend/src/i18n/he.json` (686 lines, ~33 KB):

- `statuses` — task status display labels (closed → "סגור", etc.)
- `categories` — defect category labels (electrical → "חשמל", etc.)
- `priorities` — priority labels (critical → "קריטי", etc.)
- `roles` — project role labels (project_manager → "מנהל פרויקט", etc.)
- `subRoles` — project sub-role labels.
- `trades` — contractor trade labels (~80 entries — the longest section).
- Plus per-page sections for static labels.

## Frontend duplication risk (refactor backlog)

The `STATUS_LABELS` map is currently re-declared in 8 frontend files:
`AdminUsersPage`, `HandoverTabPage`, `AdminActivityPage`,
`AdminDashboardPage`, `BuildingDefectsPage`, `ApartmentDashboardPage`,
`UnitDetailPage`, `OrgBillingPage`, plus `qcLabels.js` and
`billingLabels.js`. After BATCH 5C/5F/5G these are aligned, but each
page declares its own copy.

The right fix is to consume `tStatus()` everywhere instead of a
local map. Tracked as refactor-backlog item #6.

## Anti-patterns

- **Adding strings inline outside the i18n helper** when the same
  string appears in multiple places. The Editor's voice changes;
  inline strings drift.
- **Calling `t()` outside React render where it might run before
  `setLanguage()` is called.** The module defaults to `he`, so this
  is benign today, but if a user-language-detect runs at app start,
  early calls would render in Hebrew even for non-Hebrew users.
- **Persisting Hebrew text to DB.** Hebrew is a display layer.
  Backend stores English keys (statuses, roles, categories) — UI
  translates on render.
- **Persisting localized labels in component props.** Pass the key,
  not the label. Reusable components should call `tStatus(status)`
  themselves; props should be the key.

## When to refactor

- **Trigger:** when we ship a language switcher (en or ar). At that
  point the dev-mode warning surfaces will become real bugs and the
  frozen JSON files need a curation pass.
- **Trigger:** when the duplicated `STATUS_LABELS` maps cause a
  mismatch (post-5G they're aligned; one drift incident is enough
  to motivate centralizing — refactor-backlog item #6).
- **Pre-requisite:** decide whether labels live in code (current) or
  in a CMS (PMs editing via UI). The current code-based approach is
  fine for small label sets.
