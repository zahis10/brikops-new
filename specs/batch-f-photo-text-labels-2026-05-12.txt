BATCH F — Photo annotation: text labels (in addition to freehand drawing)
================================================================================
Date:           2026-05-12
Branch:         staging  (NOT main)
Severity:       P3 — feature request from field use (Zahi 2026-05-11).
                Currently `PhotoAnnotation.js` only supports freehand
                strokes in 3 colors (red/blue/black). PMs annotating
                defect photos in the field want to add SHORT TEXT
                labels — e.g. defect numbers ("5/27"), measurement
                values, apartment/floor callouts — in a clean pill
                style on top of the photo.

                Reference: Zahi's screenshot 2026-05-11 showing the
                desired output — green oval (existing freehand) +
                white "5/27" pill above (the new text feature).

Approach:       Extend `PhotoAnnotation.js` with a "text mode" tool
                added next to the color picker. In text mode, tapping
                the canvas opens an inline input overlay (NOT
                `prompt()` — that's ugly on mobile). User types short
                text, taps "אישור" → text renders on canvas as a
                white-background rounded pill at the tap point.

                Storage extends the existing `strokes` array with a
                discriminator `type: 'stroke' | 'text'`. No backend,
                no schema change — both types serialize into the same
                canvas-blob that already gets uploaded.

Status:         AWAITING ZAHI APPROVAL — DO NOT DEPLOY

Files changed:  Frontend only:
                  - frontend/src/components/PhotoAnnotation.js
                    (~120-150 LOC added: tool toggle state, "T"

                Total: 1 file. No backend, no DB, no env vars, no new

CROSS-PLATFORM SAFETY (lessons from earlier iPhone/Android pain)
================================================================================

The existing PhotoAnnotation.js already encodes hard-won cross-platform
knowledge (viewport lock, passive event listeners, scale compensation,
separate touch/mouse handlers). v1 PRESERVES all of these. New code
adds:

1. **Hebrew text on canvas (the main new risk):**
   - Explicit font-family stack ensures consistent rendering iOS vs
     Android. Heebo first (project's primary Hebrew font, already
     loaded), then Arial Hebrew (iOS fallback), then Noto Sans Hebrew
     (Android fallback).
   - `ctx.direction = 'rtl'` set before rendering Hebrew text so RTL
     layout is honored (digits and punctuation positioned correctly).

2. **Input field on iOS (auto-zoom + keyboard popup):**
   - Input has `font-size >= 16px` to prevent iOS auto-zoom.
   - Input is `position: fixed` near the BOTTOM of screen (above
     action buttons), NOT at the tap point. Keyboard popping up
     doesn't push it off-screen.
   - Existing viewport lock (`maximum-scale=1, user-scalable=no`)
     prevents pinch-zoom interfering.

3. **Tap targets and touch events:**
   - "T" toolbar button: 44x44 minimum (Apple HIG).
   - Overlay input opens/closes via existing touch event flow (no new
     event listener types — reuses the existing `touchstart`/`mouseup`
     handlers, just branches on `tool` state).

4. **roundRect() browser support:**
   - Safari 16+ / Chrome 99+ supported. Capacitor WebView matches
     device's Safari version → all modern iPhones + Androids covered.
   - Fallback path: if `ctx.roundRect` is undefined, fall back to
     `ctx.rect` (square pill — less pretty but functional). v1
     includes the feature-detect.

AFTER:
    for (const stroke of allStrokes) {
      // BATCH F (2026-05-12) — type discriminator. Default 'stroke'
      // for backward compat with any in-memory strokes from before
      // the migration.
      const strokeType = stroke.type || 'stroke';

      if (strokeType === 'text') {
        // Render centered pill at (x, y) with white background +
        // colored text. Font scaled relative to the canvas size.
        const fontSize = TEXT_FONT_SIZE / (scaleRef.current || 1);
        ctx.font = `bold ${fontSize}px ${FONT_STACK}`;
        ctx.direction = 'rtl';  // Hebrew layout
        ctx.textBaseline = 'middle';
        ctx.textAlign = 'center';

        const padX = TEXT_PADDING_X / (scaleRef.current || 1);
        const padY = TEXT_PADDING_Y / (scaleRef.current || 1);
        const radius = TEXT_PILL_RADIUS / (scaleRef.current || 1);

        const metrics = ctx.measureText(stroke.text || '');
        const textWidth = metrics.width;
        const textHeight = fontSize;  // approximate
        const rectWidth = textWidth + padX * 2;
        const rectHeight = textHeight + padY * 2;
        const rectX = stroke.x - rectWidth / 2;
        const rectY = stroke.y - rectHeight / 2;

        // Pill background — white with subtle shadow for legibility.
        ctx.fillStyle = '#ffffff';
        ctx.beginPath();
        if (typeof ctx.roundRect === 'function') {
          ctx.roundRect(rectX, rectY, rectWidth, rectHeight, radius);
        } else {
          // Fallback for very old browsers — square rect.
          ctx.rect(rectX, rectY, rectWidth, rectHeight);
        }
        ctx.fill();

        // Border for contrast on white-on-white backgrounds.
        ctx.lineWidth = Math.max(1, 1.5 / (scaleRef.current || 1));
        ctx.strokeStyle = '#1f2937';  // slate-800
        ctx.stroke();

        // Text on top.
        ctx.fillStyle = stroke.color || '#000000';
        ctx.fillText(stroke.text || '', stroke.x, stroke.y);
        continue;
      }

      // Default: existing freehand stroke rendering.
      if (!stroke.points || stroke.points.length < 2) continue;
      ctx.beginPath();
      ctx.strokeStyle = stroke.color;
      ctx.lineWidth = lw;
      ctx.lineCap = 'round';
      ctx.lineJoin = 'round';
      ctx.moveTo(stroke.points[0].x, stroke.points[0].y);

LOCATION 3 — PhotoAnnotation.js: branch event handlers on `tool` mode

File: frontend/src/components/PhotoAnnotation.js
Lines: ~162-247 (the useEffect that attaches touch/mouse handlers)

In `startStroke`, `handleTouchStart`, `handleMouseDown`: branch on
`toolRef.current`. When `'text'`, instead of starting a stroke, just
set `pendingText` to the tap position.

BEFORE (startStroke):
    const startStroke = (pos) => {
      const newStroke = { color: colorRef.current, points: [pos] };
      currentStrokeRef.current = newStroke;
      drawingRef.current = true;
    };

AFTER:
    const startStroke = (pos) => {
      // BATCH F (2026-05-12) — branch on tool.
      if (toolRef.current === 'text') {
        // Open inline overlay for text input at this position.
        // If pendingText already exists (user tapped, started typing,
        // then tapped a different spot to reposition before
        // committing), PRESERVE the typed value and only update the
        // position. Matches documented intent in the risk section.
        setPendingText(prev =>
          prev
            ? { ...prev, x: pos.x, y: pos.y }   // reposition, keep value
            : { x: pos.x, y: pos.y, value: '' } // first tap
        );
        return;
      }
      const newStroke = {
        type: 'stroke',
        color: colorRef.current,
        points: [pos],
      };
      currentStrokeRef.current = newStroke;
      drawingRef.current = true;
    };

`moveStroke` and `endStroke` stay UNCHANGED — they only act when
`drawingRef.current === true`, which the text branch never sets.

DESIGN DECISION (preserve-on-reposition):
- First tap in text mode opens overlay with empty value.
- Subsequent taps BEFORE committing reposition the pending text but
  PRESERVE what user has typed. Intuitive: "I tapped wrong spot,
  let me move it before saving" — without losing typed input.
- To start fresh, user taps "ביטול" → closes overlay → next tap
  starts new pendingText.

================================================================================
RISK
================================================================================
Low-medium. Frontend-only, additive feature.

Risk vectors:
- ⚠ Hebrew rendering inconsistencies iOS vs Android — mitigated by
  explicit font stack + ctx.direction='rtl'. Worst case: text
  visible but font slightly different per platform. Acceptable for
  v1 ship.
- ⚠ iOS keyboard popping up shifts viewport — mitigated by
  position:fixed input at bottom + font-size>=16px to prevent
  auto-zoom + existing viewport lock.
- ⚠ roundRect() unsupported on very old browsers — fallback to
  regular rect. Square pill instead of rounded. Functional, less
  pretty. Affects <1% of users.
- ⚠ Text overflowing image bounds if tap is near the edge — text
  may extend past the canvas. Acceptable for v1; clipping can be
  added in a follow-up.
- ⚠ User taps multiple times in text mode before committing — only
  one pendingText at a time; second tap REPOSITIONS the pending text
  (x, y updated) while PRESERVING the typed value. Matches intuitive
  "let me move it before saving" UX. To start fresh, user must tap
  "ביטול" first.
- ⚠ User changes color while typing before committing — text uses
  color at COMMIT time (final intent), not at tap time. See design
  decision in LOCATION 5 `commitPendingText` comments.

Bundle size impact: zero (no new imports, no new deps).
Performance impact: negligible (text rendering is O(1) per entry).
