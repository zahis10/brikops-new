Bug B: item-level activity DOES NOT propagate in floor-scope
  Zahi: "נניח והתחלתי לסמן משהו אחד בתוך נושא מסוים (אינסטלציה
        לדוגמא), אמור להתחלף לי במטריצה לבעבודה, זה לא קורה"

  Translation: When site manager marks ONE item inside a floor-scope
  stage (e.g. אינסטלציה), the matrix should turn blue (in_progress)
  for ALL units on that floor. It doesn't.

  Root cause analysis:
    - update_qc_item hook fires for floor-scope item PATCH ✓
    - _resolve_unit_ids_for_sync correctly returns all units on floor
      (via fallback to db.units.find by floor_id) ✓
    - For each unit, hook tries to filter items_all by unit_id
    - BUT items in floor-scope DON'T have unit_id field
    - Filter returns [] empty list per unit
    - sync_qc_stage_to_matrix called with items=[] → returns None
      (because `any(i.get("status") in ("pass", "fail") for i in [])`
       is False, AND stage_status is None)
    - No upsert → matrix stays empty

  Fix: when items don't have unit_id (floor-scope template-level),
  pass the COMPLETE items_all list to each unit's sync (not a
  per-unit filtered subset). Items represent the floor's status —
  shared across all units.

Bug C: submit-for-review missing hook
  Zahi: "מנהל שלח לאישור מנהל, אבל אני לא רואה את זה במטריצה"

  Translation: Site manager submits stage for review (sets stage
  status to "pending_review"). Matrix should turn ORANGE
  "ממתין לאישור". It doesn't.

  Root cause: Endpoint `POST /run/{run_id}/stage/{stage_id}/submit`

    - update_qc_item (#503)
    - approve_stage (#503)
    - override_approve_stage (#503)
    - reject_stage (#503-followup-2)

  Missing the 5th endpoint that changes stage state: submit_stage.

  Fix: add hook to submit_stage with stage_status="pending_review".

PART 1 — Fix Bug B: items_all vs per-unit filter
================================================================================

FILE: backend/contractor_ops/qc_router.py

LOCATION 1.1 — In the update_qc_item hook (and possibly approve/reject/
override hooks if they have the same pattern), change the per-unit
items filtering to fall back to items_all when items lack unit_id.

Current pattern (estimated — verify in STEP 0.2):
    for uid in unit_ids:
        items_for_uid = [i for i in items_all if i.get("unit_id") == uid]
        background_tasks.add_task(
            sync_qc_stage_to_matrix, ...,
            unit_id=uid, qc_items=items_for_uid, ...
        )

NEW pattern:
    # Detect whether items carry unit_id (unit-scope) or are
    # floor-shared (template-level, no unit_id).
    items_have_unit_id = any(i.get("unit_id") for i in items_all)

    for uid in unit_ids:
        if items_have_unit_id:
            # Per-unit items: filter to this unit's items
            items_for_uid = [i for i in items_all if i.get("unit_id") == uid]
        else:
            # Floor-shared items: all units share the same items
            items_for_uid = items_all

        background_tasks.add_task(
            sync_qc_stage_to_matrix, ...,
            unit_id=uid, qc_items=items_for_uid, ...
        )

Apply this pattern in ALL 4 endpoints that use _resolve_unit_ids_for_sync:
  - update_qc_item
  - approve_stage
  - override_approve_stage
  - reject_stage

If they currently use a single line `items_for_uid = items_all`
already (because Replit caught this in followup-2), document and
move on — Bug B might already be fixed via that path. Verify in
STEP 0.2.

Q1. ready_for_work persistence policy — DECIDED 2026-05-07.
    Zahi: "נשאיר את א', בקרת ביצוע תדרוס אותו זה בסדר לדעתי".

    DECISION: Option A — QC ALWAYS WINS over ready_for_work, same
    as for any other matrix status. D3 from #503 applies uniformly.

    Reasoning: ready_for_work is a planning signal. Once actual work
    begins (any QC activity), the operational state of "in_progress"
    is more relevant to the PM than the prior planning marker. PM
    can re-set ready_for_work later if needed.

    No additional helper logic required — current mapping handles
    this correctly via the standard "QC wins" path.

    T18 finalized: "test_ready_for_work_overridden_by_qc_per_d3"
    (already updated in PART 4 above).

Q2. Visual distinctness — cyan vs blue vs teal on mobile:
    cyan (ready_for_work)  #CFFAFE
    blue (in_progress)     #DBEAFE
    teal (no_findings)     #CCFBF1

    All very light, similar tones. On a 375px iPhone screen with
    32×32 cell icons, these may be hard to differentiate without
    relying on the icons (PackageCheck vs Clock vs Shield-Check).

    Recommendation: ship as designed; gather PM feedback after 1
    week of staging. If confused, simplify in v2 (e.g., consolidate
    cyan + teal, or use stronger color saturation).
