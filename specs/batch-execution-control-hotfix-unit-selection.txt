ROOT CAUSE (verified 2026-05-04 from staging smoke)
================================================================================

#476 changed the `/floors/{floor_id}/units-status` endpoint response from:
  OLD: { units: [{unit_id, status, handled_count, total, ...}, ...] }
  NEW: { stages: [{stage_id, stage_title, units: [...]}, ...] }

Replit's STEP 0.3 grep searched for the string `units-status` and concluded
"ONLY consumer = api.js:1329 (3-line axios passthrough)". This was correct
about the ENDPOINT URL but missed the SERVICE METHOD (`qcService.getUnitsStatus`)
which is consumed by TWO pages:

  ✅ frontend/src/pages/FloorDetailPage.js:140 — UPDATED in #476
  ❌ frontend/src/pages/UnitQCSelectionPage.js:33 — MISSED, still expects `data.units`

Result: when user clicks any unit-scope card on the floor view → navigates
to UnitQCSelectionPage → page calls qcService.getUnitsStatus → receives
new response shape with `stages:[]` instead of `units:[]` → page tries
`data.units.filter(...)` on undefined → React error boundary → "משהו השתבש".

LESSON for future spec writing — STEP 0 verification of "single consumer"
must search for both:
  1. The endpoint URL string (e.g. "units-status")
  2. The service method name that wraps it (e.g. "getUnitsStatus")
Updating the SPEC TEMPLATE to enforce this. Tracked separately.

FILE: frontend/src/pages/UnitQCSelectionPage.js
------------------------------------------------

DESIGN DECISION (Zahi 2026-05-04 — chose Option B):

UnitQCSelectionPage is a "select which unit to enter" picker — it shows a
list of units on the floor with aggregate completion status per unit. The
status logic is "is this unit done?" not "is this unit done for stage X?".

This hotfix implements BOTH:

  Path A — flatten across all unit-scope stages (data adapter):
    For each unit, sum handled/total across all stages, derive aggregate
    status (approved if all stages approved; in_progress if any handled;
    not_started otherwise). Preserves existing UX of the page.

  Path B — propagate ?stage_id=X URL param to StageDetailPage:
    #476 already added ?stage_id=X to FloorDetailPage onClick navigation.
    This hotfix completes the wiring: when user clicks a unit-scope card
    (e.g. "אמבטיה+אינטרפוץ") on the floor, picks a unit on this page, the
    user lands on the SAME stage they originally clicked — not on the
    first stage of the unit run (which today is hardcoded to stage_tiling).

    If no ?stage_id= is in the URL (legacy entry — direct navigation,
    bookmarks, etc.), behavior is unchanged: defaults to first stage of
    unit run, fallback "stage_tiling".

    3 extra LOC, zero risk. Closes the user-flow loop that #476 opened.

LOCATION 1.1 — Replace the data extraction + aggregation block at lines
   ~95-99.

   FIND (current — broken):
       if (!data) return null;

       const { units, floor_name } = data;
       const completedCount = units.filter(u => u.status === 'approved').length;
       const totalHandled = units.reduce((s, u) => s + u.handled_count, 0);
       const totalItems = units.reduce((s, u) => s + u.total, 0);
       const overallPct = totalItems > 0 ? Math.round((totalHandled / totalItems) * 100) : 0;

   REPLACE WITH (flatten stages → unit aggregates):
       if (!data) return null;

       // Hotfix (post-#476): the units-status endpoint now returns
       // { stages: [{units: [...]}] } instead of { units: [...] }.
       // We flatten stages back to per-unit aggregates here so the
       // existing unit-picker UX stays intact. PATH B (per-stage view
       // when ?stage_id=X is present in URL) is a future polish batch.
       const stages = Array.isArray(data.stages) ? data.stages : [];
       const floor_name = data.floor_name || '';

       // Aggregate across all unit-scope stages, per unit:
       //   - handled_count, total, pass_count, fail_count → SUM
       //   - status: 'approved' only if ALL stages approved for this unit;
       //             'in_progress' if any handled OR any in_progress;
       //             'not_started' otherwise
       const unitAggMap = new Map();
       stages.forEach(stage => {
         (stage.units || []).forEach(u => {
           const existing = unitAggMap.get(u.unit_id);
           if (!existing) {
             unitAggMap.set(u.unit_id, {
               unit_id: u.unit_id,
               unit_name: u.unit_name,
               unit_no: u.unit_no,
               handled_count: u.handled_count || 0,
               total: u.total || 0,
               pass_count: u.pass_count || 0,
               fail_count: u.fail_count || 0,
               _statuses: [u.status || 'not_started'],
             });
           } else {
             existing.handled_count += u.handled_count || 0;
             existing.total += u.total || 0;
             existing.pass_count += u.pass_count || 0;
             existing.fail_count += u.fail_count || 0;
             existing._statuses.push(u.status || 'not_started');
           }
         });
       });

       // Derive aggregate unit status from per-stage statuses
       const units = Array.from(unitAggMap.values()).map(u => {
         const allApproved = u._statuses.length > 0 && u._statuses.every(s => s === 'approved');
         const anyInProgress = u._statuses.some(s => s === 'in_progress' || s === 'approved');
         const status = allApproved ? 'approved'
                      : anyInProgress ? 'in_progress'
                      : 'not_started';
         const { _statuses, ...rest } = u;
         return { ...rest, status };
       });

       const completedCount = units.filter(u => u.status === 'approved').length;
       const totalHandled = units.reduce((s, u) => s + u.handled_count, 0);
       const totalItems = units.reduce((s, u) => s + u.total, 0);
       const overallPct = totalItems > 0 ? Math.round((totalHandled / totalItems) * 100) : 0;
