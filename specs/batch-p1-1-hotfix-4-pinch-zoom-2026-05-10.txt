CONTEXT — root cause analysis
=============================

The root cause of #32 (no pinch on iPhone) AND #33 (pixelated text on
browser pinch) is identical: the current viewer relies on the BROWSER's
native pinch-zoom. There are two failure modes depending on context:

  Mode A — In Capacitor app (WKWebView), or iOS Safari with the modal
  fullscreen: browser-level pinch may be disabled or scoped, so the
  user sees no zoom response at all → reads as "pinch broken".

  Mode B — In normal Safari, browser-level pinch DOES fire, but it's a
  CSS `transform: scale()` on the rasterized canvas. The canvas was
  rendered by react-pdf at `scale={0.6}` (fit-scale) producing a
  ~1500x1000 px image. Browser pinch to 200% turns it into a
  3000x2000 px image VISUALLY but the underlying pixels haven't
  changed → blurry/pixelated. react-pdf is unaware of the pinch and
  doesn't re-render the page at higher scale.

The same fix solves both: capture the pinch gesture in our app, update
the `scale` state programmatically, and let react-pdf re-render at the
new scale (which produces a sharp 3000x2000 canvas natively).

CONTEXT — chosen approach
=========================

Library: **@use-gesture/react** v10.3+
  - 13KB gzipped, used by Spring/Framer Motion
  - React 16.8+ → React 19 supported via stable APIs (useRef, useEffect)
  - Clean unified API for pinch + drag + wheel
  - **Cross-platform, both iOS AND Android** — see "Platform coverage"
    section below for details

Platform coverage (both required to ship):
  - **iOS Safari + Capacitor (WKWebView)** — primary target. iOS has
    its own GestureEvent API in addition to touch events; @use-gesture
    detects and uses whichever fires.
  - **Android Chrome + Capacitor Android (Chromium WebView)** — also
    primary target. Android only has touch events (no GestureEvent);
    @use-gesture computes pinch ratio from two-touch positions.
  - **Desktop browsers** — wheel zoom (Ctrl/⌘ + scroll) and Magic
    Trackpad pinch are also handled by @use-gesture's wheel/pinch
    detection. Already works via existing +/- buttons.

Why @use-gesture handles both:
  - On iOS: subscribes to `gesturestart/gesturechange/gestureend`
    (Apple-specific) AND `touchstart/touchmove/touchend` as fallback.
  - On Android: subscribes to `touchstart/touchmove/touchend` and
    computes pinch ratio from `event.touches[0]` and `event.touches[1]`
    coordinates each frame.
  - On desktop: subscribes to `wheel` events for Ctrl-scroll zoom +
    pointer events for trackpad pinch.

This means the SAME `usePinch` hook fires for all three. Our handler
reads `offset[0]` (the scale ratio) regardless of source — touch,
gesture, or wheel.

Two changes:
  1. Update `touch-action` from `'pan-x pan-y pinch-zoom'` to
     `'pan-x pan-y'`. This DISABLES browser-native pinch on the
     container so we own the gesture. One-finger pan/scroll still
     works (that's `pan-x pan-y`).
  2. Add `usePinch` hook bound to `containerRef`. On pinch, update
     `scale` state with multiplicative delta (matching the existing
     ×1.25 / ×0.8 zoom button convention from hotfix-1).

Drag-to-pan (one-finger drag to pan) is OUT of scope for this hotfix.
The visible scrollbars from hotfix-2 + native one-finger touch scroll
are sufficient for pan. Adding drag-to-pan would risk conflicting with
text selection, link clicks, and annotation interactions in P1.2.
Defer to P1.2 if needed.

LOCATION 1.3 — Add usePinch hook inside the PlanViewer component
                 (after existing useEffect that resets state on plan
                 change, ~L43, BEFORE handleLoadSuccess)

   ⚠️ CRITICAL — Pinch math correctness:

   `offset[0]` from @use-gesture/react usePinch is **cumulative from
   gesture start**, not frame-to-frame delta. If we naively multiply
   it by the LATEST `scale` state, we get exponential drift:

     Frame 1: offset=1.10, prev=1.0  → new=1.10  ✓
     Frame 2: offset=1.20, prev=1.10 → new=1.32  ❌ (should be 1.20)
     Frame 3: offset=1.30, prev=1.32 → new=1.72  ❌ (should be 1.30)

   The fix is to anchor `offset` to the scale value AT GESTURE START,
   not the live `scale` state.

   Two valid patterns; **Replit should pick whichever the verified v10
   API supports** (per STEP 0.6).

   ── PATTERN A — `from` config (cleanest, if v10 supports it) ───────

   With `from: () => [scale ?? 1.0, 0]`, @use-gesture treats `offset`
   as the ABSOLUTE current pinch state — anchored to whatever `from`
   returned at gesture start. `offset[0]` IS the new scale directly,
   not a multiplier.

     usePinch(
       ({ offset: [s], event }) => {
         if (event && event.cancelable) event.preventDefault();
         setScale(Math.min(Math.max(s, 0.1), 4.0));
       },
       {
         target: containerRef,
         eventOptions: { passive: false },
         scaleBounds: { min: 0.1, max: 4.0 },
         rubberband: true,
         from: () => [scale ?? 1.0, 0],
       }
     );

   ── PATTERN B — explicit ref + `first` flag (always works) ────────

   If `from` config isn't supported in the installed v10, use a ref
   to remember the scale at gesture start. The handler reads
   `gestureStartScaleRef.current` (frozen at gesture start) and
   multiplies by `offset[0]` (cumulative ratio).

     // Add ref ABOVE the usePinch call (alongside other refs):
     const gestureStartScaleRef = useRef(1.0);

     usePinch(
       ({ offset: [s], first, event }) => {
         if (event && event.cancelable) event.preventDefault();
         if (first) {
           gestureStartScaleRef.current = scale ?? 1.0;
         }
         const next = gestureStartScaleRef.current * s;
         setScale(Math.min(Math.max(next, 0.1), 4.0));
       },
       {
         target: containerRef,
         eventOptions: { passive: false },
         scaleBounds: { min: 0.1, max: 4.0 },
         rubberband: true,
       }
     );

   ── Why both patterns are correct ─────────────────────────────────

   PATTERN A: `from` re-evaluates at each gesture start; offset is
   relative to that anchor; absolute new scale comes out directly.
   PATTERN B: ref captures scale at gesture start (`first === true`);
   each frame multiplies that frozen value by the cumulative offset
   ratio. Same math, different mechanism.

   ── DO NOT do this (the broken pattern from spec v1) ─────────────

     setScale(prev => {
       const next = (prev ?? 1.0) * s;  // ← compounding error
       return Math.min(Math.max(next, 0.1), 4.0);
     });

   This compounds because `prev` mutates between frames, and `s` is
   already cumulative. Avoid.

   ── Common context for both patterns ──────────────────────────────

   - target: containerRef → gesture bound to scrolling div via
     addEventListener (bypasses React synthetic event system, works
     with non-passive event option needed for preventDefault).
   - eventOptions.passive: false → required for preventDefault to
     stop browser-level pinch on iOS Safari edge cases.
   - scaleBounds [0.1, 4.0] → matches +/- button bounds.
   - rubberband → soft-bounce visual feedback when pinching past
     bounds (the "I've hit the limit" feeling).
   - usePinch returns no value when bound via `target`; it's a
     side-effect hook. Just call and discard the return.

   API DRIFT FALLBACK: if both `from` config AND `first` flag are
   absent in v10 (unlikely), use `delta` (frame-to-frame ratio change)
   and accumulate manually each frame:
     setScale(prev => {
       const ratio = delta[0];  // small frame-to-frame multiplier
       return Math.min(Math.max((prev ?? 1.0) * ratio, 0.1), 4.0);
     });
   This is mathematically equivalent because each frame multiplies the
   PREVIOUS scale by the FRAME's small delta — no cumulative offset
   confusion.
