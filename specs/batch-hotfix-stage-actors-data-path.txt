ROOT CAUSE — verified 2026-05-04 via DevTools Network preview
================================================================================

Actual API response shape from `GET /api/qc/run/{run_id}` on staging
(production code path, observed live by Zahi):

   {
     "can_edit": true,
     "role": "project_manager",
     "run": {
       "id": "6adb55dc-...",
       "project_id": "616c30e0-...",
       "building_id": "...",
       "floor_id": "...",
       "scope": "floor",
       "stage_actors": {                  ← HERE — nested in run
         "stage_ceiling_prep": { "submitted_by": "...", ... },
         "stage_plaster": { ... }
       },
       "stage_statuses": { ... },
       "template_id": "..."
     },
     "stages": [...],
     "summary": {...},
     "template_name": "..."
   }

`stage_actors` is on `runData.run.stage_actors`, NOT on `runData.stage_actors`.

Frontend code from #478 (badges) and #479 (audit block) consistently reads:

   runData?.stage_actors?.[stageId]?.via_override            ← undefined ALWAYS

Resulting in:
- #478 top status badge "אושר ב-Override על־ידי X" → never renders
- #478 bottom action-bar badge → never renders
- #479 amber audit block with reason → never renders

The override action ITSELF works correctly (POST endpoint persists items +
stage_statuses + stage_actors to DB). The visibility layer is purely broken.

History — why this slipped through:

1. Spec #478 PART 1.2 said: "+1 line in get_run_detail response — exposes
   `stage_actors: run.get('stage_actors', {})`". The intent was to make
   `stage_actors` visible at top-level of the response.

2. Replit's STEP 0.2 of #478 confirmed "Backend data confirmed exposed:
   qc_router.py:1120 'stage_actors': run.get('stage_actors', {})" — but
   that grep verified the source code line existed, NOT that the data
   actually appeared at top-level of the JSON response.

3. In practice, `stage_actors` is naturally part of the `run` document
   (it's a field on db.qc_runs documents), so it comes through inside
   `run` automatically. The +1 line at qc_router.py:1120 is REDUNDANT
   and may either be ineffective (e.g. inside run dict construction,
   or shadowed) or is genuinely at top-level but the response field
   is not what we observe.

4. Frontend (both #478 + #479) trusted the spec contract and read
   from top-level. Never verified actual response shape during
   staging smoke prior to the override badge feature itself.

Conclusion: pure path mismatch. Fix is to change frontend to read from
the actual location (`runData.run.stage_actors`).

================================================================================
DESIGN DECISION — fix forward, NOT backend reshape
================================================================================

Option A (chosen) — fix frontend to read `runData.run.stage_actors`:
  - 3 occurrences in StageDetailPage.js
  - ~5 LOC total
  - Zero backend touch (data already there)
  - Zero deploy chain risk

Option B (rejected) — change backend to expose `stage_actors` at top-level:
  - Requires actually moving/duplicating the field in get_run_detail
  - Inconsistent with `stage_statuses` which lives the same way (inside `run`)
  - Adds coupling between backend response shape and frontend assumption
  - Future #478/#479-style features would have the same trap

Option A is the only correct fix. The backend code line at
qc_router.py:1120 ("stage_actors": run.get("stage_actors", {})) can stay
as-is OR be cleaned up in a future refactor — NOT in this hotfix.

PROCESS IMPROVEMENT — TO ADD TO CLAUDE.md (after this hotfix ships)
================================================================================

New iron rule for spec template:

    Whenever a UI batch reads from a NEW or EXTENDED backend field,
    STEP 0 MUST verify the ACTUAL shape of the API response, not just
    the source code that generates it. Required steps:

      (a) Identify the endpoint URL the new UI reads from
      (b) Open DevTools Network tab on staging (or curl the endpoint
          with a valid auth token)
      (c) Inspect the JSON response Preview / Response body
      (d) Confirm the field exists at the EXACT path the frontend
          will read (top-level vs nested vs key naming)
      (e) Document the verified shape in the spec STEP 0 audit

    This rule fixes the #478 + #479 trap: spec assumed top-level
    `stage_actors`; backend places it inside `run`; frontend never
    rendered the override badges/audit block until the path was
    fixed in this hotfix.
