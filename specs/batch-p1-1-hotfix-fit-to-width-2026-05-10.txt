CONTEXT
=======

Bug repro (Zahi staging smoke 2026-05-10):
  1. Open project plans page, click any A1-size architectural PDF
  2. Click "צפייה במסך מלא"
  3. PDF loads, but renders at native size (scale=1.0)
  4. PDF natural width ~2000px exceeds viewport ~1800px
  5. Two-layer flex container centers the canvas, leaving 100px clipped
     on each side
  6. macOS hides scrollbar by default — user sees no affordance to scroll
  7. User believes the right side of the plan (title block + sections)
     is "missing" or "broken"

Industry-standard fix: **default to fit-to-width**. Same as PDF.js viewer,
PDF-XChange Editor, Adobe Reader, Apple Preview. User sees the whole plan
edge-to-edge, then zooms in for details if needed.

Algorithm:
  - On PDF load, get page 1 natural dimensions via PDFDocumentProxy.getPage(1)
  - Get container client width (minus padding)
  - Calculate fit-scale = containerWidth / pageNaturalWidth
  - Cap at 1.0 — never upscale a small PDF beyond its native size
  - Use this as initial scale instead of 1.0

When user zooms in past fit-scale (via + button or pinch), content
exceeds viewport — at that point, scroll IS expected and the user has
the visual cue (they actively zoomed in) to know panning is needed.

LOCATION 1.2 — handleLoadSuccess: accept PDFDocumentProxy + compute fit-scale
-----------------------------------------------------------------------------

   CURRENT (around L42):
       const handleLoadSuccess = useCallback(({ numPages }) => {
         setNumPages(numPages);
         setLoading(false);
       }, []);

   REPLACE WITH:
       // 2026-05-10 hotfix — react-pdf onLoadSuccess passes the full
       // PDFDocumentProxy (not just {numPages}). We use it to compute
       // a fit-to-width scale on load so users see the whole plan
       // edge-to-edge by default. They can zoom in for details via
       // the + button or pinch.
       const handleLoadSuccess = useCallback(async (pdf) => {
         setNumPages(pdf.numPages);
         setLoading(false);
         try {
           if (!containerRef.current) {
             setScale(1.0);
             return;
           }
           // Subtract p-4 padding (16px each side = 32px) so content
           // doesn't immediately overflow.
           const containerWidth = containerRef.current.clientWidth - 32;
           const containerHeight = containerRef.current.clientHeight - 32;
           // 2026-05-10 — defensive guard: container may not be laid
           // out yet when handleLoadSuccess fires (modal still
           // animating in, display:none transition, mobile keyboard
           // pushing viewport). clientWidth=0 → containerWidth=-32
           // → fitScale=negative → clamped to 0.1 → mystery "tiny PDF"
           // bug. Fall back to 1.0 if dimensions look invalid; user
           // can zoom in/out manually.
           if (containerWidth <= 0 || containerHeight <= 0) {
             setScale(1.0);
             return;
           }
           const page = await pdf.getPage(1);
           const viewport = page.getViewport({ scale: 1.0 });
           const widthScale = containerWidth / viewport.width;
           const heightScale = containerHeight / viewport.height;
           // Use min so the WHOLE page fits (both dimensions). Cap at
           // 1.0 — small PDFs (e.g. apartment-only sketch) shouldn't
           // upscale above native size.
           const fitScale = Math.min(widthScale, heightScale, 1.0);
           // Round to nearest 5% so the displayed % is clean.
           const rounded = Math.max(0.1, Math.round(fitScale * 20) / 20);
           setScale(rounded);
         } catch (err) {
           console.warn('[PlanViewer] fit-scale calculation failed, falling back to 1.0:', err);
           setScale(1.0);
         }
       }, []);

   This avoids rendering <Page> with scale=null (would error in some
   react-pdf versions) and avoids flash-of-wrong-scale (rendering at
   scale=1.0 then jumping to fit-scale).

LOCATION 1.5 — Update zoom button bounds
----------------------------------------

   CURRENT (around L60):
       const zoomIn = useCallback(() => setScale(s => Math.min(s + 0.25, 4.0)), []);
       const zoomOut = useCallback(() => setScale(s => Math.max(s - 0.25, 0.5)), []);

   REPLACE WITH:
       // 2026-05-10 hotfix — use multiplicative steps (×1.25 / ×0.8) so
       // each click feels equally significant regardless of starting
       // scale. With additive +/- 0.25 and a fit-scale of (e.g.) 0.4,
       // clicking + once jumps to 0.65 (62% increase) but clicking +
       // again jumps from 0.65 to 0.9 (38% increase) — uneven.
       // Multiplicative keeps each click ~25% larger.
       const zoomIn = useCallback(() => setScale(s => Math.min((s ?? 1.0) * 1.25, 4.0)), []);
       const zoomOut = useCallback(() => setScale(s => Math.max((s ?? 1.0) * 0.8, 0.1)), []);

   Also reduce minimum from 0.5 to 0.1 — very large plans may need
   fit-scale below 0.5 (e.g. A0 plan on phone screen = ~0.2).

              flag the visual mis-scaling. Document the affected PDF
              for the follow-up batch.

         FOLLOW-UP (defer, out of scope here): track
