CONTEXT — why hotfix-2 didn't fully fix it
==========================================

After hotfix-2 lands, the JSX in PlanViewer.js is:

  <div className="flex-1 overflow-auto bg-slate-700 [&::-webkit-scrollbar]:..." dir="ltr">
    <div className="min-w-full min-h-full flex items-center justify-center p-4">
      <Document><Page scale={scale} /></Document>
    </div>
  </div>

The inner wrapper div has:
  - `min-w-full` → min-width: 100% of parent (= viewport width)
  - NO `width: ...` → width defaults to `auto` for block elements
  - `flex items-center justify-center` → centers child(ren)

CSS behavior with these constraints when child (canvas) is wider than
parent:
  - width: auto in a block context = same as 100% of parent
  - min-width: 100% reinforces this lower bound
  - **No upper bound, but no growth instruction either**
  - Net: wrapper div stays at viewport width
  - Canvas (e.g. 2000px) is wider than wrapper (e.g. 1500px)
  - With `justify-center`, canvas is centered within the wrapper, so it
    overflows the wrapper by ~250px on each side
  - The OUTER container has `overflow-auto` — but it scrolls **its own
    content**, which is the wrapper div. The wrapper is exactly the
    viewport width, so outer has NO horizontal overflow → no scrollbar
    appears, no scroll happens
  - The canvas overflow LEAKS OUT of the wrapper but is INVISIBLE to
    the outer scroll machinery

The fix: tell the wrapper to **grow with its content** while still
honoring the minimum width.

Tailwind primitive: `w-fit` (= `width: fit-content`).
With `w-fit + min-w-full`, the computed width is:
  - max(content, container)
  - When canvas fits → wrapper = container width → centering works
  - When canvas overflows → wrapper = canvas width → outer overflow-auto
    sees the overflow → scroll works → user can pan

This is the correct CSS for the centered-when-fits, scroll-when-
overflows pattern. It's how Excel viewers, Figma, Photoshop's image
canvases, and the PDF.js reference viewer all handle their canvases.
