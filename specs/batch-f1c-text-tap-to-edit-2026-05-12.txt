                After F.1b-hotfix-1, tap (no drag past 10px threshold)
                on an existing text label = currently no-op (explicitly
                reserved for F.1c per the comment in `endStroke`).
                F.1c repurposes this slot: pure tap → open edit
                overlay prefilled with the label's text/color/size →
                commit REPLACES the stroke in place; cancel leaves
                unchanged.

                Note: F.1c is IN-SESSION edit only. After Save closes
                the modal, the JPEG is rasterized and the strokes
                array is destroyed. Re-opening the saved photo would
                show a fresh modal with no strokes (re-edit-after-
                save is a separate architectural batch — out of
                scope here).

Approach:       Frontend-only. Mirror the existing F commit pattern
                exactly — same pendingText overlay, same input,
                same אישור / ביטול buttons. Add an optional
                `editingIndex` field on the pendingText state that
                tells commitPendingText whether to REPLACE
                (editingIndex set) or APPEND (editingIndex undefined,
                F behavior).

                When entering edit, also call `setColor(stroke.color)`
                + `setTextSize(stroke.size)` so the toolbar picker
                UI reflects the current label's properties (and the
                refs will pick those up via existing sync useEffects).

RISK
================================================================================
Low. Pure additive feature; reuses existing pendingText overlay,
commitPendingText callback structure, and the F.1b "pure tap" slot
that was explicitly reserved.

- ⚠ `editingIndex` is an OPTIONAL field on pendingText. Code checks
  `typeof curr.editingIndex === 'number'` to differentiate edit
  from new. Missing field = new (F default).
- ⚠ `setColor` + `setTextSize` calls in endStroke trigger React
  re-renders. The toolbar picker reflects new values before the
  user even sees the input — feels instant.
- ⚠ Replace via `.map()` creates a new array reference each commit.
  Necessary because React reconciles by reference; mutating in
  place wouldn't trigger re-render of components that depend on
  strokes state.
- ⚠ `_bbox` is intentionally NOT copied to the replacement stroke
  (new textStroke object has fresh fields). The next redraw
  populates _bbox with the EDITED label's actual dimensions —
  hit-test stays accurate for subsequent taps.
- ⚠ Cancel suppression of size-reset only applies when EDITING.
  Pure new-label cancel still resets to medium (F.1a behavior
  preserved).
- ⚠ Undo over edit removes the (edited) stroke entirely.
  Architectural consistency with F.1b drag; not a new defect.
- ⚠ Re-edit AFTER save (re-opening JPEG) is NOT addressed —
  out of scope, separate backlog item.

Bundle size: zero (no new imports).
Performance: 1 extra array .map() per commit (O(n)). Negligible.
Backend / schema / env / payment / billing: zero touch.

Why F.1c is safe to ship after F + F.1a + F.1b + hotfix-1:
- The "tap = no-op" slot was explicitly reserved.
- Hit-test, drag, and bbox cache already battle-tested green.
- Edit reuses the same overlay + commit pipeline as create.
- Only diff: branch on editingIndex.
