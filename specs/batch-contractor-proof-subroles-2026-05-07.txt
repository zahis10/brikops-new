BUG REPRODUCTION (Zahi 2026-05-07 production)
================================================================================

A contractor user with project membership role like 'plumber' or
'electrician' opens an assigned defect on /tasks/{id}.

Today:
  - Frontend TaskDetailPage L786 evaluates:
        showContractorProof = isContractor && isAssignee && status_ok
    isContractor (L234) checks projectRole === 'contractor' OR
    user.role === 'contractor'. Contractor's membership role is 'plumber',
    so isContractor=false → upload card NOT rendered.

  - Even if upload card were rendered and proof submitted, backend
    tasks_router.py L861 rejects:
        if project_role != 'contractor' and user['role'] != 'contractor':
            raise 403 'Only contractors can submit proof'

  - Manager never sees showManagerDecision because task.status never
    advances to 'pending_manager_approval' (which depends on contractor
    submitting proof). User-visible symptom: "המצב תמונה לא קיים גם
    לאדמין".

Root cause:
  Backend onboarding_router.py L185 registers contractors with specific
  SUBCONTRACTOR_ROLES = ['plumber', 'electrician', 'aluminum', 'painter',
  'tiler', 'carpenter', 'mason', 'hvac_tech', 'glazier', 'locksmith',
  'welder', 'insulator', 'roofer', 'landscaper', 'general_worker'].

  These are all conceptually contractors, but the literal string
  'contractor' is not their role. Both the frontend isContractor check
  AND the backend /contractor-proof guard hardcode the literal
  'contractor' string and miss all 15 subcontractor trade roles.

Fix strategy (minimum blast radius):
  Backend normalizes the role at the API boundary.
    1. New helper `_is_contractor_role(role)` in router.py.
    2. tasks_router.py L500: when computing `user_project_role` returned
       to frontend, map any subcontractor trade to 'contractor' so
       existing frontend code (and any future caller) treats them
       uniformly.
    3. tasks_router.py L861: relax the contractor-proof guard to use
       `_is_contractor_role(project_role)` (also accepts the global
       user.role check).

  Frontend stays UNCHANGED — relies on the backend normalization.

PART 1 — Add `_is_contractor_role` helper
================================================================================

FILE: backend/contractor_ops/router.py
--------------------------------------

Locate the existing `_get_contractor_trade_key` function (around L66).
ADD immediately above or below it:

  # Subcontractor trade roles. Source of truth is
  # onboarding_router.SUBCONTRACTOR_ROLES — duplicated here intentionally
  # to avoid a circular import (onboarding_router imports from this file).
  # If onboarding_router.SUBCONTRACTOR_ROLES changes, update this list too.
  _SUBCONTRACTOR_TRADE_ROLES = frozenset({
      'plumber', 'electrician', 'aluminum', 'painter', 'tiler',
      'carpenter', 'mason', 'hvac_tech', 'glazier', 'locksmith',
      'welder', 'insulator', 'roofer', 'landscaper', 'general_worker',
  })


  def _is_contractor_role(role: Optional[str]) -> bool:
      """True if `role` is the literal 'contractor' OR a specific
      subcontractor trade role (plumber, electrician, etc).

      All subcontractor trade roles are conceptually contractors and
      should be granted contractor-only permissions (e.g. submitting
      proof on assigned defects). Hardcoded checks for `role ==
      'contractor'` miss subroles and silently break flows.

      See CLAUDE.md "כלל ברזל" — production roles are more specific
      than 'contractor'.
      """
      if not role:
          return False
      return role == 'contractor' or role in _SUBCONTRACTOR_TRADE_ROLES

- _get_contractor_trade_key (works for all roles already).
- _trades_match (works on trade_key, not role).
- MANAGEMENT_ROLES at L413.
- Any other helper.

LOCATION 2.2 — Normalize user_project_role at L500:

  FIND (currently L500):
      task_data['user_project_role'] = membership['role'] if membership['role'] != 'none' else ('contractor' if (is_assignee or is_company_member) else 'none')

  REPLACE WITH:
      _raw_role = membership['role'] if membership['role'] != 'none' else ('contractor' if (is_assignee or is_company_member) else 'none')
      # Normalize subcontractor trade roles (plumber, electrician, etc)
      # to 'contractor' for frontend compatibility — frontend's
      # isContractor check expects the literal 'contractor' string.
      task_data['user_project_role'] = 'contractor' if _is_contractor_role(_raw_role) else _raw_role

  NOTE: this preserves 'none' for unauthorized callers (defensive — they
  shouldn't reach this line anyway, but if a future code path skips the
  guard, we don't want to silently grant contractor view).

LOCATION 2.3 — Broaden contractor_proof guard at L861:

  FIND (inside contractor_proof, around L860-862):
      project_role = await _get_project_role(user, task['project_id'])
      if project_role != 'contractor' and user['role'] != 'contractor':
          raise HTTPException(status_code=403, detail='Only contractors can submit proof')

  REPLACE WITH:
      project_role = await _get_project_role(user, task['project_id'])
      if not _is_contractor_role(project_role) and user['role'] != 'contractor':
          raise HTTPException(status_code=403, detail='Only contractors can submit proof')

  This rejects non-contractors (managers, viewers) but accepts plumbers,
  electricians, etc. — anyone in SUBCONTRACTOR_ROLES.
