CONTEXT
-------
Batch 505 added a mandatory ToS consent gate (Israeli Spam Law) across all
register/link forms. Replit added the checkbox to four forms:

  Form                              | Location  | Checkbox id
  ----------------------------------|-----------|--------------------------
  renderInviteAccept (existing user)| L871      | "onb-invite-terms"     ✅
  renderPhoneStep — register branch | L1193     | (in register branch)   ✅
  renderPhoneStep — link branch     | L1193?    | (verify-otp form)      ✅
  renderDetails                     | L1579     | (in form)              ✅

But the FIFTH form — `renderInviteLoginNeeded` (L902-994), shown when an
invite is sent to a phone with NO existing account — was MISSED. It still
calls `handleAcceptInvite` (which enforces termsAccepted at L634), so:

  1. New contractor receives WhatsApp invite link
  2. Clicks → /onboarding?invite=<id>
  3. Enters phone, gets OTP (debug code in staging), submits
  4. Phone is new → backend returns no token → frontend sets
     step='invite-login-needed' (L494)
  5. User fills name/password/email, clicks "צור חשבון והצטרף"
  6. handleAcceptInvite checks `if (!termsAccepted)` → toast.error(
     'יש לאשר את תנאי השימוש') → dead-end. termsAccepted is
     `false` by default and there is no UI in this form to flip it.

Verified on staging via Chrome MCP smoke test using invite
b010939f-58b0-40e7-a042-d1f578a1a207 (phone +972501234567) — form rendered
without checkbox, no SSO buttons either.

THIS HOTFIX adds only the ToS checkbox + button-disable. SSO buttons in the
invite flow are a separate, larger UX item (deferred to its own spec) because
they require careful phone-skip handling — the user's phone is already
verified at this step, but the existing socialFlow='register' branch asks for
phone again. That's a UX redesign, not a hotfix.

         </label>
       </div>

       <Button type="submit" disabled={loading || !termsAccepted} className="w-full bg-green-600 hover:bg-green-700 text-white h-12 text-base font-bold">
           {loading ? <Loader2 className="w-5 h-5 animate-spin ml-2" /> : null}
           {t('onboarding', 'create_and_join')}
         </Button>
       </form>
     </div>
   );

NOTE: The `disabled` change adds `|| !termsAccepted` so the user can't even
submit a request that the backend will reject. This matches the pattern used
by the existing register/link/SSO forms (L1036, L1058, L1073).
