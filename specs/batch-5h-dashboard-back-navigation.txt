CONTEXT
-------
Reproduce: Open project dashboard → click any KPI card (e.g. "פתוחים") →
URL changes to /control?workMode=defects&statusChip=open → click the
in-app back button (ArrowRight in header) → user is sent to /projects
(project list), not back to /dashboard.

Result: user has no way back to the dashboard except clicking "דשבורד"
in the top tab strip. The natural "back" instinct fails.

ROOT CAUSE — investigated 2026-04-29:
  ProjectControlPage.js:3643 has a HARDCODED back button:
      <button onClick={() => navigate('/projects')} title="חזרה לפרויקטים">
  It always goes to /projects regardless of where the user came from.
  Browser back goes correctly to /dashboard (push history), but most
  users reach for the in-app arrow first.

EXISTING CONVENTION (don't reinvent):
  ProjectTasksPage.js already solves the same problem using a URL query
  param `?from=dashboard`:
    line 81:  const fromDashboard = searchParams.get('from') === 'dashboard';
    line 358: navigate(fromDashboard ? `/projects/${projectId}/dashboard`
                                     : getProjectBackPath(...))
  This batch uses the SAME pattern in ProjectControlPage. Query param
  (not state.from) survives refresh / deep-link, matches the existing
  codebase convention, and minimizes risk of breaking other flows.

VERIFY 4.5 — `from=dashboard` URL param persistence (intentional, document):
   When user clicks a tab inside /control after arriving with `from=dashboard`,
   the param SURVIVES because setSearchParams uses {replace: true} (line 3411)
   and merges existing params. This is intentional — user still wants the
   "back to dashboard" affordance even after exploring tabs within /control.

   Edge case: if user bookmarks the URL with `from=dashboard` and opens it
   from the bookmark next session, back arrow will go to /dashboard
   (technically incorrect — the bookmark wasn't "from dashboard"). Acceptable
   tradeoff — bookmark of a deep-linked URL with query params is rare,
   and the worst case is "back goes to dashboard instead of project list,"
   which is still a sensible destination.

   Document this in review.txt under "Intentional behaviors". No code action.
