CONTEXT

#469 (Defect "בטיחות" tag) shipped to prod successfully. Code review of the
diff identified 3 follow-up items:

SF1 — URL persistence missing (ProjectControlPage.js)
  The safety filter chip uses local React state (`safetyFilter`) that resets
  on page refresh. The cleanup handler at line ~3879 calls
  `next.delete('safetyOnly')` — but no code ever SETS that URL param. So the
  delete is dead code, AND the filter doesn't survive refresh.

  The codebase already has the URL-derived pattern for sibling filters:
  `urlStatusChip = searchParams.get('statusChip')` at line ~3314 and
  `urlOverdue = searchParams.get('overdue') === 'true'` at line ~3315.

  Fix: replace `useState(false)` for `safetyFilter` with the URL-derived
  pattern — `urlSafetyOnly = searchParams.get('safetyOnly') === 'true'`.
  Add to `hasFilterParams` derivation. Toggle handler writes/deletes from
  URL via `setSearchParams`. The dead `delete('safetyOnly')` becomes live.

SF2 — Touch target on safety chip < 44px (ProjectControlPage.js)
  The safety filter chip uses `px-3 py-1.5` ≈ 28px height. Apple HIG +
  Material Design + CLAUDE.md require ≥44px on Capacitor. The chip lives
  in its OWN `<div>` (separate from neighboring filter chips), so we can
  enlarge it without breaking sibling alignment.

  Fix: change to `px-4 py-3` (~48px height with text + icon).

NTH — Three orange shades document (NewDefectModal.js + ProjectControlPage.js)
  Initial review flagged 3 different orange usages across safety UI as
  "inconsistent." Re-examination shows they are intentional emphasis levels
  within the orange-500 hue family (Material Design pattern):
    - text-orange-500          → icon / text emphasis
    - bg-orange-500 + white    → high-emphasis active state (filter chip)
    - bg-orange-100 + 700 text → low-emphasis badge (defect pill)

  This is correct, not a bug. To prevent future devs from "fixing" them
  inconsistently, add a brief comment block at the first usage in each
  file explaining the intent.

       const urlSafetyOnly = searchParams.get('safetyOnly') === 'true';

   CURRENT:

   REPLACE WITH:

   Note: hasFilterParams already depends on urlSafetyOnly via the new
   derivation in 1.1, so the array is internally consistent.

LOCATION 1.5 — Update the chip's onClick + className + aria-pressed to read
   from / write to URL (around the "🛡️ בטיחות בלבד" chip ~line 3835):

   CURRENT:
       <button
         onClick={() => setSafetyFilter(prev => !prev)}
         className={`inline-flex items-center gap-1.5 px-3 py-1.5 rounded-full text-xs font-medium transition-colors ${
           safetyFilter
             ? 'bg-orange-500 text-white hover:bg-orange-600'
             : 'bg-slate-100 text-slate-700 hover:bg-slate-200'
         }`}
         aria-pressed={safetyFilter}
       >
         🛡️ בטיחות בלבד
       </button>

   REPLACE WITH (also includes SF2 padding fix — px-4 py-3):
       <button
         onClick={() => {
           const next = new URLSearchParams(searchParams);
           if (urlSafetyOnly) {
             next.delete('safetyOnly');
           } else {
             next.set('safetyOnly', 'true');
           }
           setSearchParams(next, { replace: true });
         }}
         className={`inline-flex items-center gap-2 px-4 py-3 rounded-full text-sm font-medium transition-colors ${
           urlSafetyOnly
             ? 'bg-orange-500 text-white hover:bg-orange-600'
             : 'bg-slate-100 text-slate-700 hover:bg-slate-200'
         }`}
         aria-pressed={urlSafetyOnly}
       >
         🛡️ בטיחות בלבד
       </button>

   IMPORTANT — what changed inside the className:
     - `gap-1.5` → `gap-2` (pairs with bigger text)
     - `px-3 py-1.5` → `px-4 py-3` (SF2 fix — ≥44px touch target)
     - `text-xs` → `text-sm` (visual balance with bigger padding;
       still smaller than primary CTAs)

LOCATION 1.6 — KEEP the existing `next.delete('safetyOnly')` at line ~3879
