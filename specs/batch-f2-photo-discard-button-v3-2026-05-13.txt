BATCH F.2 v3 — Photo annotation: discard button (with onDiscard callback)
================================================================================
Date:           2026-05-13
Branch:         staging  (NOT main)
Severity:       P3 — UX gap discovered 2026-05-12 by Zahi during F.1
                staging test. PhotoAnnotation modal has Save + Undo
                but NO discard. User who takes a photo by mistake is
                forced to tap Save (uploads anyway).

                Zahi: "צילמתי תמונה, אבל נניח שהתחרטתי ואני רוצה
                לבטל, אין אפשרות לבטל, חייב ללחוץ רק שמור".

                v1: original spec used `onSave(null, false)` as
                discard signal. BROKEN — that signal is already used
                by `handleSave` when user taps Save with zero
                annotations, and 6 of 7 consumers UPLOAD the raw
                photo on that signal. v1's discard would have
                accidentally uploaded the unwanted photo.

                v2: re-spec with updated line numbers (post-F.1+F-keyboard).
                Still had the v1 bug — same signal collision.

                v3 (THIS): introduces a NEW callback prop `onDiscard`
                that's distinct from `onSave`. PhotoAnnotation's X
                button calls `onDiscard` (with confirm guard). Each
                of the 5 consumers (7 render sites) wires up
                `onDiscard` to clear the gating state (no upload).

                Architecturally cleaner: separate "save with no
                annotations" (upload raw) from "discard entirely"
                (don't upload).

Approach:       Frontend-only. Two halves:

                Half 1: PhotoAnnotation
                  - Add `onDiscard` prop + `onDiscardRef` (mirror of
                    `onSave` / `onSaveRef`).
                  - X button in top toolbar (LEFT side in RTL).
                  - handleDiscard callback: confirm if strokes exist;
                    on confirm/no-confirm → call `onDiscardRef.current?.()`
                    if provided, else fall back to
                    `onSaveRef.current(null, false)` (backward compat
                    for any future consumer that forgets to wire
                    onDiscard).

                Half 2: 5 consumer files (7 render sites)
                  - Each <PhotoAnnotation> gets a new `onDiscard`
                    prop that clears the gating state (e.g.
                    `setPendingFile(null)`).

Status:         AWAITING ZAHI APPROVAL — DO NOT DEPLOY

Files changed:  Frontend only:
                  - frontend/src/components/PhotoAnnotation.js
                    (~25 LOC: import + state ref + handler + button)
                  - frontend/src/pages/TaskDetailPage.js (2 instances)
                  - frontend/src/pages/StageDetailPage.js (1 instance)
                  - frontend/src/components/NewDefectModal.js (2 instances)
                  - frontend/src/pages/HandoverSectionPage.js (1 instance)
                  - frontend/src/components/handover/HandoverItemModal.js (1 instance)
                Total: 6 files. No backend, no DB, no env vars, no
                new dependencies (X icon already in lucide-react).
                ~+50 LOC net across all files.

Baseline:       origin/staging HEAD = 1aec4fc (F-keyboard v2, 799 LOC
                PhotoAnnotation).

        BEFORE `handleSave` (L593).

    setStrokes(prev => {

ADD AFTER:

  // BATCH F.2 (2026-05-13) — discard handler. Confirms before
  // closing if user has made annotations. Prefers onDiscard
  // callback (no upload); falls back to onSave(null, false) for
  // backward compat with any consumer that didn't wire onDiscard
  // (NOTE: as of this batch, all 7 known consumers do).
  const handleDiscard = useCallback(() => {
    if (strokesRef.current.length > 0) {
      if (!window.confirm('לבטל את כל השינויים ולסגור?')) return;
    }
    if (onDiscardRef.current) {
      onDiscardRef.current();
    } else {
      // Fallback: legacy abandonment signal. Consumer may interpret
      // as "upload raw photo" — that's the v1 bug we're fixing here
      // by adding onDiscard to all known consumers. Future-proof
      // fallback only.
      onSaveRef.current(null, false);
    }
  }, []);

RISK
================================================================================
Low. Pure additive feature with explicit opt-in per consumer.

- ⚠ The `onDiscard` prop is OPTIONAL with a backward-compat
  fallback (`onSave(null, false)`). If a future consumer adds
  PhotoAnnotation without passing onDiscard, the X button still
  works but falls back to upload-raw (v1 bug). The fallback is
  intentional to avoid silent breakage in future flows.
- ⚠ All 7 known render sites are updated in this batch. New
  render sites added later must explicitly opt-in via onDiscard
  (developer should be aware).
- ⚠ The confirm threshold is `strokesRef.current.length > 0`. If
  user opens text overlay (pendingText set) but hasn't committed,
  strokes is still empty → discard silently. Acceptable: the user
  intent is "I want this gone."
- ⚠ NewDefectModal Instance A's existing onSave handler already
  has early-return for `(null, false)`. The new onDiscard prop is
  redundant safety but harmless.
- ⚠ window.confirm is a native dialog. Different visual per
  platform (iOS / Android / desktop) but functional. **Known UX
  consideration:** dialog text is Hebrew (`'לבטל את כל השינויים
  ולסגור?'`) but native BUTTON LABELS may render in English
  (Cancel/OK) on iOS Safari + Capacitor, depending on platform
  localization. Acceptable for v3 shipping. If smoke reveals
  jarring UX (Hebrew question + English buttons), a custom modal
  is a ~20-LOC follow-up batch using existing Radix Dialog
  pattern (see PATTERNS section below).

**FOLLOW-UP IF NEEDED — custom discard confirm (NOT in v3 scope):**
If real users complain about the English Cancel/OK on the Hebrew
discard prompt, replace `window.confirm` with an in-component
modal. Reference pattern from `TaskDetailPage` reopen confirm
(Radix Dialog with Hebrew אישור/ביטול buttons). Estimated +30 LOC,
1 file (PhotoAnnotation.js). Should NOT block v3.
- ⚠ Save semantics unchanged. "Save with no annotations" still
  uploads raw photo. Only X is the new escape hatch.

Bundle size: zero (X icon already in lucide-react tree).
Performance: zero impact.
