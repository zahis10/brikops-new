DECISIONS (Zahi 2026-05-06)
================================================================================

D1. Direction: ONE-WAY ONLY (QC → Matrix). **PERMANENT decision** by
    Zahi 2026-05-06: "חשוב לי שהכיוון יהיה כיוון אחד, בקרת ביצוע לא
    אמורה להשתנות בשום שלב כתוצאה מהמטריצה, המטריצה בסך הכל שואבת
    את המידע מבקרת הביצוע ולא להיפך".
    Reverse direction (Matrix → QC) is **never** going to happen.
    QC has photo-evidence requirements that bypassing via matrix
    edits would compromise. This is an architectural lock, not a
    deferred feature.

D2. Granularity: ITEM-LEVEL CONTINUOUS sync (option B from spec discussion).
    Matrix cell updates on every QC item status change, not just when
    the stage is officially closed. Aggregate mapping below.

D3. Conflict resolution: QC ALWAYS WINS. Manual matrix edits on
    QC-template stages are transient — overwritten on next QC change.
    UI shows warning in edit dialog so PM understands.

D4. Architecture: SYNC WRITE. Every QC change triggers a write to
    `execution_matrix_cells`. Single source of truth for read = the cell
    document. Audit chain preserved. Marked with `synced_from_qc: true`.

D5. Scope of sync: UNIT-scope QC stages only. Floor-scope stages
    (e.g. "אישור סומסום בקומה") deferred to a follow-up — they require
    multi-cell update logic that's significantly more complex.

D6. PM custom stages (tag or status added in matrix's StageManagementDialog):
    NEVER synced. PM edits them freely as today.

D7. Backfill: NO. Historical QC runs that closed before #503 deploys
    don't auto-populate the matrix. Forward-looking only — first QC
    activity after deploy fires sync. PMs can manually fill historical
    cells if they want a clean view.

================================================================================
THE MAPPING — QC items → Matrix cell status (one stage at a time)
================================================================================

For a single (unit_id, stage_id) pair, gather all items in the latest
active QC run for that stage. Compute a single matrix status:

  IF run does not exist OR all items still ממתין (no manual touch):
      → matrix cell DELETED if it was synced before, OR no-op if never synced
      → cell appears empty in UI

  ELSE IF any item is לא תקין (status != "passed", may be "failed"):
      → matrix cell `status = "not_done"` (red)
      → fires immediately on first failed item — PM sees red the moment
        a defect is recorded

  ELSE IF all items are תקין (closed + passed):
      → matrix cell `status = "completed"` (green)

  ELSE (some items תקין, some still ממתין, NONE לא תקין):
      → matrix cell `status = "in_progress"` (blue)
      → PM sees that work is happening on this cell

Override-close (PM closes stage with override per #478):
  → matrix cell `status = "completed"` (treated same as natural close)
  → audit entry notes the override

NOT mapped (NEVER auto-set):
  - "partial" — manual-only status; PM uses for "kitchen done, baths pending"
  - "not_relevant" — manual-only; PM marks "this stage doesn't apply here"
  - "no_findings" — manual-only; visual distinction from "completed"

PMs can still set these manually via CellEditDialog. They DO NOT survive
a subsequent QC sync (per D3 — QC wins). UI warns about this.

    def _compute_matrix_status_from_qc_items(items, run):
        # CRITICAL: closed run wins over item-level inspection.
        # Override-close (#478) closes the run with items still pending.
        # Without this branch, sync would return "in_progress" instead
        # of "completed" — breaks T5 (override-close mapping).
        if run and run.get("closed"):
            if any(i.get("status") == "failed" for i in items):
                return "not_done"  # closed with at least one failure
            return "completed"  # closed naturally OR via override

        # Run is open — apply item-level continuous logic
        if any(i.get("status") == "failed" for i in items):
            return "not_done"  # red on first failure
        passed = [i for i in items if i.get("status") == "passed"]
        if not passed:
            return None  # all pending or no items — leave cell empty
        if len(passed) == len(items):
            return "completed"  # all items passed but run not yet closed
        return "in_progress"  # some passed, some pending

Upsert logic:
    cell = await db.execution_matrix_cells.find_one({
        "project_id": project_id,
        "unit_id": unit_id,
        "stage_id": stage_id,
    })
    new_status = _compute_matrix_status_from_qc_items(items, run)
    if new_status is None:
        # Optionally delete an existing sync-only cell
        if cell and cell.get("synced_from_qc"):
            await db.execution_matrix_cells.delete_one({"id": cell["id"]})
        return None
    audit_entry = {
        "type": "qc_sync",
        "actor_id": actor["id"],
        "actor_name": actor.get("name") or "QC sync",  # `or` not default arg — None values trigger fallback
        "ts": _now(),
        "before": cell.get("status") if cell else None,
        "after": new_status,
    }
    if cell:
        await db.execution_matrix_cells.update_one(
            {"id": cell["id"]},
            {
                "$set": {
                    "status": new_status,
                    "synced_from_qc": True,
                    "last_qc_sync_at": _now(),
                    "last_updated_at": _now(),
                    "last_updated_by": actor["id"],
                },
                # AUDIT PRUNING — keep last 50 entries to prevent
                # unbounded growth (PMs may rapid-fire QC updates,
                # producing many audit entries per cell over time).
                "$push": {
                    "audit": {
                        "$each": [audit_entry],
                        "$slice": -50,
                    },
                },
            },
        )
    else:
        new_doc = {
            "id": str(uuid.uuid4()),
            "project_id": project_id,
            "unit_id": unit_id,
            "stage_id": stage_id,
            "status": new_status,
            "note": None,
            "text_value": None,
            "synced_from_qc": True,
            "last_qc_sync_at": _now(),
            "last_updated_at": _now(),
            "last_updated_by": actor["id"],
            "created_at": _now(),
            "audit": [audit_entry],
        }
        await db.execution_matrix_cells.insert_one(new_doc)
    return await db.execution_matrix_cells.find_one(
        {"unit_id": unit_id, "stage_id": stage_id, "project_id": project_id},
        {"_id": 0},
    )

PART 3 — Backend: wire sync into qc_router endpoints
================================================================================

EDIT FILE: backend/contractor_ops/qc_router.py  (+~75 LOC)
  (BackgroundTasks pattern is verbose: imports + 3 endpoint signatures +
   `_run_sync` inner async fn × 3 + try/except × 3 + add_task × 3.
   ~25 LOC per endpoint × 3 endpoints = ~75 LOC total.)

Per STEP 0.2, identify the 3 endpoint handlers that change QC state:
  - Item update (per-item status flip)
  - Stage close/approve
  - Override-close (#478)

In EACH of the 3 handlers, AFTER the QC write succeeds and BEFORE
returning the response. Uses **FastAPI BackgroundTasks** so sync runs
AFTER the HTTP response is sent — QC user sees fast response (no
~100-300ms blocking on the sync's 4 DB ops):

    from fastapi import BackgroundTasks
    from contractor_ops.qc_to_matrix_sync import sync_qc_stage_to_matrix

    @router.put("/qc/items/{item_id}")
    async def update_qc_item(
        item_id: str,
        ...,
        background_tasks: BackgroundTasks,
        user: dict = Depends(get_current_user),
    ):
        # ... existing QC item update logic — runs synchronously
        await db.qc_items.update_one({"id": item_id}, {...})
        # ... whatever response is built today

        # #503 — schedule matrix sync to run AFTER response is sent.
        # Best-effort: failure logs but never blocks QC.
        async def _run_sync():
            try:
                run = await db.qc_runs.find_one({...latest run for this stage...})
                items = await db.qc_items.find({...all items of that stage...}).to_list(500)
                await sync_qc_stage_to_matrix(
                    db,
                    project_id=run["project_id"],
                    unit_id=run["unit_id"],
                    stage_id=stage_id,
                    actor={
                        "id": user["id"],
                        "name": user.get("display_name") or user.get("name"),
                    },
                    qc_run=run,
                    qc_items=items,
                )
            except Exception as e:
                logger.warning(
                    f"[#503] QC→matrix sync failed for stage_id={stage_id}: {e}"
                )

        background_tasks.add_task(_run_sync)
        return response  # original QC response, NOT awaiting sync

Trade-offs of BackgroundTasks vs inline await:
  ✅ QC response is fast (~100-300ms saved per request)
  ✅ Sync still runs after response (matrix still updates)
  ✅ Try/except still wraps it — failure doesn't crash the worker
  ⚠️  Matrix may lag 0.5-2s behind QC update (acceptable — PM doesn't
      see the matrix in the same second they click QC)
  ⚠️  If sync fails, the user is already gone — error is logged-only,
      no UI feedback. Same as inline-await with try/except in this regard.

Why best-effort + background:
  - Matrix is auxiliary; QC is primary truth
  - If sync fails, QC users completely unaffected
  - QC perf doesn't degrade (high-frequency item updates stay snappy)
  - Logged warning lets us debug post-hoc

Repeat the same pattern in the OTHER 2 handlers identified in STEP 0.2:
  - Stage close/approve endpoint
  - Override-close endpoint (#478)

  Mapping (any QC item change triggers re-sync):
    no items touched   → cell empty
    1+ passed, others pending → "in_progress" (blue)
    any failed         → "not_done" (red)  [fires immediately]
    all passed         → "completed" (green)
    override-close     → "completed" (green) with audit "override"

  Architecture (per Zahi decisions 2026-05-06):
    - One-way only (Matrix → QC reverse defer to a future batch)
    - Item-level continuous (cell flips on every item change, not just stage close)
    - QC always wins — manual matrix edits on QC-template stages are
      transient (overwritten on next QC change)
    - PM custom stages (tag/status added in matrix) NEVER synced —
      remain freely editable
    - Sync write architecture — matrix_cells is single source of truth
      for read; updated on QC change via best-effort hook

  Resilience:
    - Sync wrapped in try/except — failure logs but doesn't block QC writes
    - Feature flag MATRIX_QC_SYNC_ENABLED (default "false" — phased
      rollout: deploy with flag off → verify zero impact → flip on in
      staging → smoke 24h → flip on in prod during low-traffic window).
    - BackgroundTasks pattern (FastAPI) — sync runs AFTER HTTP response
      so QC requests don't slow down by ~100-300ms.
    - Floor-scope QC stages SKIPPED (D5) — defer to follow-up batch
