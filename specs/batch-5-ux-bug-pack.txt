BUG #2 — Tasks list shows stale "open" status after close
================================================================================

LOCATION
--------
frontend/src/pages/ProjectTasksPage.js — the tasks list page.

CONTEXT (Zahi confirmed 2026-04-28):
   "אחרי סגירה הרשימה לא רעננה ומראה עדיין 'פתוח'"
   (After closing, the list isn't refreshed and still shows 'open')

ROOT CAUSE (most likely)
   The page caches `tasks` state on mount and doesn't refetch when the user
   navigates BACK to it from TaskDetailPage. React Router doesn't unmount
   the parent route when navigating to a child route in some patterns, so
   when navigating back the list state is whatever it was before.

FIX (recommended — combine location.key + window focus, no infinite loops)

   Use `location.key` (NOT `location.pathname`). The `key` changes on EVERY
   navigation including back-navigation to the same path. `pathname` would
   miss navigations like /tasks/Y → /tasks (pathname changes here, OK) but
   would fail to retrigger on programmatic re-navigation to the same route.

   Implementation (per STEP 0.5 finding):

     If loadTasks IS already useCallback:
        const location = useLocation();
        useEffect(() => {
          loadTasks();
          const onFocus = () => loadTasks();
          window.addEventListener('focus', onFocus);
          document.addEventListener('visibilitychange', onFocus);
          return () => {
            window.removeEventListener('focus', onFocus);
            document.removeEventListener('visibilitychange', onFocus);
          };
        }, [location.key, loadTasks]);

     If loadTasks is NOT useCallback (avoid infinite loop):
        const location = useLocation();
        useEffect(() => {
          loadTasks();   // call directly inside, NOT in deps
          const onFocus = () => loadTasks();
          window.addEventListener('focus', onFocus);
          document.addEventListener('visibilitychange', onFocus);
          return () => {
            window.removeEventListener('focus', onFocus);
            document.removeEventListener('visibilitychange', onFocus);
          };
          // eslint-disable-next-line react-hooks/exhaustive-deps
        }, [location.key]);

   Why both location.key AND focus listener:
     - location.key catches the in-app navigation back to the list.
     - focus + visibilitychange catch the case of user switching to another
       browser tab (or device sleep) and coming back — also a refresh trigger.

   Option B (fallback if Option A causes problems): sessionStorage dirty flag.
     After close in TaskDetailPage, set sessionStorage.setItem('tasks_dirty',
     '1'). On ProjectTasksPage mount, check + clear + refetch if set.
     Use this only if Option A breaks something.

VERIFY
------
- Open a defect in the list → close it → navigate back → the defect shows
  the correct closed status without a manual page refresh.
- Reopen flow (Bug #1) similarly: after reopen, navigate back → the defect
  shows the correct status.

BUG #3 — Company dropdown escapes UserDrawer
================================================================================

LOCATION
--------
frontend/src/components/UserDrawer.js — the contractor profile drawer.

CONTEXT
   When the drawer is open and the user clicks the "בחר חברה" select, the
   dropdown's options panel renders ABOVE/OUTSIDE the drawer's visual area.
   It overlaps the page background (or other modals) instead of staying
   "inside" the drawer.

INVESTIGATION
   First, identify what library/component the company dropdown uses:
     grep -nE "Select|<select|RadixSelect|Listbox|Popover" \
         frontend/src/components/UserDrawer.js
   Likely candidates and fixes:

   (a) Plain <select> (browser native):
       Native <select> opens via the OS picker — should never "escape".
       If THIS is the case, the visual report is misleading; ask Zahi for
       a screenshot.

   (b) Custom <Select> component (in frontend/src/components/ui/):
       Check for a Portal that mounts outside the drawer. The fix is to
       either (i) ensure the Portal target has z-index higher than the
       drawer overlay, or (ii) mount the dropdown inside the drawer's
       Portal (so it inherits the same stacking context).

   (c) Radix Select via @radix-ui/react-select:
       Radix uses a Portal by default. Wrap the SelectContent like:
         <SelectPrimitive.Portal container={drawerRef.current}>
           ...
         </SelectPrimitive.Portal>
       OR set `position="popper"` and rely on the Portal landing on body.
       If z-index is the issue, add `z-[10000]` to the SelectContent's
       className (drawer is likely z-[9999]).

   (d) Headless UI Listbox / Combobox:
       Wrap with a higher-z-index container or use the `Transition`
       wrapper with a fixed-position popper.

REQUIRED FIX
   - Make the dropdown options panel render on top of the drawer overlay
     and visually contained within (or aligned to) the input that triggered
     it — without overflowing onto the page underneath.
   - Acceptance criteria: dropdown is keyboard-navigable, click-outside
     closes it (without closing the drawer), and visually appears to be
     "part of" the drawer.

VERIFY
------
- Open the drawer, click the company select, dropdown appears properly
  positioned and visually contained (no leakage to the page background).
- Keyboard navigation works (Tab, Esc).
- Click outside the dropdown: it closes; the drawer stays open.
