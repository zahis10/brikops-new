================================================================================
BATCH F-keyboard — PhotoAnnotation canvas resizes when mobile keyboard pops
================================================================================
Date:           2026-05-12
Branch:         staging  (NOT main)
Severity:       P3 — UX annoyance on iPhone reported by Zahi
                2026-05-12 during F.1c testing. When the on-screen
                keyboard appears (label create OR edit), the canvas
                appears to "zoom in" / clip. Root cause: canvas
                display dimensions are computed ONCE on image load
                using `window.innerHeight - 120`, but the visual
                viewport shrinks when keyboard pops. Canvas style
                stays at original dimensions → bottom of image is
                hidden behind keyboard → appears cropped.

                This bug exists in F (since shipped 2026-05-12) but
                wasn't noticed until F.1c testing because labels
                are mostly created at the TOP of the image (visible
                area when keyboard up) — only became prominent when
                Zahi re-tapped labels for edit.

                Independent of F.1c logic — pure viewport / sizing
                concern. Ships AFTER F.1c to avoid merge conflicts
                in PhotoAnnotation.js (or before, if F.1c is held).

Approach:       Frontend-only. Listen to `window.visualViewport`
                resize events (iOS Safari 13+ / Chromium). On
                resize, recompute the display scale + canvas CSS
                dimensions, then redraw to refresh bbox cache.

                Falls back gracefully on browsers without
                visualViewport API (older Android WebView, very old
                Safari) — listens to `window.resize` instead.
                On those, keyboard pop doesn't fire resize, but
                neither did the original code work there. No
                regression.

Status:         AWAITING ZAHI APPROVAL — DO NOT DEPLOY

Files changed:  Frontend only:
                  - frontend/src/components/PhotoAnnotation.js
                    (~+35 LOC: extract sizing into useCallback +
                    add useEffect with resize listener)
                Total: 1 file. No backend, no DB, no env vars, no
                new dependencies.

Baseline:       origin/staging HEAD at time of spec writing was
                a410b29 (F.1b-hotfix-1, 706 LOC). If F.1c ships

SCOPE — what's IN, what's OUT
================================================================================

IN this batch:
  ✅ Extract the canvas display-dimension computation from the
     image-load useEffect into a standalone `applyDisplayScale`
     useCallback. It reads `canvasRef`, `imgRef`, sets
     `scaleRef.current`, updates `canvas.style.width/height`, and
     calls `redraw(strokesRef.current)`.
  ✅ Image-load useEffect calls `applyDisplayScale` AFTER setting
     `canvas.width/height` (intrinsic resolution).
  ✅ NEW useEffect attaches `visualViewport.resize` listener (and
     `window.resize` fallback) → calls `applyDisplayScale` on
     resize. Cleanup on unmount.
  ✅ Uses `window.visualViewport.height` when available (excludes
     keyboard, system UI). Falls back to `window.innerHeight` for
     compatibility.

OUT of this batch:
  ❌ Custom keyboard handling (Capacitor `@capacitor/keyboard`
     plugin) — visualViewport API is already supported in
     Capacitor WebView (iOS 13+ / Android Chromium).
  ❌ Scroll-to-input-on-focus behavior — already handled by browser.
  ❌ Re-layout of toolbar / action bar on keyboard pop — they're
     in a flex column with shrink-0 / flex-1, so they reflow
     automatically. Only canvas needs explicit dimension update.
  ❌ Animations / transitions on resize — the resize is fast
     enough (debounced by browser).

LOCATION 1 — Extract applyDisplayScale callback
================================================================================
File: frontend/src/components/PhotoAnnotation.js
Insert: AFTER the `redraw` useCallback definition (currently around
        L150 in F.1b-hotfix-1 state).

ANCHOR (find the end of redraw callback):
  const redraw = useCallback((allStrokes) => {
    ...
  }, [getLineWidth]);

ADD AFTER:
  // BATCH F-keyboard (2026-05-12) — recompute canvas display
  // dimensions based on the current visual viewport. Called on
  // image load AND on viewport resize (mobile keyboard pop). The
  // intrinsic canvas resolution (canvas.width / canvas.height)
  // stays fixed — we only update CSS display size + scaleRef.
  // After resize, redraw refreshes the bbox cache for hit-testing.
  const applyDisplayScale = useCallback(() => {
    const canvas = canvasRef.current;
    const img = imgRef.current;
    if (!canvas || !img) return;

    const w = canvas.width;
    const h = canvas.height;
    if (!w || !h) return;

    // Prefer visualViewport (iOS Safari 13+ / Chromium) — gives
    // visible area excluding the on-screen keyboard. Fall back to
    // window.inner* for environments without the API.
    const vv = window.visualViewport;
    const containerW = vv ? vv.width : window.innerWidth;
    const containerH = (vv ? vv.height : window.innerHeight) - 120;

    const displayScale = Math.min(containerW / w, containerH / h, 1);
    scaleRef.current = displayScale;

    canvas.style.width = Math.round(w * displayScale) + 'px';
    canvas.style.height = Math.round(h * displayScale) + 'px';

    // Redraw at the new scale — recomputes text fontSize, pill
    // dimensions, and refreshes the _bbox cache on each text stroke.
    redraw(strokesRef.current);
  }, [redraw]);

LOCATION 3 — Listen to visualViewport.resize + window.resize
================================================================================
File: frontend/src/components/PhotoAnnotation.js
Insert: NEW useEffect, ideally near the existing viewport-lock useEffect
        (the one that sets `viewport.content = 'width=device-width...'`).

ANCHOR (find this useEffect — exists in F):
  useEffect(() => {
    const viewport = document.querySelector('meta[name="viewport"]');
    const originalViewport = viewport?.content;
    if (viewport) {
      viewport.content = 'width=device-width, initial-scale=1, maximum-scale=1, user-scalable=no';
    }
    return () => {
      document.body.style.pointerEvents = '';
      document.body.style.overflow = '';
      if (viewport && originalViewport !== undefined) viewport.content = originalViewport;
    };
  }, []);

ADD AFTER it (as a new sibling useEffect):
  // BATCH F-keyboard (2026-05-12) — listen for viewport changes
  // (mobile keyboard appearance is the primary use case) and
  // recompute the canvas display scale. Without this, the canvas
  // keeps its original dimensions and gets clipped behind the
  // keyboard, appearing zoomed-in.
  useEffect(() => {
    if (!loaded) return; // wait until image loaded

    const vv = window.visualViewport;
    if (vv) {
      vv.addEventListener('resize', applyDisplayScale);
    }
    window.addEventListener('resize', applyDisplayScale);

    return () => {
      if (vv) {
        vv.removeEventListener('resize', applyDisplayScale);
      }
      window.removeEventListener('resize', applyDisplayScale);
    };
  }, [loaded, applyDisplayScale]);

Notes:
- `loaded` gate prevents firing before the canvas + image are ready.
- `applyDisplayScale` reference comes from useCallback at LOCATION 1.
- Both `visualViewport` and `window` listeners registered for
  defense in depth — most browsers fire one or the other; some
  fire both. The handler is idempotent (computes desired state,
  not delta).
- Per STEP 0.5, `redraw` is stable (deps = [getLineWidth], and
  getLineWidth deps = []). So `applyDisplayScale` is stable, and
  this useEffect's deps change only via `loaded` (false → true
  once on mount, never again). Listener attaches ONCE on image
  load, stays attached. No detach/reattach churn.

================================================================================
LOCATION 3 — FALLBACK (only if STEP 0.5 finds redraw UNSTABLE)
================================================================================
Skip this section if STEP 0.5 confirmed redraw is stable (expected).

If `redraw` has unstable deps (e.g. depends on a prop that changes
per render), the resize useEffect would churn — detaching and
re-attaching listeners on every render. To prevent this, use a
ref-stable wrapper:

  // BATCH F-keyboard — ref-stable wrapper to avoid listener churn
  // when applyDisplayScale identity changes between renders.
  const applyDisplayScaleRef = useRef(applyDisplayScale);
  useEffect(() => {
    applyDisplayScaleRef.current = applyDisplayScale;
  }, [applyDisplayScale]);

  const stableResizeHandler = useCallback(() => {
    applyDisplayScaleRef.current();
  }, []);

Then in the resize useEffect, register `stableResizeHandler` (which
has empty deps = stable identity) instead of `applyDisplayScale`.
Change the resize useEffect's deps array from
`[loaded, applyDisplayScale]` to `[loaded, stableResizeHandler]`.

================================================================================
FUTURE NOTE — RAF throttling (DO NOT implement in v1)
================================================================================
visualViewport.resize can fire at high frequency:
  - Keyboard slide animations: ~10-15 events over 200ms
  - Desktop window resize drag: continuous
  - iOS Safari address bar show/hide: frequent on scroll

Each fires `applyDisplayScale` → `redraw` → O(n) over all strokes.
For typical photo annotation (≤10 strokes): perf negligible.
For dense annotation (50+ strokes): possible slight jank during
keyboard animation on slow devices.

**If jank is reported in field testing**, wrap applyDisplayScale
in requestAnimationFrame coalescing — multiple resize events within
one frame collapse into a single update:

  const rafRef = useRef(null);
  const applyDisplayScaleRaf = useCallback(() => {
    if (rafRef.current) return;
    rafRef.current = requestAnimationFrame(() => {
      rafRef.current = null;
      applyDisplayScale();
    });
  }, [applyDisplayScale]);

  // Use applyDisplayScaleRaf in the resize listener instead.

Skip for v1 — almost certainly unnecessary for typical density.
Document only.

================================================================================
RISK
================================================================================
Low. Pure additive — extracted logic + new listener. No changes
to drawing, drag, edit, or save logic.

- ⚠ `applyDisplayScale` calls `redraw(strokesRef.current)` which
  populates _bbox on each text stroke. This happens during
  resize, which could be frequent on keyboard show/hide. Redraw
  is O(n) over strokes. For typical photo annotations (≤10
  strokes), perf is negligible.
- ⚠ visualViewport.resize on Android sometimes fires twice
  (once for outer viewport, once for keyboard inset). Both calls
  are idempotent — second is a no-op.
- ⚠ If user has zoomed the page manually (pinch zoom — though
  user-scalable=no prevents it), visualViewport.height could be
  smaller than expected. Acceptable: canvas just gets smaller.
- ⚠ The intrinsic canvas resolution (canvas.width / canvas.height)
  is NOT changed by resize — that's set once on image load.
  Only CSS display dimensions change. So saved JPEGs are still
  at full resolution.
- ⚠ Existing label coords (x/y in canvas px) don't shift on
  resize. The bbox cache recomputes with new scale → consistent
  for hit-test.
- ⚠ Some browsers (very old) may not fire `window.resize` for
  keyboard appearance. Those would lack visualViewport too →
  fallback path is also a no-op there. No regression vs current.

Bundle size: zero.
Performance: 1 extra redraw per keyboard show/hide. Negligible.
