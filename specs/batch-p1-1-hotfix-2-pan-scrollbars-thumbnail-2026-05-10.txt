CONTEXT — Bug 1 details
=======================

Reproducer: open A1 PDF, viewer opens at 61% fit-scale (whole plan
visible ✅). Click + once. Now at ~76%. Content exceeds viewport.
Expected: scroll horizontally to see right side. Actual: only left side
visible, scroll seems blocked.

Three contributing causes:

a) **`dir="rtl"` on scrolling container** — The outer modal sets
   `dir="rtl"` for Hebrew labels in the top/bottom bars. The scrolling
   container (`<div ref={containerRef} className="flex-1 overflow-auto">`)
   inherits this. Chrome's RTL scroll handling is inconsistent with
   `flex items-center justify-center` — the initial scroll position may
   be anchored to one edge, and scrollLeft direction is mirrored.
   The PDF canvas itself is LTR content; we don't need RTL on the
   scroller.

b) **`Math.round` can round UP** — current code:
       const rounded = Math.max(0.1, Math.round(fitScale * 20) / 20);
   For fitScale=0.585: round(11.7)=12, /20=0.60. The displayed scale
   (0.60) is HIGHER than the actual fit (0.585), so even at "fit-to-
   width" the canvas slightly overflows the viewport. Should round
   DOWN to be safe.

c) **macOS hidden scrollbars** — by default, mac Chrome hides scrollbars
   until the user scrolls. With no scroll attempt yet, user sees no
   indication that scrolling IS possible. We can force visible scrollbars
   via webkit-scrollbar CSS.

Fixing all three together gives high-confidence resolution.

CONTEXT — Bug 2 details
=======================

The detail-modal preview shows "התוכן הזה חסום. יש לפנות אל הבעלים של
האתר כדי לפתור את הבעיה." for every PDF.

Root cause: app's Content-Security-Policy header (verified earlier in
this session, set by Cloudflare Pages):

  content-security-policy: ... frame-src https://accounts.google.com
  https://appleid.apple.com; frame-ancestors 'none'

S3 PDF URLs (`brikops-staging-files.s3...`) are NOT in the frame-src
allowlist, so the iframe is blocked. This is a SECURITY-INTENTIONAL
restriction — we should NOT loosen the CSP just to show PDF previews.

Solution: use the **thumbnail_url** that's already generated by the
backend on every plan upload (plans_router.py:222 and L407 — both
project-plan and unit-plan upload paths trigger
`_generate_and_save_thumbnail` async). The thumbnail is a PNG, served
via the same S3 bucket as `<img src=...>` (img tags don't enforce CSP
frame-src — they're not iframes).

UnitPlansPage already does this correctly (L601-604):

      {detailPlan.thumbnail_url ? (
        <img src={detailPlan.thumbnail_url} ... />
      ) : (detailPlan.file_type || '').startsWith('image/') ? (
        <img src={detailPlan.file_url} ... />
      ) : (detailPlan.file_type === 'application/pdf') ? (
        <iframe src={detailPlan.file_url} ... />   // ← still here as fallback
      ) ...

ProjectPlansPage misses the `thumbnail_url` check entirely and goes
straight to image OR iframe (L1004-1015):

      {(detailPlan.file_type || '').startsWith('image/') ? (
        <img src={detailPlan.file_url} ... />
      ) : (detailPlan.file_type === 'application/pdf') ? (
        <iframe src={detailPlan.file_url} ... />   // ← always blocked
      ) ...

Fix: add `thumbnail_url` check first in ProjectPlansPage (mirror
UnitPlansPage logic). The iframe stays as ULTIMATE fallback for plans
that haven't generated thumbnails yet (e.g. just uploaded, async
worker hasn't completed) — it'll still show "blocked" for those, but
that's a transient state.

LOCATION 1.1 — Math.floor for fit-scale (be conservative — never overshoot)

   CURRENT (~L72, in handleLoadSuccess):
       // Round to nearest 5% so the displayed % is clean.
       const rounded = Math.max(0.1, Math.round(fitScale * 20) / 20);
       setScale(rounded);

   REPLACE WITH:
       // 2026-05-10 hotfix-2 — Math.floor (not Math.round) so we never
       // round UP past the true fit-scale. Math.round(11.7)=12 → 0.60
       // when actual fit is 0.585 → 0.60 already overflows by ~3%.
       // floor(11.7)=11 → 0.55 → strictly within fit. User can zoom IN
       // to show full-width if they want.
       const rounded = Math.max(0.1, Math.floor(fitScale * 20) / 20);
       setScale(rounded);


LOCATION 1.2 — dir="ltr" + Tailwind arbitrary scrollbar modifiers

   Confirmed: project uses Tailwind 3.4.17 with arbitrary modifiers
   already in use (frontend/src/components/ui/calendar.jsx,
   ui/table.jsx). Switch to inline Tailwind instead of JS CSS
   injection — cleaner, no module-level side effect, all styles
   colocated with the component.

   CURRENT (~L122, the viewer area div):
       <div
         ref={containerRef}
         className="flex-1 overflow-auto bg-slate-700"
         style={{ touchAction: 'pan-x pan-y pinch-zoom' }}
       >

   REPLACE WITH:
       {/* 2026-05-10 hotfix-2 — three fixes layered here:
            a) dir="ltr" on the scroll container so RTL inheritance from
               the outer modal doesn't mess up Chrome's RTL scroll
               handling with `flex justify-center`. PDF canvas content
               is intrinsically LTR; only the chrome (top/bottom bars)
               needs RTL.
            b) Tailwind arbitrary `[&::-webkit-scrollbar]` modifiers so
               macOS users see scrollbars permanently — visual cue that
               pan is possible. Pattern aligned with calendar.jsx /
               table.jsx (project convention, Tailwind 3.4+). No JS
               injection, no module side-effect.
            c) touchAction kept for native pinch-zoom on mobile. */}
       <div
         ref={containerRef}
         className="
           flex-1 overflow-auto bg-slate-700
           [&::-webkit-scrollbar]:w-3
           [&::-webkit-scrollbar]:h-3
           [&::-webkit-scrollbar-thumb]:bg-white/35
           [&::-webkit-scrollbar-thumb]:rounded-full
           [&::-webkit-scrollbar-thumb]:border-2
           [&::-webkit-scrollbar-thumb]:border-transparent
           [&::-webkit-scrollbar-thumb]:bg-clip-content
           [&::-webkit-scrollbar-thumb]:hover:bg-white/55
           [&::-webkit-scrollbar-track]:bg-black/20
           [&::-webkit-scrollbar-corner]:bg-black/20
         "
         dir="ltr"
         style={{ touchAction: 'pan-x pan-y pinch-zoom' }}
       >

   NO file-top CSS injection block needed.
