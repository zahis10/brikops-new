ROOT CAUSE — Zahi's prod test 2026-05-05
================================================================================

After #495 deployed:
  ✅ PM hides template stage "טיח" → matrix re-renders without "טיח" column.
     base_stages_removed=["stage_taich"] persists in matrix_config DB.
  ❌ PM opens StageManagementDialog later → "טיח" NOT shown in section 2.
     Result: PM cannot toggle "טיח" back to visible. Data appears lost.

WHY: Backend `_resolve_visible_stages` filters out hidden stages BEFORE
returning the `stages` array:

  for s in tpl.get("stages", []):
      if s["id"] in base_removed:
          continue  # ← hidden stages NOT in output
      base_stages.append({...})

Frontend `StageManagementDialog` does:
  const initialBase = (stages || []).filter(s => s.source === 'base');

`initialBase` only contains VISIBLE stages. Hidden ones never reach the
section 2 render → no UI to unhide them.

CRITICAL: The DB data is intact. `base_stages_removed` array is correctly
saved. Hidden stage's cells are still in `execution_matrix_cells`. The
issue is purely UI visibility — frontend doesn't have the data to render
the hidden rows.

This is why we need a separate `template_stages` field (all template
stages, regardless of hide status).

================================================================================
DESIGN DECISIONS
================================================================================

D1. **Backend exposes `template_stages` in /matrix response**
    Contains ALL base stages from qc_template (including hidden ones).
    Frontend filters by hide state for visibility, but renders all in
    the dialog so PM can toggle.

D2. **Frontend reads `template_stages` in dialog, not filtered `stages`**
    Section 2 of StageManagementDialog uses `data.template_stages` for
    its base list. The hide state comes from `data.base_stages_removed`
    (already exposed in #495).

D3. **Section 1 rename: "עמודות מטה-דאטה" → "עמודות מותאמות"**
    The word "מטה-דאטה" implies metadata only (tag values like
    "מס' חדרים", "סוג מכירה"). Per Zahi's prod test, this masks the
    fact that PM can ALSO add status-type columns here (just like
    template stages, but project-scoped).

D4. **Subtitle clarifies both types**
    "תגיות (טקסט חופשי) או סטטוסים (6 ערכים — כמו עמודות בקרת ביצוע)"
    Makes it explicit that PM can add any of the 2 types.

D5. **AddStageForm radio label clarification**
    "סטטוס (6 ערכים)" → "סטטוס (6 ערכים — כמו עמודות בקרת ביצוע)"
    Explicitly tells PM that this option creates an execution-control-
    style column, addressing the "where do I add execution stages"
    confusion.

D6. **NO change to AddStageForm logic** — the feature is already there
    (radio button), only labels clarified. PM can add any type today,
    just unclear from the UX.
