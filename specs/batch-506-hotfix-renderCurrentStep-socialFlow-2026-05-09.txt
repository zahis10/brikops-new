CONTEXT
-------
Batch 506 added Google + Apple SSO buttons to renderInvitePhoneStep (the
invite landing screen). The buttons render correctly. Clicking them fires
handleGoogleSignIn / handleAppleSignIn → backend /auth/social returns
either:
  - link_required (existing user with same Google ID) → frontend sets
    socialFlow='link' + auto-calls socialSendOtp (sends OTP to user's
    REGISTERED phone)
  - registration_required (new user, no Google linkage) → frontend sets
    socialFlow='register' (waits for user to enter phone)

In BOTH cases, the user needs the renderPhoneStep UI to take over —
that's where the socialFlow='link'/'register'/'otp' branches live (L1043,
L1143, L1204). But renderCurrentStep routes invite flow to
renderInvitePhoneStep regardless of socialFlow:

  // Current (broken):
  const renderCurrentStep = () => {
    if (isInviteFlow) {
      if (step === 'invite-accept') return renderInviteAccept();
      if (step === 'invite-error') return renderInviteError();
      if (step === 'invite-login-needed') return renderInviteLoginNeeded();
      if (step === 'otp') return renderOtpStep();
      if (step === 'login-password') return renderLoginPassword();
      return renderInvitePhoneStep();   // ← always returns this
                                         //    even when socialFlow is set
    }
    ...
  };

So after Google OAuth in invite flow:
  - socialFlow='link' (or 'register') is set
  - step is still 'phone' (original invite landing)
  - renderCurrentStep returns renderInvitePhoneStep
  - renderInvitePhoneStep doesn't know about socialFlow → user sees the
    same form, no OTP input

Fix: in renderCurrentStep, when socialFlow is set, return renderPhoneStep
regardless of isInviteFlow. socialFlow is null by default, so this only
triggers AFTER user clicks Google/Apple.

This also incidentally fixes a related issue: my prior 506 spec added
setStep('phone') in handleSocialAuthResult registration_required as a
"linchpin" — but it was a no-op because step was already 'phone' AND
the isInviteFlow branch in renderCurrentStep routes step='phone' to
renderInvitePhoneStep. The actual fix is in renderCurrentStep, not in
handleSocialAuthResult.

NOT IN SCOPE — separate batch:
  - Backend `/auth/social/verify-otp` LINK branch does NOT process
    invite_token. After this hotfix, an existing user clicking Google on
    an invite link will authenticate successfully but won't get added to
    the invited project. Needs backend update to process invite_token in
    link flow (apply membership, mark invite accepted). Track as
    follow-up batch 506-followup-link-invite.
  - Existing-user flow on Apple SSO (untested by Zahi per his decision
    to skip Apple).

   CURRENT (lines 1684-1693):
       const renderCurrentStep = () => {
         if (isInviteFlow) {
           if (step === 'invite-accept') return renderInviteAccept();
           if (step === 'invite-error') return renderInviteError();
           if (step === 'invite-login-needed') return renderInviteLoginNeeded();

   REPLACE WITH:
       const renderCurrentStep = () => {
         // 2026-05-09 — 506 hotfix: when user clicks Google/Apple from the
         // invite landing screen, handleSocialAuthResult sets socialFlow=
         // 'link'/'register'. The OTP-input + phone-input UIs for SSO live
         // in renderPhoneStep (L1043+), NOT in renderInvitePhoneStep.
         // Without this early return, the user gets stuck on the invite
         // landing form with no way to enter OTP after OAuth.
         if (socialFlow) return renderPhoneStep();
         if (isInviteFlow) {
           if (step === 'invite-accept') return renderInviteAccept();
           if (step === 'invite-error') return renderInviteError();
           if (step === 'invite-login-needed') return renderInviteLoginNeeded();
           if (step === 'otp') return renderOtpStep();
           if (step === 'login-password') return renderLoginPassword();
           return renderInvitePhoneStep();
         }

NOTES:
  - For NON-invite flow: socialFlow set → renderPhoneStep → SAME behavior as
    before (renderPhoneStep was already the target for step='phone' which
    is what gets used after `setStep('phone')` from 506 LOCATION 1.1).
    So no behavior change for non-invite users.
  - For INVITE flow with socialFlow=null (i.e. user hasn't clicked Google/
    Apple): no change, still renders renderInvitePhoneStep as before.
  - Only behavior change: invite flow with socialFlow set → now correctly
    routes to renderPhoneStep instead of being stuck at renderInvitePhoneStep.
  - handleSocialBack (L391) clears socialFlow=null, which would route back
    to renderInvitePhoneStep automatically. Good back-navigation behavior
    preserved.
