CONTEXT — REAL PROBLEM (Zahi 2026-05-04)
================================================================================

> "יש לי מנהל פרויקט שהתחבר לאפליקציה, אבל הוא כבר סיים את כל השלבים
>  הראשוניים של יציקת תקרה בלוקים וכדומה, הוא לא יכול באמת לסגור את
>  הסעיפים."

A PM joined an existing project mid-construction. The early stages
(יציקת תקרה, בלוקים, etc.) were COMPLETED before the app was deployed.
Items have no photos, no notes, no documentation in app — but the work
IS physically done. The current submit/approve flow blocks closure
because:
  - submit_stage requires every item to have status (not pending)
  - submit_stage requires photos on items with required_photo
  - submit_stage requires notes on failed items
  - approve_stage only callable when status=pending_review

Result: project shows 0% complete forever. PM cannot use the app for
oversight.

This batch adds an "Override close" path: the PM can manually mark a
stage as approved WITHOUT the per-item validation, with full audit trail
(who, when, reason). Restricted to project_manager role + super_admin.
Other valid use cases for override:
  - Documentation uploaded to wrong unit/stage and needs cleanup
  - Quality issue resolved verbally on-site, no photo taken
  - Sub-role mismatch (work_manager closed a stage normally requiring
    safety_officer signoff)

================================================================================
DESIGN DECISIONS (defaults — Zahi may revise)
================================================================================

D1. Stage-level only — no item-level or bulk override in MVP. PM Overrides
    one stage at a time. If multiple stages need closure, PM clicks into
    each individually. Bulk is a v2 polish.

D2. Auto-mark items as "pass" with override flag — when a stage is
    overridden, all its items get `status="pass"` + `marked_via_override=true`.
    This keeps the UI consistent (items show as "תקין"), the audit trail
    preserved (the override flag is queryable), and aggregate counts
    sensible. The alternative (leave items as "pending") would create a
    confusing UI where stage is approved but items appear unmarked.

D3. Reason is mandatory, min 10 chars, max 500 — prevents trivial
    reasons like "אישור" or "ok". Forces meaningful documentation.

D4. RBAC: project_manager ONLY (membership.role on this project) +
    super_admin. NOT management_team generally. Override is a powerful
    action; restricting prevents accidental closures by junior staff.
    Junior management can still mark items + submit normally.

D5. UI placement: StageDetailPage only in MVP. PM clicks into a stage
    card, sees an "Override close" button visible only to project_manager
    + super_admin, click → confirm dialog with reason textarea → submits.
    Future polish: add badge on FloorDetailPage stage cards showing
    "via override" + a per-stage quick-action button.

D6. Endpoint URL: POST /qc/run/{run_id}/stage/{stage_id}/override-approve
    (consistent with existing /approve, /reject, /reopen siblings).

D7. Audit event type: "qc_stage_override_approved" (new event distinct
    from "qc_approved") — makes querying overrides trivial.

D8. Notification action: "approved_via_override" (new action distinct
    from "approved") — recipients can be informed, future polish can
    style notifications differently.

   ADD:
       @router.post("/run/{run_id}/stage/{stage_id}/override-approve")
       async def override_approve_stage(
           run_id: str,
           stage_id: str,
           body: OverrideApproveBody,
           user: dict = Depends(get_current_user)
       ):
           """PM Override — close a stage WITHOUT per-item validation.

           Use case: PM joins existing project mid-construction; old stages
           physically done but no in-app documentation. Normal submit/approve
           flow blocks because items have no photos/notes/status.

           Restricted to:
             - membership.role == 'project_manager' on this project
             - OR super_admin
           Junior management_team CANNOT override.

           Side effects:
             - All items in the stage marked 'pass' with marked_via_override=true
             - stage_statuses[stage_id] = 'approved'
             - stage_actors[stage_id] populated with override metadata
             - Audit log entry: 'qc_stage_override_approved'
             - Notifications: action='approved_via_override'
           """
           db = get_db()

           run = await db.qc_runs.find_one({"id": run_id}, {"_id": 0})
           if not run:
               raise HTTPException(status_code=404, detail="QC run not found")

           project_id = run["project_id"]

           # RBAC — project_manager only (membership-scoped) + super_admin
           role = await _get_project_role(user, project_id)
           if role != "project_manager" and not _is_super_admin(user):
               raise HTTPException(
                   status_code=403,
                   detail="Override close requires project manager role"
               )

           reason = (body.reason or "").strip()
           # Pydantic already validates min/max length, but defensive check:
           if len(reason) < 10:
               raise HTTPException(status_code=400, detail="סיבה חייבת להכיל לפחות 10 תווים")

           stage_statuses = run.get("stage_statuses", {})
           if stage_statuses.get(stage_id) == "approved":
               raise HTTPException(status_code=400, detail="שלב זה כבר אושר")

           tpl = await _get_template(db, run=run, project_id=project_id)
           # Verify stage exists in template (defensive)
           tpl_stage = next((s for s in tpl["stages"] if s["id"] == stage_id), None)
           if not tpl_stage:
               raise HTTPException(status_code=404, detail="Stage not found in template")

           actor = _actor_name(user)
           now = _now()

           # Side effect 1 — mark all items in this stage as pass + override flag
           await db.qc_items.update_many(
               {"run_id": run_id, "stage_id": stage_id},
               {"$set": {
                   "status": "pass",
                   "marked_via_override": True,
                   "updated_by": user["id"],
                   "updated_at": now,
               }}
           )

           # Side effect 2 — stage status + override metadata
           await db.qc_runs.update_one(
               {"id": run_id},
               {"$set": {
                   f"stage_statuses.{stage_id}": "approved",
                   f"stage_actors.{stage_id}.approved_by": user["id"],
                   f"stage_actors.{stage_id}.approved_by_name": actor,
                   f"stage_actors.{stage_id}.approved_at": now,
                   f"stage_actors.{stage_id}.via_override": True,
                   f"stage_actors.{stage_id}.override_reason": reason,
               }}
           )

           # Audit — distinct event type for queryability
           await _audit("qc_stage", stage_id, "qc_stage_override_approved", user["id"], {
               "run_id": run_id,
               "floor_id": run.get("floor_id"),
               "project_id": project_id,
               "stage_id": stage_id,
               "override_reason": reason,
               "actor_name": actor,
               "stage_label": _stage_label(tpl, stage_id),
           })

           # Notifications — distinct action so future UIs can style.
           # AMENDMENT (review feedback 2026-05-04): filter out the PM
           # who performed the override — no point notifying them of their
           # own action (annoying + creates phantom notification badges).
           try:
               recipients = await _get_stage_recipients(db, project_id, stage_id, "approved")
               # Self-notification filter — exclude actor from recipients
               recipients = [r for r in recipients if r.get("user_id") != user["id"]]
               if recipients:
                   await _create_qc_notification(
                       db, recipients,
                       project_id=project_id, stage_id=stage_id,
                       floor_id=run.get("floor_id"), building_id=run.get("building_id"),
                       stage_code=stage_id, stage_label_he=_stage_label(tpl, stage_id),
                       actor_id=user["id"], actor_name_str=actor,
                       action="approved_via_override", run_id=run_id,
                   )
           except Exception as e:
               logger.error(f"[QC] Failed to create notifications for override: {e}")

           logger.info(
               f"[QC] Stage {stage_id} OVERRIDE APPROVED in run {run_id} "
               f"by user {user['id']}, reason: {reason[:50]}"
           )

           return {
               "status": "approved",
               "stage_id": stage_id,
               "via_override": True,
           }
