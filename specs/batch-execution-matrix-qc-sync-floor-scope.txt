PROBLEM (Zahi staging test 2026-05-06)
================================================================================

Test results from staging with MATRIX_QC_SYNC_ENABLED=true:

  ✅ Test ב' (Override-approve on ריצוף דירה): worked
  ❌ Test א' (item update): no sync
  ❌ Test ג' (multi-cell floor-scope): no sync (expected per D5)
  ❌ Test ד' (RED on first failed): no sync
  ❌ Test ה' (manual override conflict): no sync (depends on tests א/ד)

Root cause: helper has D5 enforcement:

    # D5 — only unit-scope stages sync. Floor-scope stages deferred.
    if stage_scope != "unit":
        return None

And the codebase consistently uses `tpl_stage.get("scope", "floor")`
as DEFAULT — meaning ANY stage in qc_template that doesn't have
explicit `"scope": "unit"` is treated as floor and skipped.

Audit of seed templates — only ONE stage has explicit scope: "unit":
  stage_ceiling_prep   (default → floor)  ❌ skipped
  stage_blocks_row1    (default → floor)  ❌ skipped
  stage_plumbing       (default → floor)  ❌ skipped
  stage_electrical     (default → floor)  ❌ skipped
  stage_plaster        (default → floor)  ❌ skipped
  stage_waterproof     (default → floor)  ❌ skipped
  stage_acoustic       (default → floor)  ❌ skipped
  stage_tiling         scope: "unit"      ✅ syncs

Result: 7 of 8 base stages don't sync. The matrix sync feature is
~12% functional in production.

================================================================================
SOLUTION
================================================================================

Remove the 2-line skip in the helper. The existing logic already
handles BOTH scopes correctly:

  - Unit-scope: run.unit_id → sync ONE matrix cell
  - Floor-scope: items have per-unit unit_ids → caller (qc_router)
    aggregates distinct unit_ids and calls helper per-unit (this is
    Replit's drift C multi-cell aggregation, ALREADY IMPLEMENTED).

The skip was over-defensive. Removing it lets the existing aggregation
work for both scopes uniformly.

D5 update: NO LONGER deferred. Floor-scope stages sync correctly via
the per-unit aggregation that's already in place.

🚨 PROCESS LESSON LEARNED — for future spec checklists
================================================================================

Original #503 spec assumed default field values matched the conceptual
intent (i.e. unit-scope was common). Real production data showed only
1 of 8 stages had explicit scope: "unit"; the rest defaulted to floor
and were silently skipped.

CHECKLIST ADDITION for future specs that gate on field values:
  - Verify default field values match REAL production data, not just
    the schema definition.
  - Run a quick `grep` over the seed data / real DB sample to count
    how many records have the value the spec assumes.
  - If <50% match the assumed default → spec is high-risk → either
    update the default or document the gap explicitly.

This is added to the BrikOps standing rules cycle — see CLAUDE.md
update queued for after #503 ships.
