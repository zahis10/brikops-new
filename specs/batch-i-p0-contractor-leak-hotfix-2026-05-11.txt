BATCH I — P0 HOTFIX: Contractor leaks into management views (security)
================================================================================
Date:           2026-05-11
Branch:         staging  (NOT main)
Severity:       **P0 — SECURITY / PRIVACY**
                During Batch G smoke (2026-05-11), Zahi tested as a real
                contractor user clicking the WhatsApp deep-link to a
                defect. After viewing the defect, clicking the back
                arrow landed the contractor on
                `/projects/{id}/buildings/{id}/defects` — the
                management-only BuildingDefectsPage.

                **TWO bugs combined cause this:**

                1. **Frontend bug** (TaskDetailPage.js:807-819):
                   Back button defaults to BuildingDefectsPage if
                   task has building_id, regardless of user role.
                   Contractor lands on an admin page.

                2. **Backend bug** (projects_router.py:1397):
                   `/buildings/{id}/defects-summary` calls
                   `_check_project_read_access` which contractors
                   PASS (they're project members). Backend returns
                   the full building data including all tasks
                   (`db.tasks.find({'building_id': X})` — no role
                   filter). Contractor can see colleagues' defects.

                Real-world impact: contractor on plot 808 reached
                BuildingDefectsPage and saw 3 active defects
                (filtered by their installer category client-side,
                but backend returned ALL tasks unfiltered). The page
                is meant for PMs/management only — never for
                contractors.

Approach:       (per Zahi 2026-05-11 — Option 1)
                Contractors going back from a defect deep-linked via
                WhatsApp should land on **ContractorDashboard** (`/`),
                NOT on any management view.

                Backend `/buildings/{id}/defects-summary` requires
                management role (project_manager or management_team).
                Returns 403 for contractors regardless of project
                membership.

@router.get("/buildings/{building_id}/defects-summary")
async def building_defects_summary(building_id: str, user: dict = Depends(get_current_user)):
    db = get_db()
    building = await db.buildings.find_one({'id': building_id, 'archived': {'$ne': True}}, {'_id': 0})
    if not building:
        raise HTTPException(status_code=404, detail='Building not found')
    project_id = building['project_id']

    # BATCH I (P0 — 2026-05-11): gate by management role.
    # Previously used _check_project_read_access which contractors PASS
    # (they're project members). This page returns aggregate data and
    # is intended for PMs/management only. Contractors get redirected
    # to ContractorDashboard via the frontend; defense-in-depth here
    # ensures backend never leaks the data even if someone hand-crafts
    # the URL.
    await _check_project_read_access(user, project_id)  # keep — ensures user belongs to project at all
    project_role = await _get_project_role(user, project_id)
    if project_role not in MANAGEMENT_ROLES:
        raise HTTPException(
            status_code=403,
            detail='רק מנהל פרויקט או צוות ניהול יכולים לצפות בסיכום ליקויי בניין'
        )

Why both checks:
- `_check_project_read_access` still useful — ensures user is part of
  the project at all (some random user with a guessed building_id
  shouldn't even get to the role check).
- `_get_project_role` + `MANAGEMENT_ROLES` — narrows to management.
  Returns 'project_manager' for super admins and org owners (per
  router.py:372-390), so they retain access automatically.

================================================================================
RISK
================================================================================
Medium-low. P0 priority but very small surface area.

Risk vectors:
- ⚠ **Org owner / super admin regression** — `_get_project_role` returns
  'project_manager' for these cases (router.py L373, L387). Both
  bypass MANAGEMENT_ROLES check (since 'project_manager' IS in the
  tuple). Verified safe in helper code.
- ⚠ **Management role typo / drift** — if backend's MANAGEMENT_ROLES
  ever drifts from ('project_manager', 'management_team'), the check
  could lock out legitimate users. STEP 0 verified the constant at
  router.py:413. If extended in future, this endpoint extends naturally.
- ⚠ **Contractor with mixed role** — a user who is contractor in
  Project A but PM in Project B. `_get_project_role(user, project_id)`
  scopes correctly to the specific project. So a PM-in-B opening a
  building summary for project B passes; same user as contractor-in-A
  on project A is correctly blocked. No cross-project leakage.
- ⚠ **Frontend race condition** — if backend deploys after frontend,
  there's a brief window where the new TaskDetailPage routes to `/`
  but the BuildingDefectsPage backend still permits contractors.
  No harm — frontend just bypasses the issue. Reverse order (backend
  first, frontend lags) also safe — contractor's old back-button
  takes them to BuildingDefectsPage which 403s and bounces.

Bundle size impact: zero (no new imports).
Backend code size impact: ~6 LOC.
