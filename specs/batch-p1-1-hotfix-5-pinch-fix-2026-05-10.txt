CONTEXT — failure analysis
==========================

What's deployed (hotfix-4):
  usePinch(
    ({ offset: [s], event }) => {
      if (event?.cancelable) event.preventDefault();
      setScale(Math.min(Math.max(s, 0.1), 4.0));
    },
    {
      target: containerRef,
      eventOptions: { passive: false },
      scaleBounds: { min: 0.1, max: 4.0 },
      rubberband: true,
      from: () => [scale ?? 1.0, 0],
    }
  );

Observed on iPhone:
  - Two-finger pinch: NO zoom response (#32 still broken).
  - Touching canvas with one finger: scale instantly collapses to
    fit-scale (~15% on iPhone) regardless of current scale.

Two distinct root causes (likely):

**Cause A (the touch-collapse bug):**

`from: () => [scale ?? 1.0, 0]` is defined in a closure during each
render. The closure captures `scale` from THAT render. On re-render
(after +/- button zoom), a new closure is created with new scale.

BUT — @use-gesture/react's internal state machine may cache the
config from when the gesture was last bound, NOT re-evaluate on every
render. If that's the case:
  - Initial mount: scale=null → fit-scale=0.15 computed → re-render
  - useGesture binds with from() returning [0.15, 0]
  - User clicks + → scale=0.6 → re-render
  - But useGesture's bound from() still returns [0.15, 0] (stale)
  - User touches canvas → useGesture fires (perhaps even on single
    touch as "gesture start") → offset=[0.15, 0] (the stale from value)
  - Handler: setScale(clamp(0.15)) → scale collapses to 0.15

PATTERN B (explicit ref) is immune: the ref is updated on EVERY render
via useEffect, so reading `gestureStartScaleRef.current` always sees
the latest scale.

**Cause B (the no-pinch-on-iOS bug):**

iOS Safari fires GestureEvent (Apple's proprietary multi-touch API) in
addition to TouchEvents. @use-gesture/react v10 has two code paths:
  - Default: subscribes to native GestureEvent on iOS, computes pinch
    from gesture.scale (Apple's reported pinch ratio).
  - With `pointer: { touch: true }`: uses Pointer Events / Touch Events
    only, computes pinch from two-touch coordinates manually.

On iOS Capacitor (WKWebView), GestureEvent firing is sometimes
suppressed (especially when content is inside scroll containers with
custom touch-action). The default code path may register zero pinches.

Adding `pointer: { touch: true }` forces the cross-platform Touch Event
path, which is what Android uses anyway and which is more reliable on
iOS Capacitor.

Defensive: also add a temporary `console.log` in the handler so Zahi
can verify the handler IS firing (or NOT) via Safari Web Inspector +
Capacitor debug bridge.

LOCATION 1.2 — Replace PATTERN A with PATTERN B in usePinch

   CURRENT (the usePinch hook from hotfix-4):
     usePinch(
       ({ offset: [s], event }) => {
         if (event && event.cancelable) event.preventDefault();
         setScale(Math.min(Math.max(s, 0.1), 4.0));
       },
       {
         target: containerRef,
         eventOptions: { passive: false },
         scaleBounds: { min: 0.1, max: 4.0 },
         rubberband: true,
         from: () => [scale ?? 1.0, 0],
       }
     );

   REPLACE WITH:
     // 2026-05-10 hotfix-5 — PATTERN B (explicit ref + first flag).
     //
     // PATTERN A's `from` config was vulnerable to closure staleness:
     // @use-gesture appears to cache the bound config and not re-
     // evaluate `from()` on every render. Result: `from()` always
     // returned the INITIAL scale (the fit-scale, e.g. 0.15), so any
     // touch on the canvas reset scale to 0.15.
     //
     // PATTERN B reads scale from a ref that's updated every render
     // via useEffect (LOCATION 1.1). On gesture start (`first` flag),
     // we capture the current ref value into gestureStartScaleRef,
     // then multiply by the cumulative offset[0] each frame to compute
     // the new absolute scale.
     //
     // ALSO: `pointer: { touch: true }` forces the cross-platform Touch
     // Event path. Default uses iOS GestureEvent which is sometimes
     // suppressed on Capacitor WKWebView. Touch Events are what Android
     // already uses; standardizing on them gives consistent behavior.
     //
     // Debug log: helps verify the handler IS firing on iPhone/Android.
     // Wrapped in `process.env.NODE_ENV !== 'production'` so CRA's
     // build-time dead-code-elimination strips it entirely from the
     // production bundle. No manual "remember to remove" step needed,
     // no risk of log shipping to prod.
     const gestureStartScaleRef = useRef(1.0);
     usePinch(
       ({ offset: [s], first, event, type }) => {
         if (event && event.cancelable) event.preventDefault();
         if (first) {
           gestureStartScaleRef.current = scaleRef.current ?? 1.0;
         }
         const next = gestureStartScaleRef.current * s;
         const clamped = Math.min(Math.max(next, 0.1), 4.0);
         if (process.env.NODE_ENV !== 'production') {
           // eslint-disable-next-line no-console
           console.log('[PlanViewer pinch]', {
             type, first, offset_s: s,
             startScale: gestureStartScaleRef.current,
             liveScale: scaleRef.current,
             nextClamped: clamped,
           });
         }
         setScale(clamped);
       },
       {
         target: containerRef,
         eventOptions: { passive: false },
         scaleBounds: { min: 0.1, max: 4.0 },
         rubberband: true,
         pointer: { touch: true },
       }
     );

   Key differences from hotfix-4:
   - Removed `from` config (was the staleness vector).
   - Added `pointer: { touch: true }` to force Touch Event path on iOS.
   - Read scale from `scaleRef.current` (always fresh) not closure.
   - Capture gesture-start scale on `first` frame to a ref.
   - Multiply gesture-start scale by cumulative offset[0] (PATTERN B).
   - Added temporary console.log for debugging on real device.

**Option A — Native Capacitor plugin:**
  - `@capacitor-community/pinch-zoom` or similar (verify maintenance)
  - Works at native layer, bypasses WebView gesture issues entirely
  - Pros: no WebView gesture quirks
  - Cons: native build required (`./ship.sh`), Android + iOS plugin
    parity needed, heavier dependency

**Option B — Manual touch event handlers (lowest layer):**
  - Listen to native `touchstart` / `touchmove` / `touchend` on the
    container directly (React onTouch* props)
  - Track two-touch distance, compute pinch ratio frame-to-frame
  - ~50 LOC, no library dependency
  - Pros: lowest layer = most reliable, works identically on iOS
    and Android because both fire TouchEvent
  - Cons: more code to maintain, doesn't get us desktop wheel zoom
    for free (but +/- buttons cover desktop)

**Option C — Defer pinch to P1.4 polish (RECOMMENDED if iterating):**
  - Ship P1.1 with +/- buttons + scrollbar pan only
  - Field-test for 1-2 weeks (WUyoughong + others)
  - Real users may not even need pinch (gloves + dirty hands +
    field conditions favor buttons over gestures anyway)
  - If users complain, address pinch in P1.4 with fresh approach
    informed by actual usage patterns
  - Pros: stops the iteration spiral, gets P1.1 stable in field,
    informed by data not theory
  - Cons: delays "professional viewer" feel by ~2 weeks
