CONTEXT — WHAT 5B GOT WRONG
---------------------------
Batch 5B Bug B added refresh-on-focus to UnitDetailPage. That part works —
the page DOES refetch when user returns. But the staleness Zahi saw isn't
caused by missing refresh. It's caused by a status-string mismatch between
backend and frontend:

  - Two different "close" endpoints write DIFFERENT status strings to DB:
      • POST /tasks/{id}/force-close   (פאנל ניהול direct close)
        → writes status='approved'   (backend/contractor_ops/tasks_router.py:982)
      • POST /tasks/{id}/approve      (after pending_manager_approval)
        → writes status='closed'

  - Frontend assumes ONLY 'closed' means done. Status maps don't include
    'approved'. The list-item renderer falls back to STATUS_LABELS.open
    when status is unmapped:

        const statusInfo = STATUS_LABELS[task.status] || STATUS_LABELS.open;

    → A task closed via פאנל ניהול renders with the RED "פתוח" badge
      forever, even though backend correctly set status='approved'.

  - Backend KPI / bucket aggregations only count 'closed' as closed in
    several places. Tasks in 'approved' status are counted as NEITHER open
    NOR closed → "סגורים 0 / פתוחים 1 / בטיפול 1" + 3 visible cards adds
    up wrong because the approved card is in NO bucket.

WHY THIS MATTERS — Pre-launch this is the most common close path during
manual QA: the manager taps "סגירה ידנית" or "אישור ישיר" in the panel.
Production users won't have a contractor sending photos for every defect.
If we ship this bug, every directly-closed defect will look open forever.

DECISION — Treat 'approved' as a closed-equivalent for DISPLAY and
AGGREGATION, but keep both as distinct DB values for audit trail
(force-close vs. proof-then-approve are meaningfully different events).

TEST 1 — force_close writes 'approved' and audit fields:
  - Setup: create a task in status='in_progress', PM membership.
  - Call: POST /api/tasks/{id}/force-close with body
    {'reason': 'בדיקה ידנית', 'close_type': 'no_contractor_response'}
  - Assert: 200 OK
  - Assert: r.json()["task"]["status"] == "approved"
  - Assert: force_closed_by == PM user id
  - Assert: force_closed_reason == 'בדיקה ידנית'
  - Assert: force_closed_at is ISO timestamp string
  - Assert: force_closed_type == 'no_contractor_response'

TEST 2 — KPI rollup includes 'approved' in closed bucket:
  - Setup: create 3 tasks in same project: 1 open, 1 closed (normal), 1 approved (force-closed).
  - Call: GET /api/stats/projects/{project_id}/summary
  - Assert: response.open_defects == 1 (NOT 2 — the approved one must NOT count as open)
  - Assert: response.per_building_defects[building_id].closed includes
    BOTH the closed and approved tasks (i.e., 2 closed, not 1).

TEST 3 — Project dashboard counts approved as closed:
  - Same 3-task setup.
  - Call: GET /api/projects/{project_id}/dashboard
  - Assert: closed_count == 2 (closed + approved combined).

TEST 4 — Approved task can be reopened:
  - Setup: force-close a task to status='approved'.
  - Call: POST /api/tasks/{id}/reopen
  - Assert: 200 OK
  - Assert: status changed to 'reopened'.
  - Assert: force_closed_* fields are unset.

These 4 tests are CHEAP to write (~30-40 minutes of Replit time) and
HIGH-VALUE — they cover the entire force_close → aggregation → reopen
loop. Without them, the next person who adds a status will face the
same 3-batch debugging cycle.
