TWO BUGS, ONE BATCH
================================================================================

BUG A — Contractor can't click on a project in MyProjectsPage:
  User who is PM in project A AND contractor in project B sees both
  in MyProjectsPage. PM project click works (→ /control). Contractor
  project click does nothing visible.

  Root cause (frontend/src/utils/navigation.js L10-11):
      } else if (role === 'contractor') {
          navigate('/projects');
      }

  /projects route renders ProjectsHome (App.js L147). ProjectsHome
  checks user.role globally → it's 'project_manager' (since user is PM
  in project A) → returns MyProjectsPage. Same page user is already on.
  The click effectively no-ops.

  An existing escape hatch exists at App.js L163:
      const fromContractorReminder = searchParams.get('src') === 'wa';
  When src=wa is present, ProjectsHome forces ContractorDashboard
  regardless of global role. This is currently used only by WhatsApp
  reminder links.

  Fix: extend the escape hatch to also accept src=contractor, and have
  navigateToProject pass src=contractor + the project id.

BUG B — PM cannot assign a defect to a company-only (no specific contractor):
  Today's management panel "קבלן" dropdown lists:
   - "לא שויך" — clears assignment.
   - Each contractor user — assigns specific user, auto-fills company.
   - Per company that has contact_name AND has NO contractors in the
     project — assigns company only (no user).

  After P0 batch (2026-05-08), any contractor in the assigned company
  can submit proof — so company-only assignment is a useful workflow.
  But the PM has no way to PICK company-only when the company DOES
  have contractors registered (the filter at L1087 excludes those).

  Root cause (TaskDetailPage L1086-1088):
      const contactFallbacks = projectCompanies
          .filter(pc => pc.contact_name && !projectContractors.some(...))
          .map(...)

  Fix: add a separate "company-only" option list that includes ALL
  project companies (regardless of whether they have contractors),
  visually distinct from specific-contractor options.

================================================================================
KNOWN LIMITATION (document, defer fix)
================================================================================

contactFallbacks and companyOnlyOptions both emit the same value
format: `__contact__${pc.id}`. On the backend, both write the same
row: { company_id: pcId, assignee_id: null }.

Implication: when the page reloads after PM picked a contact_name
option (e.g. "ירון יורון (גמרים)"), the label resolver will display
"🏢 גמרים — כל החברה" because the resolver only sees company_id +
assignee_id=null, with no way to distinguish "contact-name intent"
from "company-only intent".

Both options route to the same workflow (any company member can
submit proof), so functionally there's no difference. Only the
display label changes after reload.

Acceptable trade-off for P1. Proper fix would require a backend
`assignment_type` field (e.g. 'contact_name' vs 'company_only').
Deferred until a PM reports actual confusion.

Add to CLAUDE.md under known limitations.
