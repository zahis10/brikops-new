BATCH 510 — Billing banner project-aware (suppress on projects where user is contractor/viewer) (P1)
================================================================================
Date:           2026-05-09
Branch:         staging  (NOT main)
Severity:       P1 — UX bug. Contractors invited to projects of OTHER orgs
                see a misleading "התשלום פג תוקף — מצב צפייה בלבד" banner
                because the banner reads the LOGGED-IN USER's org billing,
                not the CURRENT PROJECT's org billing. Confusing — may make
                contractors think the project they were invited to is broken
                when in fact the OTHER org pays and the project is fine.
Files changed:  1 frontend (no backend changes)
                  Frontend: frontend/src/components/TrialBanner.js
                  (~15 lines added: useEffect to fetch project membership +
                   1 new state + 1 new early-return check)
Pre-flight:     The bug is per-user, not per-project. Banner uses
                useBilling() which fetches /billing/me (user's own org).
                When user is contractor in Project A (paid by Org A) but
                also PM in Project B (own Org B, expired), entering
                Project A still triggers the banner because user.role=
                project_manager and their org_billing is read_only.

                Discovery: prod test 2026-05-09 — Zahi as user "צחי שמי"
                with Project 808 (PM, expired org) + Project 810
                (contractor, other org paid). Banner showed on BOTH.

                Backend `/api/projects/{id}` already returns `my_role`
                for any user with read access (contractor/viewer included)
                — confirmed projects_router.py:374-396. So frontend-only
                fix is viable: extract projectId from URL, fetch project,
                read my_role, suppress banner if contractor/viewer.

The role check at L49 uses `roleDisplay` (from billing.role_display, which
is the user's role in their OWN ORG) and `user.role` (platform-level role,
not project-level). Neither reflects "what's my role in the project I'm
currently looking at?"

For Zahi's case:
  - user.role = 'project_manager' (he's PM somewhere, his platform role)
  - billing.role_display = 'owner' or 'project_manager' (he owns his org)
  - Neither is 'contractor', so L49 doesn't return null
  - Banner renders based on billing.read_only = true (his own org expired)

What we want:
  - When viewing Project A where Zahi is contractor → suppress banner
    (he can't pay for someone else's org anyway)
  - When viewing Project B where Zahi is PM (his own org) → show banner
    (his org is expired, he needs to renew)

PHASED FIX SCOPE — this batch only:
  - Suppress banner when user is `contractor` or `viewer` in CURRENT
    project. Fixes the most common false-positive (the contractor case).
  - **Phase 1 = triage, Phase 2 = the real fix.** The deeper architectural
    correction is to read the CURRENT project's org subscription rather
    than the logged-in user's org — that handles ALL edge cases (contractor
    cross-org, PM cross-org, viewer cross-org, etc.). Phase 1 covers the
    most common ~90% of confusion (contractor cross-org) and ships fast.

Out of scope (Phase 2 — track as follow-up batch):
  - Banner reads currentProject.org.subscription_status instead of
    user.org.subscription_status. Handles: user is PM in someone else's
    org's project where their org pays but user's own org is expired.
    Edge case; ship Phase 1 first to unblock contractors today, do the
    architectural fix when there's bandwidth.

Known optimization (defer — also Phase 2):
  - This batch adds 1 axios call to /api/projects/{id} per project-page
    visit (TrialBanner fetches my_role independently from
    ProjectControlPage's own fetch of the same endpoint). Duplicate
    fetch — wasteful, ~5–10 ms per page, but not broken (browser cache
    will dedupe most of it within session).
  - Future: bubble my_role up via a shared "current project context"
    so banner reads from cached state. Defer until measurable.

   REPLACE WITH:
       const TrialBanner = () => {
         const { user, token } = useAuth();
         const { billing, loading, isReadOnly, isOwner, canManageBilling, isOrgPm, setShowPaywall } = useBilling();
         const location = useLocation();
         const navigate = useNavigate();
         const [dismissed, setDismissed] = useState(() => {
           try { return sessionStorage.getItem('trial_banner_dismissed') === 'true'; } catch { return false; }
         });

         // 2026-05-09 — Track user's role in the CURRENT project (when on a
         // project page). Banner suppresses for contractor/viewer because
         // they can't pay for the project's owning org anyway. Fixes a
         // false-positive where contractor in Project A (paid by Org A)
         // saw expired banner just because their OWN org's subscription
         // was expired — confusing, made them think Project A was broken.
         const projectMatch = location.pathname.match(/^\/projects\/([a-f0-9-]{36})/);
         const projectId = projectMatch ? projectMatch[1] : null;
         const [currentProjectRole, setCurrentProjectRole] = useState(null);

         useEffect(() => {
           if (!projectId || !token) {
             setCurrentProjectRole(null);
             return;
           }
           let cancelled = false;
           axios.get(`${BACKEND_URL}/api/projects/${projectId}`, {
             headers: { Authorization: `Bearer ${token}` },
           })
             .then(res => {
               if (!cancelled) setCurrentProjectRole(res.data?.my_role || null);
             })
             .catch(() => {
               if (!cancelled) setCurrentProjectRole(null);
             });
           return () => { cancelled = true; };
         }, [projectId, token]);

         if (AUTH_PAGES.includes(location.pathname)) return null;
         if (PROJECTS_LIST_PAGES.includes(location.pathname)) return null;
         if (!user || loading || !billing) return null;

NOTES on LOCATION 1.2 + 1.3 combined:
  - The new useEffect runs on every projectId/token change (rare —
    typically once per project visit). One axios call per project page
    visit; lightweight (~2KB response, cached by browser if URL matches).
  - State lives in the banner component; doesn't pollute global context.
  - cancelled flag prevents stale state writes if user navigates fast.
  - When user navigates from /projects/A to /projects/B → useEffect
    fires again with new projectId → role updates correctly.
  - When user navigates from /projects/A to /admin → projectId becomes
    null → currentProjectRole reset to null → banner uses default behavior
    (driven by user.role / billing.role_display) — same as before fix.
  - useEffect runs BEFORE the early returns at L41-43 (because hooks must
    not be called conditionally per React rules). The early returns at
    L41-43 are still hit if location is in AUTH_PAGES etc., and the
    useEffect's fetch is harmless (returns 404 or whatever — caught).
