DESIGN — Excel pattern, drawer placement, dynamic sections
================================================================================

CORE METAPHOR (Zahi 2026-05-05): "תמיד תחשוב על זה כעל טבלת אקסל...
טבלת אקסל משופרת". Every column in the matrix gets its own filter.
AND between columns, OR within a column.

PLACEMENT: Drawer from right (mobile + desktop). Decision made by Zahi
2026-05-05 over alternatives (inline chips / popover). Reasoning:
  - 5+ filter categories don't fit in inline chips on mobile
  - Drawer is the established pattern in BrikOps (BuildingDefectsPage)
  - Zero pixels stolen from the matrix grid (critical on 30-unit screens)
  - Roomy enough for dynamic per-column sections + saved views

OPENING: Floating Action Button (FAB) at bottom-right of matrix page.
Active filter count rendered as a numeric badge on the FAB. Clicking
opens the drawer.

FILTER LOGIC:
  - Within a section (a single column's checkboxes): OR
    Example: tag "סוג מכירה" with [שוק חופשי, משתכן] checked → unit
    matches if cell value is EITHER.
  - Between sections (different columns): AND
    Example: building=A AND סוג מכירה=שוק חופשי AND טיח status=הושלם
    → unit must satisfy ALL.
  - Empty value: explicit "(ריק)" checkbox per section. NOT default-on.

DRAWER STRUCTURE (top-down):
  1. Header: title "סינון" + active count badge + X close button
  2. Saved views row: pills of user's saved views + "+" to save current
  3. Free-text search input: matches unit_no + cell text_value + cell note
  4. Group "📍 מיקום":
       - בניין: checkboxes per building (from data.buildings)
       - דירה: text input (filter by unit_no contains)
  5. Group "📝 מידע כללי" — DYNAMIC:
       - Renders one section per stage where stage.type === "tag"
       - Each section: checkboxes for distinct text_value across all
         cells of that stage + "(ריק)" + optional "+ ערך" text input
         for free-text "contains" matching
  6. Group "🔨 שלבי בקרת ביצוע" — DYNAMIC:
       - Renders one section per stage where stage.type === "status"
       - Each section: 6 status checkboxes (הושלם / בוצע חלקית / בתהליך
         / לא הושלם / אין צורך / אין חוסרים) + "(ריק)"
  7. Footer: "החל (N)" + "נקה הכל"

SECTION COLLAPSIBLE BEHAVIOR (Zahi explicit request 2026-05-05):
  "תוודא שיש חץ לקיפול נושאים או משהו כדי שלא יהיה עמוס מדי"
  - Default: ALL sections collapsed (just header row, no body)
  - Header shows: name + icon + active count badge if active + chevron
  - Chevron: ▸ collapsed (default), ▾ expanded
  - Click anywhere on header row → toggle expand/collapse
  - Auto-expand when loading a saved view that has values in that section
  - "Expand all / Collapse all" mini-button at top of drawer (post-MVP polish — out of 2D-1 scope, see open question Q3)

FILTER STATE LOCATION:
  - Live in component state (useMatrixFilters hook), NOT URL params.
  - Refresh resets to default.
  - Persistence = saved views (backend).
  - URL persistence deferred to 2D-2 (open question Q5 below).

PERFORMANCE:
  - Filtering is computed in useMemo over `units` + `cellsByUnitStage`
  - 30 units × 11 stages = 330 cells max. O(units × stages) per render.
  - No backend round-trip on filter change (everything is in-memory).

PART 1 — Filter state hook (useMatrixFilters)
================================================================================

NEW FILE: frontend/src/hooks/useMatrixFilters.js  (~120 LOC)

Responsibilities:
  - Hold filter state (building_ids, apartment_search, search_text,
    stage_status_filters, tag_value_filters)
  - Expose toggle helpers (toggleBuilding, toggleStageStatus, etc.)
  - Compute `filteredUnits` via useMemo (Excel logic: AND across, OR within)
  - Compute `activeCount` per section + total
  - Reset, setFromSavedView (load entire state in one call)

API:
    const {
      filters,
      filteredUnits,
      activeCount,        // { total, building, apartment, search,
                          //   stage_status: {stage_id: count},
                          //   tag_value: {stage_id: count} }
      toggleBuilding,
      toggleStageStatus,  // (stageId, statusValue) → toggle in array
      toggleTagValue,     // (stageId, value)
      setApartmentSearch,
      setSearchText,
      reset,
      loadSavedView,      // (viewObj) → applies viewObj.filters
      currentFilterPayload, // serializable shape matching backend schema
    } = useMatrixFilters({ units, cellsByUnitStage, stages });

Filter logic (filteredUnits):
    return units.filter(u => {
      // 1. Building (AND, OR within)
      if (filters.building_ids.length > 0
          && !filters.building_ids.includes(u.building_id)) return false;

      // 2. Apartment search (substring on unit_no)
      if (filters.apartment_search) {
        const q = filters.apartment_search.toLowerCase();
        if (!String(u.unit_no).toLowerCase().includes(q)) return false;
      }

      // 3. Per-stage status filters (AND across stages, OR within statuses)
      for (const [stageId, allowed] of Object.entries(
        filters.stage_status_filters || {}
      )) {
        if (!allowed?.length) continue;
        const cell = cellsByUnitStage[`${u.id}::${stageId}`];
        const status = cell?.status ?? '__empty__';
        if (!allowed.includes(status)) return false;
      }

      // 4. Per-tag value filters
      for (const [stageId, allowed] of Object.entries(
        filters.tag_value_filters || {}
      )) {
        if (!allowed?.length) continue;
        const cell = cellsByUnitStage[`${u.id}::${stageId}`];
        const value = cell?.text_value ?? '__empty__';
        if (!allowed.includes(value)) return false;
      }

      // 5. Global free-text search across cells + unit_no + notes
      if (filters.search_text) {
        const q = filters.search_text.toLowerCase();
        const inUnitNo = String(u.unit_no).toLowerCase().includes(q);
        const inCells = stages.some(s => {
          const c = cellsByUnitStage[`${u.id}::${s.id}`];
          return (c?.text_value ?? '').toLowerCase().includes(q)
              || (c?.note ?? '').toLowerCase().includes(q);
        });
        if (!inUnitNo && !inCells) return false;
      }

      return true;
    });

Distinct value helpers (memoized — used by tag sections):
    distinctTagValues(stageId) → unique list of cell.text_value across
                                 all cells of that stage (for checkboxes)

Empty-value sentinel:
    Use string '__empty__' for "(ריק)" mapping. NOT null/undefined to keep
    array.includes() checks clean. Frontend-only sentinel; the saved-view
    payload converts '__empty__' to literal "" before sending to backend.

  T4. activeCount counts every active section correctly (regression for

  - "🔨 שלבי בקרת ביצוע" — shown only if `stages.some(s => s.type === 'status')`

Sections rendering:
  Stages are sorted by `.order` (already sorted upstream by backend).
  Custom stages (negative order) render before template stages — same
  RTL right-to-left order as the matrix grid view, so the user's mental
  model maps 1:1.

"החל" button:
  - Currently does NOT exist as separate apply step — filters apply LIVE
    as user toggles. The "החל" button just CLOSES the drawer.
  - Why live: 30 units × 11 stages = 330 cells. Filtering is fast
    (memoized). Live feedback matches the Excel mental model — user
    sees rows disappear/appear in real-time.
  - Button label updates: "החל (N)" if N > 0, otherwise "סגור".

"נקה הכל" button:
  - Calls `reset()` from the hook → clears every filter section.
  - Saved-view selection cleared too (no longer "loaded").

Saved-view "+" button → opens a tiny SavedViewSaveDialog (PART 4 below).

Saved-view DELETE: each saved-view pill renders an "×" mini-button at
its right edge (in RTL — visually right side of the pill, not pixel-right).
Click "×" → calls `onDeleteSavedView(view.id)` → matrixService.deleteSavedView
→ refresh list → pill disappears.

  Pill markup (~5 LOC):
    <span className="view-pill" onClick={() => onLoadSavedView(view)}>
      {view.title}
      <button
        type="button"
        onClick={(e) => { e.stopPropagation(); onDeleteSavedView(view.id); }}
        className="ml-1 text-xs opacity-60 hover:opacity-100"
        aria-label={`מחק תצוגה ${view.title}`}
      >×</button>
    </span>

  Confirm before delete: window.confirm? → NO. Use a tiny inline
  confirm dialog matching #495 pattern? → defer. For 2D-1 ship the "×"
  with NO confirm — saved views are easy to recreate (just re-set
  filters + save). Worst case: undo the save right after.

  ⚠️ Why include in 2D-1 (was Q2): without delete UI, users accumulate
  stale views over time → support burden ("delete my old views via
  the DB please"). 5 LOC eliminates this entire support category.

Q1. SavedViewSaveDialog UX — Radix modal-in-modal vs inline input?
    Recommend: Radix modal (consistent with #495 stage management,
    accessible focus-trap). Inline input would reduce LOC by ~50 but
    creates focus-trap edge cases.
    Default: Radix modal (PART 4 above).

Q2. Saved-view delete UX — long-press? swipe-left? right-click?
    DECIDED 2026-05-05 (Replit feedback adopted): include minimal "×"
    button on each pill (PART 3 above, ~5 LOC). No confirm dialog
    (cheap to recreate views). Without this, users accumulate stale
    views → support load. Worth the 5 LOC.
    DECIDED: include in 2D-1.

Q3. "פתח/סגור הכל" button — include in 2D-1?
    Recommend: defer to 2D-2 (Zahi already separated 2D-2 explicitly).
    Default: skip; sections start collapsed; user expands what they need.

Q4. Swipe-to-close drawer on mobile?
    Recommend: defer to 2D-2. Radix Dialog backdrop-click + ESC + X
    button cover all entry points.
    Default: skip.

Q5. URL param persistence (e.g. ?building=A&stage_status[plaster]=completed)?
    Recommend: defer. Saved views give the durability story; URL would
    introduce a serialization format that has to be versioned.
    Default: skip; mention as 2E candidate in roadmap.

Q6. Should "(ריק)" be default-checked on every section?
    Recommend: NO. Default = nothing checked = show all. "(ריק)" is
    an EXPLICIT filter for "I want to see units missing a value here".
    Default: nothing checked.
