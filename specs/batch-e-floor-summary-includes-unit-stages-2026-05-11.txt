DEFINITIONS — what each status means
================================================================================

For a UNIT-SCOPE stage on a given floor (e.g. "ריצוף דירה" with 5 units):
  - "rejected"     ⟺ ANY unit has rejected status (PRIORITY — wins over
                     all other states. Rejected = work failed QC, needs
                     redo, not progress.)
  - "approved"     ⟺ ALL 5 units have approved status for this stage
                     AND no rejections AND total_units > 0
  - "in_progress"  ⟺ At least 1 unit has approved OR in_progress status
                     AND not all units approved AND no rejections
  - "not_started"  ⟺ Zero units have any progress (no approved,
                     no in_progress, no rejected)
  - "is_empty"     ⟺ Stage has no QC items in template (excluded from
                     all counts — neither numerator nor denominator)
  - "no_units"     ⟺ Floor has 0 units of any kind (excluded — also
                     overlaps with Batch D's filter)

Status priority (rejected > approved > in_progress > not_started)
applies at BOTH unit-stage level and floor-badge level. The whole
chain propagates: 1 unit rejected → stage rejected → floor rejected
→ red badge in BuildingQCPage. This is by design — PMs need rejected
work surfaced for immediate attention.

For the FLOOR's badge (delegated to `_compute_floor_badge`):
  - "approved"      ⟺ ALL stages (floor-scope + unit-scope, non-empty,
                      non-no-units) are "approved"
  - "rejected"      ⟺ ANY stage has rejected status (propagates up
                      from rejected unit-stages too)
  - "pending_review"⟺ At least one stage is pending_review
  - "in_progress"   ⟺ At least one stage has progress; not all approved
  - "not_started"   ⟺ All stages have zero progress

# Unit-scope stage aggregation helper (added 2026-05-11, batch E)
# Computes per-stage "approved/in_progress/not_started" status for
# unit-scope stages on a given floor, by aggregating across all unit
# runs for that floor. Used by /qc/floors/{floor_id}/run and
# /qc/floors/batch-status to include unit-scope stages in floor badge
# calculation.
# ──────────────────────────────────────────────────────────────────────

async def _compute_unit_scope_stage_statuses(
    db,
    project_tpl: dict,
    floor_id: str,
    floor: dict = None,
) -> list:
    """Per-floor version. Returns list of dicts in the same shape as
    `stages_data` used by `_compute_floor_badge`, one per unit-scope stage.

    Each dict: {"id": stage_id, "done": done_count, "computed_status": status}

    Where:
      done       = total units that have any progress (approved + in_progress + rejected)
      computed_status:
        "rejected"      → ANY unit in this stage has rejected status (priority over all)
        "approved"      → all units in this stage are approved AND no rejections
        "in_progress"   → at least 1 unit has progress; not all approved
        "not_started"   → 0 units have any progress

    Excludes:
      - Stages with no items in template (is_empty)
      - Stages on floors with 0 units (returns [])

    Used by /qc/floors/{floor_id}/run (single floor — per-floor query is fine).
    For /qc/floors/batch-status, use `_compute_unit_scope_stage_statuses_batch`
    which batches queries across all floors (3 queries total vs N*3).
    """
    if floor is None:
        floor = await db.floors.find_one(
            {"id": floor_id, "archived": {"$ne": True}},
            {"_id": 0}
        )
        if not floor:
            return []

    # Get all units on this floor
    units = await db.units.find(
        {"floor_id": floor_id, "archived": {"$ne": True}},
        {"_id": 0, "id": 1}
    ).to_list(500)
    unit_ids = [u["id"] for u in units]

    # If no units, no unit-scope stages apply
    if not unit_ids:
        return []

    # Get all unit-scope runs for these units
    unit_runs = await db.qc_runs.find(
        {"unit_id": {"$in": unit_ids}, "scope": "unit"},
        {"_id": 0, "id": 1, "unit_id": 1, "stage_statuses": 1}
    ).to_list(1000)

    # Map run_id → stage_statuses (per unit)
    runs_by_unit = {r["unit_id"]: r for r in unit_runs}

    return _aggregate_unit_stages_in_memory(project_tpl, unit_ids, runs_by_unit)


async def _compute_unit_scope_stage_statuses_batch(
    db,
    project_tpl: dict,
    floor_ids: list,
) -> dict:
    """Batch version for /qc/floors/batch-status. Returns dict:
        {floor_id: stages_data}

    Uses 2 queries total regardless of how many floors:
      1. All units across all floors (single $in query)
      2. All unit-scope runs for those units (single $in query)

    Then aggregates per-floor in memory. Replaces per-floor query loop
    which would cost N*2 round trips.

    For a typical project (~20 floors, ~6 units/floor):
      OLD: 40 DB queries
      NEW: 2 DB queries
    """
    if not floor_ids:
        return {}

    # 1. Single query for all units across all floors
    all_units = await db.units.find(
        {"floor_id": {"$in": floor_ids}, "archived": {"$ne": True}},
        {"_id": 0, "id": 1, "floor_id": 1}
    ).to_list(2000)

    units_by_floor = {}
    all_unit_ids = []
    for u in all_units:
        fid = u["floor_id"]
        units_by_floor.setdefault(fid, []).append(u["id"])
        all_unit_ids.append(u["id"])

    # 2. Single query for all unit-scope runs across all floors
    runs_by_unit = {}
    if all_unit_ids:
        all_runs = await db.qc_runs.find(
            {"unit_id": {"$in": all_unit_ids}, "scope": "unit"},
            {"_id": 0, "id": 1, "unit_id": 1, "stage_statuses": 1}
        ).to_list(2000)
        runs_by_unit = {r["unit_id"]: r for r in all_runs}

    # 3. Aggregate per floor in memory (no more DB queries)
    result = {}
    for floor_id in floor_ids:
        unit_ids = units_by_floor.get(floor_id, [])
        if not unit_ids:
            result[floor_id] = []  # floor with 0 units → no unit-scope stages
            continue
        result[floor_id] = _aggregate_unit_stages_in_memory(
            project_tpl, unit_ids, runs_by_unit
        )

    return result


def _aggregate_unit_stages_in_memory(
    project_tpl: dict,
    unit_ids: list,
    runs_by_unit: dict,
) -> list:
    """Pure function — given units and their runs (already fetched), compute
    per-stage aggregated status. No DB access.

    Extracted so per-floor and batch versions share identical logic.
    """
    # Filter to unit-scope stages with template items (skip is_empty)
    unit_stages = [s for s in project_tpl["stages"]
                   if s.get("scope") == "unit"
                   and len(s.get("items", [])) > 0]

    total_units = len(unit_ids)
    stages_data = []

    for stage in unit_stages:
        stage_id = stage["id"]
        approved = 0
        in_progress = 0
        not_started = 0
        rejected = 0

        for unit_id in unit_ids:
            run = runs_by_unit.get(unit_id)
            if not run:
                not_started += 1
                continue
            stage_status = run.get("stage_statuses", {}).get(stage_id, "draft")
            if stage_status == "approved":
                approved += 1
            elif stage_status == "rejected":
                # rejected = work failed QC, needs redo. NOT progress.
                # Tracked separately so the stage's badge propagates correctly.
                rejected += 1
            elif stage_status in ("draft", "not_started"):
                not_started += 1
            else:
                # in_progress, ready, submitted, pending_review — actively being worked
                in_progress += 1

        # Status priority: rejected > approved > in_progress > not_started.
        # ANY rejection means the stage needs attention — propagates up to
        # floor badge via _compute_floor_badge ("rejected" wins over others).
        if rejected > 0:
            computed = "rejected"
        elif approved == total_units and total_units > 0:
            computed = "approved"
        elif approved == 0 and in_progress == 0:
            computed = "not_started"
        else:
            computed = "in_progress"

        # Match shape of floor-scope stages_data for _compute_floor_badge.
        # `done` counts any unit with non-not-started status (so floor's
        # `has_any_marked` correctly fires).
        stages_data.append({
            "id": stage_id,
            "done": approved + in_progress + rejected,
            "computed_status": computed,
            # extras for stage_summary (not used by _compute_floor_badge):
            "_scope": "unit",
            "_approved_units": approved,
            "_in_progress_units": in_progress,
            "_not_started_units": not_started,
            "_rejected_units": rejected,
            "_total_units": total_units,
        })

    return stages_data

def _build_stage_summary(floor_stages_data: list, unit_stages_data: list) -> dict:
    """Combine floor-scope + unit-scope stages into a single summary
    for the frontend. Returns:
      {
        "total_stages": int,
        "approved_stages": int,
        "in_progress_stages": int,
        "not_started_stages": int,
        "pending_review_stages": int,
        "rejected_stages": int,
      }
    """
    all_stages = floor_stages_data + unit_stages_data
    return {
        "total_stages": len(all_stages),
        "approved_stages": sum(1 for s in all_stages if s["computed_status"] == "approved"),
        "in_progress_stages": sum(1 for s in all_stages if s["computed_status"] == "in_progress"),
        "not_started_stages": sum(1 for s in all_stages if s["computed_status"] == "not_started"),
        "pending_review_stages": sum(1 for s in all_stages if s["computed_status"] in ("pending_review", "submitted")),
        "rejected_stages": sum(1 for s in all_stages if s["computed_status"] == "rejected"),
    }

================================================================================
PERFORMANCE — ALREADY OPTIMIZED IN THIS BATCH
================================================================================
This batch addresses performance concerns upfront, NOT as follow-up:

**Two helpers, two use cases:**

1. `_compute_unit_scope_stage_statuses` — per-floor version.
   - Used by `/qc/floors/{floor_id}/run` (single floor — no batching needed).
   - 2 DB queries (units + unit-runs).

2. `_compute_unit_scope_stage_statuses_batch` — multi-floor version.
   - Used by `/qc/floors/batch-status` (dashboard view).
   - 2 DB queries TOTAL regardless of N floors.
   - Fetches all units across all floors in one `{"floor_id": {"$in": ...}}` query.
   - Fetches all unit-runs in one `{"unit_id": {"$in": ...}}` query.
   - Aggregates per-floor in memory via `_aggregate_unit_stages_in_memory`.

3. **Existing qc_items query also batched** — the original endpoint code
   queried qc_items inside the per-floor loop (N queries). This batch
   pre-fetches all qc_items for all floor-scope runs in 1 `$in` query.

**Net query count:**
- BEFORE this batch: ~40 queries for 20-floor project (floor-scope runs + qc_items per floor in loop).
- AFTER this batch: 4 queries total — runs + items + units + unit-runs.
- ProjectDashboard / ProjectControlPage / MapView calling batch-status
  in parallel: ~12 queries total instead of ~120 (assuming 3 callers).

**Why this matters:** Dashboard load on mobile 4G can already feel
slow. Adding N queries per call would make it worse. Batching is
preventive, not optional — same approach as the existing pattern
in BuildingDefectsPage (batch grid loaders).

If we observe further slowness, future optimization:
- Single Redis cache for batch-status with 60s TTL keyed by project_id
- Move to MongoDB aggregation pipeline for stage-level rollup
Both out of scope; the current 4-query design is sufficient for
typical project sizes (up to ~50 floors × ~10 units each = 500 units).
