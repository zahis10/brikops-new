ספרינט UX/NX למסך QC Stage — מקצועי, נוח, נגיש, ומזמין שימוש
=============================================================

שם הספרינט:
QC Stage Experience Sprint (V1.5) — Clarity, Accessibility, Flow

מטרת על
--------
להפוך את חוויית העבודה במסך שלב QC לחוויה:
- ברורה (מה חסר / מה נדרש / מה הצעד הבא)
- קלה ומהירה לשטח (פחות חיפוש, פחות ניסוי וטעייה)
- נגישה ונעימה לשימוש (גם במובייל, גם תחת לחץ)
- שקופה ניהולית (מי צילם / מי סימן / מי אישר / מתי)
- עם discoverability טוב להרשאות (איך PM מגדיר מאשרים)

עקרונות UX מחייבים (Guiding Principles)
----------------------------------------
1) Mobile-first אמיתי  
   רוב השימוש הוא מהשטח. כל החלטת UI נבדקת קודם במובייל.

2) Action-first  
   המשתמש צריך להבין מיד:
   - מה חסר
   - למה זה חסר
   - איך מתקנים
   - איך מתקדמים

3) Progressive disclosure  
   לא להעמיס. מידע חשוב תמיד גלוי, מידע משני נפתח לפי צורך.

4) Consistency  
   אין סתירות בטקסטים/חוקים (למשל "הערה חובה" מול "אופציונלי").

5) Role clarity  
   המשתמש צריך להבין:
   - האם מותר לו לאשר
   - אם לא, למה לא
   - מי כן
   - איפה PM מנהל מאשרים

6) Accessible by default  
   ניגודיות, כפתורים גדולים, states ברורים, feedback inline + toast.

היקף הספרינט (Scope)
--------------------
פוקוס: מסך Stage Detail + נקודת כניסה לניהול מאשרי QC + תצוגות היסטוריה/אחריות.

לא נכנסים בספרינט זה:
- פיצ'רים עסקיים חדשים כבדים
- שינוי מודל הרשאות מהותי
- אנליטיקה תפעולית מתקדמת (זמני ביצוע קומות/שלבים)
- WhatsApp/SMS/Email flows

מה נבנה בפועל (Deliverables)
----------------------------

A) Header/Top Summary ברור ומכוון פעולה
---------------------------------------
להוסיף אזור עליון קבוע וברור במסך Stage Detail עם:
1. סטטוס השלב (Draft / Ready / Pending Review / Approved / Rejected / Reopened)
2. "מה חסר כדי לשלוח" (Summary actionable)
   - חסרות תמונות: X
   - חסרות הערות חובה: Y
   - סעיפים שלא סומנו: Z
3. CTA ראשי:
   - "קח אותי לסעיף הראשון שחסר"
4. CTA משני:
   - "הצג רק סעיפים חסרים"

דרישות UX:
- הסיכום תמיד גלוי (sticky חלקי/קבוע בראש המסך במובייל)
- הטקסט חייב להיות חד וברור, לא כללי ("לא ניתן לשלוח" בלבד לא מספיק)

B) דרישת תמונה — להפוך לחוויה ברורה ולא מעיקה
----------------------------------------------
בעיה קיימת:
- המשתמש לא תמיד מבין מתי תמונה חובה ולמה.
- חסימה קיימת אבל לא אינטואיטיבית.

פתרון UX:
1. בכל סעיף עם photo_required:
   - תגית בולטת עם אייקון מצלמה: "חובה תמונה"
   - טקסט עזרה קצר: "נדרש לצורך אישור השלב"
2. אם המשתמש מנסה לסמן תקין/לא תקין בלי תמונה:
   - חסימה (כבר קיימת לוגית) + Inline error מתחת לסעיף
   - כפתור ברור: "הוסף תמונה"
   - toast משלים (לא במקום inline)
3. לאחר העלאת תמונה:
   - state ברור: "תמונה הועלתה"
   - thumbnail ברור + metadata (ראה סעיף C)

מטרת UX:
להעביר את המשתמש ממצב "למה זה לא עובד?" למצב "אה, חסרה לי תמונה — הנה כפתור".

C) Chain of Responsibility — מי עשה מה ומתי (ברור וקריא)
---------------------------------------------------------
להציג בצורה בולטת וקריאה (לא טקסט קטן/חתוך):
1. לכל תמונה:
   - "צולם ע״י {שם}" / "צולם • {שעה/תאריך}"
2. לכל שינוי סטטוס סעיף:
   - "סומן ע״י {שם} • {שעה}"
3. לכל שלב (בכרטיס Audit Summary):
   - נשלח ע"י
   - אושר ע"י
   - נדחה ע"י
   - נפתח מחדש ע"י
   - תאריך/שעה לכל פעולה

כללי תצוגה:
- שמות לא נחתכים בלי ellipsis מסודר
- timestamp בפורמט אחיד
- fallback תקין:
  - null (מוסתר לפי הרשאות) -> לא להציג שם בכלל
  - "" (legacy) -> "משתמש לא ידוע"

הרשאות תצוגה:
- PM: רואה מלא
- Approvers (בסcope שלהם): רואים מלא בשלבים הרלוונטיים
- משתמש רגיל: תצוגה מצומצמת (לפי backend filtering שכבר קיים)

D) Timeline / History — תמיד זמין, תמיד ברור
---------------------------------------------
להפוך את היסטוריית הפעולות לרכיב אמין ונגיש:
1. כרטיס קבוע: "היסטוריית פעולות"
2. מצבים ברורים:
   - Loading (skeleton)
   - Empty state: "אין היסטוריה עדיין"
   - Error state: "לא הצלחנו לטעון היסטוריה" + "נסה שוב"
3. הצגת אירועים בצורה כרונולוגית:
   - submit
   - reject (+ reason)
   - approve
   - reopen (+ reason)
   - (אופציונלי) item updates / photo uploads אם יש ערך גבוה

מטרת UX:
שקיפות מלאה בלי שהמשתמש יצטרך "לנחש" אם יש באג או פשוט אין היסטוריה.

E) שיפור מבנה כרטיסי הסעיפים (Item Cards) — פחות עומס, יותר היררכיה
---------------------------------------------------------------------
מבנה מוצע לכל סעיף:
1. כותרת סעיף (גדול וברור)
2. תגיות חובה (חובה תמונה / הערה חובה במקרה fail)
3. פעולות ראשיות (תקין / לא תקין)
4. אזור מדיה (thumbnail + העלאה)
5. שורת מטא-דאטה (מי צילם / מי סימן / מתי)
6. שדה הערה (עם label עקבי לפי סטטוס)
7. Inline errors (אם קיימים)

שיפורים ספציפיים:
- להפחית טקסט אפור קטן שמרגיש "רעש"
- להדגיש primary action ברור
- spacing נעים לאצבע (touch targets)
- states ברורים: default / filled / error / success / disabled

F) עקביות חוקים וטקסטים (Notes V1 Rule) — בלי סתירות
-----------------------------------------------------
חוק V1 המאושר:
- הערה חובה רק כאשר מסומן "לא תקין"
- כאשר מסומן "תקין" — הערה אופציונלית

מה ליישם ב-UI:
1. labels
2. placeholders
3. helper text
4. validation messages
5. disabled explanation summary

דוגמאות copy:
- Fail: "חובה לציין הערה כשמסמנים לא תקין"
- Pass: "אפשר להוסיף הערה (אופציונלי)"

אין מצב שבו במקטע אחד כתוב "חובה" ובשני "אופציונלי" לאותו מצב.

G) שליחה לאישור — UX שמסביר מה חסר ומוביל לפתרון
-------------------------------------------------
לפני/בזמן disabled submit:
1. תיבת הסבר ברורה (כבר התחיל — לשפר):
   - "לא ניתן לשלוח לאישור"
   - "חסרות תמונות ב-X סעיפים"
   - "חסרות הערות ב-Y סעיפים"
   - "יש Z סעיפים שטרם סומנו"
2. כפתורי ניווט מהירים:
   - "קח אותי לסעיף הראשון שחסר"
   - "הצג רק חסרים"
3. highlight זמני לסעיף בעייתי (2–3 שניות)

מטרת UX:
במקום תחושת חסימה — תחושת הכוונה.

H) Discoverability להרשאות אישור (PM + צוות)
----------------------------------------------
בעיה:
לא ברור בכלל איך PM נותן הרשאת אישור למישהו מהצוות.

פתרון UX:
1. נקודת כניסה ברורה במסך Project Control / Project Settings:
   - כרטיס/שורה: "מאשרי QC"
   - תת-טקסט: "PM מאשר כברירת מחדל. ניתן להוסיף מאשרים נוספים לפי שלבים"
2. במסך Stage כשהסטטוס pending_review:
   - אם המשתמש PM: CTA קטן "ניהול מאשרי QC"
   - אם המשתמש אינו מורשה:
     "אין לך הרשאת אישור לשלב זה"
     + "פנה למנהל הפרויקט להוספת הרשאה"
3. במסך "מאשרי QC":
   - כותרת ברורה
   - הסבר קצר על ברירת מחדל (PM)
   - רשימת מאשרים קיימים + scope (כל השלבים / שלבים ספציפיים)
   - CTA להוספת מאשר

מטרת UX:
למנוע מצב שהפיצ'ר קיים אבל "לא נמצא".

I) Microcopy / Tone (עברית לשטח)
--------------------------------
הטקסטים צריכים להיות:
- קצרים
- ישירים
- לא טכניים מדי
- מסבירים מה לעשות, לא רק מה קרה

דוגמאות סגנון:
- במקום: "validation failed"
  להשתמש: "כדי להמשיך צריך להשלים תמונה בסעיף זה"
- במקום: "permission denied"
  להשתמש: "אין לך הרשאת אישור לשלב זה"

J) נגישות (Accessibility) — חובה
--------------------------------
1. כפתורים/טאבים:
   - שטח לחיצה מינימלי נוח (mobile touch target)
2. ניגודיות:
   - טקסט אפור לא חלש מדי
3. States:
   - לא להסתמך רק על צבע (להוסיף אייקון/טקסט)
4. פוקוס/מקלדת:
   - states ברורים גם בדסקטופ
5. Screen reader readiness (אם ריאלי):
   - labels ברורים לכפתורי העלאת תמונה/סימון תקין/לא תקין

K) ביצועים נתפסים (Perceived Performance)
-----------------------------------------
גם בלי שינוי backend כבד:
1. skeletons במקום "ריק" בזמן טעינה
2. lazy expand לפרטים משניים
3. polling לא אגרסיבי במסך stage
4. fallback מהיר אם timeline/metadata נכשל (לא לשבור את המסך)

Acceptance Criteria (חובה לספרינט)
-----------------------------------
1. משתמש מבין תוך 2–3 שניות למה אי אפשר לשלוח לאישור
2. בכל סעיף עם תמונה חובה — ברור לפני הלחיצה שצריך תמונה
3. בכל סעיף עם תמונה — ניתן לראות מי צילם ומתי (אם מותר לפי הרשאה)
4. בכל סעיף שסומן — ניתן לראות מי סימן ומתי (אם מותר לפי הרשאה)
5. היסטוריית פעולות תמיד מציגה אחד מהמצבים: loading / empty / error / data
6. אין סתירות בטקסטי הערות חובה/אופציונלי (לפי חוק V1)
7. PM מוצא "מאשרי QC" בלי לחפש עמוק
8. משתמש לא מורשה לאישור מקבל הודעה ברורה + הנחיה מה לעשות
9. מסך stage מרגיש שמיש במובייל גם עם 10+ סעיפים (גלילה, כפתורים, קריאות)

מדדי הצלחה (Success Metrics) — V1 פרקטי
----------------------------------------
לבדוק ידנית / בסיסית אחרי פריסה:
1. ירידה בכמות "למה לא נותן לי לשלוח?" (פידבק משתמשים)
2. זמן ממוצע להשלמת שלב (תחושתית + בדיקות ידניות) מתקצר
3. פחות טעויות בסימון ללא תמונה/הערה
4. PM מצליח להגדיר מאשר נוסף בלי עזרה חיצונית

תיעדוף ביצוע (Execution Order)
-------------------------------
שלב 1 (ערך מיידי גבוה):
- A) Top Summary (מה חסר)
- B) דרישת תמונה UX
- F) עקביות הערות חובה/אופציונלי
- G) Submit guidance

שלב 2:
- C) מי צילם / מי סימן (תצוגה קריאה)
- D) Timeline states (loading/empty/error/retry)

שלב 3:
- E) Refactor כרטיסי סעיפים (visual hierarchy)
- H) Discoverability של "מאשרי QC"

Definition of Done (DoD)
------------------------
הספרינט ייחשב "גמור" רק אם:
1. הכל עובד ב-Preview וב-Production
2. נבדק במובייל (iPhone) על flow מלא:
   סימון -> צילום -> הערה (אם צריך) -> שמירה -> שליחה לאישור -> approve/reject/reopen
3. אין שגיאות console קריטיות במסך stage
4. כל הטקסטים בעברית עקביים וברורים
5. יש screenshots/recording קצר לפני/אחרי (Before/After UX)

מה אני רוצה לקבל בסיום הספרינט
-------------------------------
1. סיכום שינויים מסודר (Backend/Frontend)
2. רשימת מסכים/רכיבים ששופרו
3. סרטון/צילומי מסך של:
   - חסימת תמונה + inline guidance
   - summary "מה חסר"
   - timeline states
   - תצוגת "מי צילם/מי סימן"
   - כניסה למסך "מאשרי QC"
4. רשימת open issues (אם נשאר משהו קטן)
5. המלצה לספרינט הבא (רק אחרי שה-UX הזה יושב טוב)

הערה חשובה
-----------
המטרה בספרינט הזה היא לא "להוסיף עוד אפשרויות", אלא לגרום למשתמש להרגיש:
"זה ברור, זה קל, זה עובד איתי — לא נגדי."
