היי,
יש לנו כמה תיקונים/שיפורים אחרי ה-Publish של M8. חשוב: להתחיל שלב-שלב, עם עדיפות ברורה. 
הנושא הכי קריטי הוא חשש שמידע/משתמשים “נעלמים” אחרי עדכון — זה P0 ואסור שזה יקרה בשום מצב.

============================================================
P0 — Data Integrity: משתמשים/מידע “נעלמים” אחרי עדכון (חובה לפני כל דבר אחר)
רקע:
- יצרתי משתמשים (לדוגמה “צחי אינסטלטור” עם מספר טלפון) ועכשיו הוא לא מופיע.
- יש חשש שאחרי publish/דיפלוי הנתונים מתאפסים או שמתחברים ל-DB אחר.
- אסור שלעולם משתמש יימחק או מידע מהפרויקט יימחק. אם זה קורה — אין זכות קיום לאפליקציה.

יעד:
להוכיח בצורה חד-משמעית שאין מחיקות/איפוסים, ושפרודקשן תמיד מחובר לאותו DB.

שלבים לביצוע (P0):
1) Verify DB Identity בפרודקשן
   - להדפיס בלוג בעת עלייה (וגם אופציונלית במסך super_admin “System” קטן):
     - dbName / cluster/host (חלקי), usersCount, projectsCount, env name
   - לוודא שפרודקשן מתחבר ל-MONGODB_URI הנכון (לא dev / לא staging / לא local).
   - לוודא שאין fallback URI שגורם להתחבר ל-db זמני.

2) Fail-fast בפרודקשן
   - אם NODE_ENV=production וחסר MONGODB_URI (או לא תקין) -> האפליקציה חייבת להיכשל בעלייה (לא לעלות עם DB ברירת מחדל).
   - אותו דבר ל-secrets קריטיים אחרים (JWT, encryption keys וכו') אם רלוונטי.

3) לנטרל כל Seed/Reset אוטומטי
   - לחפש בכל הקוד: seed, reset, initDB, bootstrap, ensureAdmin, createDefault*
   - לוודא ש-seed לא רץ בפרודקשן.
   - seed מותר רק אם RUN_SEED=true ידני + רק בסביבת dev (וגם אז לא למחוק כלום).

4) לא לאפשר Hard Delete ל-Users/Projects
   - לוודא שאין deleteOne/deleteMany שמוחקים users/projects/memberships.
   - אם יש “מחיקה” — להפוך ל-soft delete:
     - deletedAt / disabledAt, ולהסתיר ב-UI כברירת מחדל.
   - כל שינוי סטטוס/מחיקה רכה חייבים Audit.

5) בדיקת אמת (Proof)
   - לאתר את המשתמש “צחי אינסטלטור” לפי טלפון/שם ישירות ב-DB (Mongo query) ולהראות:
     - האם קיים? האם סתם לא משויך לפרויקט? האם disabledAt?
   - לצרף צילום/לוג קצר ל-PROOF_PACKAGE.md:
     - db identity + counts
     - query proof על user קיים/לא קיים + הסבר למה

רק אחרי ש-P0 נסגר (יש הוכחה שאין איפוס/חיבור ל-db אחר/seed) — עוברים ל-P1/P2.

============================================================
P1 — Join with Code (Option A מוסכם): הפרדה בין Project Code לבין Join Code
רקע:
- בפרויקט “מגדלי ים” קוד פרויקט SEA-001.
- במסך “הצטרפות עם קוד” הזנתי SEA-001 וקיבלתי: “קוד הצטרפות לא נמצא”.
- החלטה: לא משתמשים ב-SEA-001 להצטרפות. SEA-001 הוא קוד פרויקט לתצוגה/זיהוי בלבד.

דרישה (Option A):
1) להוסיף לכל פרויקט Join Code נפרד (אקראי) למשל 6–8 תווים:
   - שדה חדש: joinCode
   - ייחודי לכל פרויקט
   - ניתן לרגנרציה ע"י PM/Admin (עם אישור)
2) UI/Copy:
   - לשנות את הטקסטים במסך הצטרפות כך שיהיה ברור שזה “קוד הצטרפות” (ולא “קוד פרויקט”).
   - הודעת שגיאה: “קוד הצטרפות לא נמצא” נשארת, אבל להוסיף עזרה קצרה: “קוד ההצטרפות מתקבל ממנהל הפרויקט”.
3) Backend:
   - Endpoint שמקבל joinCode בלבד ומחזיר פרויקט/בקשת הצטרפות בהתאם ל-flow שקיים אצלנו.
   - לא לבצע fallback ל-projectCode.
4) ניהול joinCode:
   - בתוך פרויקט (Settings או Team) להציג:
     - “קוד הצטרפות” + כפתור העתקה
     - “רגנרציה לקוד” (PM/Admin) עם אישור כפול
   - Audit events:
     - PROJECT_JOIN_CODE_REGENERATED (actor, projectId, old/new, timestamp, reason אופציונלי)
5) Acceptance:
   - ניסיון להצטרף עם SEA-001 נכשל (כי זה לא joinCode) — וזה תקין לפי ההחלטה.
   - הצטרפות עם joinCode תקף עובדת.
   - UI ברור ובלתי מבלבל (Project Code ≠ Join Code).

============================================================
P1 — Team tab: לחיצה על חבר צוות → פרופיל
דרישה:
- בטאב “צוות” בתוך פרויקט: כל שורה של חבר צוות תהיה clickable.
- לחיצה תנווט לדף פרופיל המשתמש (קיים או חדש) לפי userId.
- הרשאות:
  - חברי פרויקט: צפייה בסיסית.
  - admin/pm/super_admin: צפייה מורחבת + פעולות ניהול.

============================================================
P2 — פרופיל משתמש: לאדמין לערוך הכל + תיקון RTL טלפון במסך הזה
רקע:
- בתוך פרופיל משתמש אני מצפה לאפשר עריכה מלאה כאדמין:
  - שם, אימייל, טלפון, הארכת רישיון/Trial (הקוד “בגדול כבר קיים”).
- בנוסף במסך הפרופיל יש עדיין הצגת טלפון הפוכה (972...+), כלומר RTL לא סגור שם.

דרישה:
- להוסיף מצב Edit לפרופיל משתמש (admin/super_admin בלבד):
  - name, email, phone (עם step-up אם צריך), license/trial extension
  - reason חובה לשינויים רגישים
- כל שינוי: Audit עם old/new/actor/reason
- לתקן RTL במסך הזה עם <bdi dir="ltr"> לכל מספר טלפון.

============================================================
P2 — Projects: Archive/Pause + (אופציונלי) Soft Delete
רקע:
- מצופה שתהיה אפשרות להשהות/לארכב פרויקט לא פעיל.
- מחיקה אמיתית לא רצויה; אם חייבים — רק soft delete.

דרישה:
- סטטוסים: Active / Archived
- Archived לא מופיע ברשימה כברירת מחדל, אבל יש פילטר “הצג ארכיון”.
- אם מוסיפים “מחיקה”:
  - רק super_admin
  - רק soft delete (deletedAt)
  - אישור כפול + audit

============================================================
P2 — ברשימת הפרויקטים אין צורך ב"חברות קבלן" (להעביר פנימה)
דרישה:
- להוציא את הבלוק “חברות קבלן” מעמוד/רשימת פרויקטים ברמת מערכת.
- זה צריך להיות בתוך פרויקט עצמו (Project Settings / Contractors).

============================================================
סדר עבודה מחייב
1) P0 Data Integrity — לסגור + להוכיח (db identity, no seed/reset, no hard delete).
2) P1 Join Code (Option A) + Team click-to-profile.
3) P2 פרופיל משתמש עריכה מלאה + RTL fix שם + Archive projects + להזיז “חברות קבלן” פנימה.

בסוף כל שלב:
- לשלוח לי proof קצר: מה בוצע, screenshots/לוגים, ואיך אני בודק ידנית שזה עובד.
