איגוד מלא של כל מה שדיברנו עליו לאחרונה (Billing + Project Card + Admin Billing KPIs) + Guardrails “לא לשבור כלום”

מטרת העל
ליישר את כל חוויית החיוב כך שתהיה:
- שקופה (תוקף רישיון נכון, בלי “ניסיון” שגוי)
- שימושית (אפשר “חידוש מראש” מכל מקום רלוונטי)
- אמינה (אין self-extend; pending_review + אסמכתא + אישור SA בלבד)
- מדדים אדמיניים נכונים (MRR / משולם החודש, בלי מספרים לא קשורים)

========================================================
A) Project Dashboard — כרטיס “חיוב פרויקט” (2 תיקונים)
========================================================

A1) חידוש מראש מהכרטיס גם כשהמנוי פעיל
בעיה:
גם כשהרישיון פעיל, בעל הארגון רוצה לחדש עכשיו לשנה/חודש (חידוש מראש). חייב CTA נגיש גם במסך הפרויקט.

תיקון:
- להוסיף בכרטיס “חיוב פרויקט” CTA ברור: "חידוש מראש"
- הפעולה (מומלץ, מינימום שינוי):
  ניווט ל: /billing/org/<org_id>#renew
  (שימוש בסקשן החידוש שכבר קיים ב-org billing: חודשי/שנתי + preview + בקשת תשלום)
- הרשאות:
  - מי שיש לו can_manage_billing: רואה “חידוש מראש”
  - PM/role_restricted: רואה “בקש שדרוג מבעלים” (handoff) בלבד

Proof:
- צילום מסך: כרטיס חיוב פרויקט מציג “חידוש מראש” גם כשהמנוי פעיל
- לחיצה מובילה לסקשן חידוש עם preview ותאריך “הרישיון יהיה בתוקף עד”

A2) תיקון תג “ניסיון” שגוי כשה-paid_until בעתיד
בעיה:
מופיע תג כחול “ניסיון” למרות ש-paid_until עתידי.

תיקון:
- כל תצוגת “ניסיון/פעיל” חייבת להיות מונעת ONLY ע"י subscription_status time-based (snapshot),
  ולא לפי status legacy או שדות ישנים.
כלל:
- אם paid_until > עכשיו => להציג “פעיל” (לא ניסיון)
- ניסיון רק אם trial_end_at > עכשיו AND (paid_until ריק או <= עכשיו)

Proof:
- צילום מסך: paid_until עתידי → אין “ניסיון”, כן “פעיל”

========================================================
B) Billing Manual V1.1 — Trust-Safe Flow (לשמור + לוודא UI תקין בכל מקום)
========================================================

זרימה:
- Owner/billing_admin/org_admin:
  - “שלח בקשת תשלום”
  - “סימנתי ששילמתי” -> status=pending_review (ללא שינוי רישיון)
  - “העלה אסמכתא” (אופציונלי)
- Super Admin:
  - “אשר תשלום” -> רק אז paid_until מתקדם + request -> paid
  - “דחה בקשה” עם סיבה
  - צפייה/הורדת אסמכתא
נעילות:
- mark-paid שמאריך paid_until = super_admin בלבד (403 לכל השאר)
- PUT payment-config = super_admin בלבד

Proof:
- Owner: pending_review ללא שינוי paid_until
- SA: approve -> paid_until מתקדם
- Owner מקבל 403 על mark-paid
- receipt download עם presigned url

========================================================
C) Admin Billing Dashboard — מדד “הכנסה חודשית ₪450” לא תואם חשבוניות
========================================================

בעיה:
מוצג “הכנסה חודשית ₪450” למרות שיש חשבונית/חיובים אחרים (לדוגמה חשבונית ₪6,800 בסטטוס “הונפק”).
נדרש להגדיר/לפצל מדדים כדי למנוע בלבול.

תיקון מוצרי/UX:
להציג שני מדדים נפרדים וברורים (או לשנות כותרת בהתאם למה שמחושב):

C1) “צפי הכנסה חודשית (MRR)”
- חישוב: סכום total_monthly של כל הארגונים הפעילים (effective_access=full_access) שאינם מושעים.
- לא תלוי בסטטוס חשבוניות.

C2) “הכנסה משולמת החודש”
- חישוב: סכום invoices בסטטוס paid עם paid_at בתוך החודש הנוכחי.
- חשבונית “הונפקה” לא נכנסת למדד זה עד שתשולם.

אם נשארים עם מדד אחד בלבד:
- חובה לשנות כותרת שתשקף מה הוא באמת (MRR או משולם החודש), בלי “הכנסה חודשית” עמומה.

Proof:
- JSON/לוג שמראה:
  - אילו שדות נסכמים ל-MRR
  - אילו חשבוניות נספרות ל“משולם החודש”
- צילום מסך Admin Billing אחרי תיקון שמראה שני מדדים (או כותרת מדויקת)

========================================================
E) Guardrails מחייבים (No-regression — לא לגרוס/לא לשבור קיים)
========================================================
1) Incremental בלבד
- לא משנים שמות endpoints קיימים
- לא משנים shape של responses קיימים
- לא מוחקים שדות/אוספים קיימים
- רק מוסיפים שדות optional / מסכים / CTA / לוגיקה קטנה

2) Source of truth נשאר כמו היום
- DB נשאר מקור אמת: paid_until / subscription_status / _resolve_access
- V1.1 נשאר Trust-safe: pending_review + SA approve בלבד
- mark-paid שמאריך רישיון = super_admin בלבד

3) UI changes מינימליים
- בכרטיס חיוב פרויקט: רק להוסיף CTA שמנווט ל-/billing/org/<org_id>#renew (לא לבנות מסך חדש)
- לתקן תג “ניסיון” רק ע"י שימוש ב-subscription_status
- Admin billing metrics: להוסיף עוד כרטיס מדד או לשנות כותרת — בלי לשבור דוחות/רשימות

4) אין סגירה בלי הוכחות
- אם משהו נשבר: חובה root cause + fix + הוכחה (UI+API)
- חובה E2E smoke קצר: login → billing → payment request → pending_review → SA approve

בבקשה לבצע לפי הסדר: A (Project card) + C (Admin metrics) + לוודא B לא נשבר.
