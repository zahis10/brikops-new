M9 Billing v1 — Phase 2 (High/Sensitive) | Pricing XL + Hebrew UX + Billing Workflow/Handoff

Objective
Implement Billing v1 Phase 2 on top of the completed Phase 1 foundation:
1) add XL pricing support (plan + unit tier),
2) fix Hebrew billing status labels in UI (e.g. "trialing" -> "ניסיון"),
3) implement billing workflow/handoff (PM starts project, org billing role completes billing setup),
4) add authorized write flows for project billing (plan selection + contracted units),
5) keep org-level payer model and strict org/project scoping.

Context / Non-negotiables
- Billing payer is the Organization (org), not project members.
- Each organization may have multiple projects.
- Each project belongs to one organization context (org unifies all companies/teams in that project).
- Company != Organization (do not merge/duplicate concepts).
- Super Admin retains full control across all orgs/projects.
- Viewer remains strictly read-only (no actions).
- Phase 2 is still NO payment gateway / NO card charging yet.
- Contracted units drive billing. Observed units are monitoring-only (warning/insight), not automatic billing changes.

Pricing Model (replace/extend current seed + UI display)
A) Plans (product level)
- בסיסי (Basic): project_fee_monthly = 1200
- מקצועי (Pro): project_fee_monthly = 2000
- XL: project_fee_monthly = 3500

B) Unit tiers (flat monthly fee by contracted_units, NOT per-unit)
- tier_s: עד 50 יחידות -> monthly_fee = 900
- tier_m: 51–200 יחידות -> monthly_fee = 2400
- tier_l: 201–500 יחידות -> monthly_fee = 4800
- tier_xl: 501+ יחידות -> monthly_fee = 8500

Formula
- project monthly_total = project_fee_snapshot + tier_fee_snapshot
- org subscription total_monthly = sum(project_billing.monthly_total for active/paused? see rule below)

Rule for org total aggregation (implement explicitly)
- Include project_billing rows with status = active only in total_monthly (default)
- Exclude paused/archived from billed total
- Still display paused/archived in org UI with status badge and monthly_total snapshot for reference

T203 — Billing workflow / handoff model (Org pays, PM can start)
Goal:
PM can start project/trial and see billing readiness, but billing setup/activation is completed by org billing authority.

Implement minimal Phase 2 workflow (no payment gateway):
Project billing setup state (use existing fields + minimal additions; avoid overengineering)
- project_billing.status remains: active | paused | archived
- Add setup/workflow state field (new) on project_billing OR nested object:
  - setup_state: trial | pending_handoff | pending_billing_setup | ready | active
  - recommended minimal flow:
    - trial (default, no plan selected / no contracted_units)
    - pending_handoff (PM requested org billing action)
    - pending_billing_setup (org billing role acknowledged but not completed)
    - ready (plan + contracted_units set, awaiting manual activation)
    - active (subscription/org access active and project_billing active)
- If you prefer fewer states, propose a lean equivalent BEFORE implementing.

Handoff actions (manual in-app, no external payment)
- PM (project_manager) can:
  - view project billing page/card
  - submit handoff request for org billing setup (action only on same project/org)
  - optionally attach billing contact note/email (non-sensitive text)
- Org billing roles (owner/org_admin/billing_admin) can:
  - assign/update plan + contracted_units
  - set project_billing active/paused
  - mark handoff acknowledged / complete billing setup
- Super Admin can do all actions
- management_team:
  - read project billing only (no writes)
- contractor/viewer:
  - no billing access (existing behavior preserved)

Audit events required for workflow actions
- billing_handoff_requested
- billing_handoff_acknowledged (if implemented)
- project_billing_setup_completed (or equivalent)
- project_billing_updated (already exists; continue explicit before/after)

Acceptance:
- PM can request handoff but cannot set org billing plan/contracted_units unless also org billing role/owner
- owner/org_admin/billing_admin can complete setup
- all writes produce audit events with actor + org_id + project_id + before/after payload summary

T204 — Authorized project billing write endpoints (Phase 2)
Add/extend endpoints (behind BILLING_V1_ENABLED):
1) Project billing update endpoint (org billing role only)
- PATCH /api/billing/project/{project_id}
- Allowed: super_admin, org owner, org_admin, billing_admin
- Blocked: PM (unless also org billing role), management_team, contractor, viewer
Payload (minimal):
- plan_id (optional update)
- contracted_units (optional update)
- status (active/paused/archived) [optional]
- setup_state (optional, validated transitions)
Behavior:
- Validate project exists and has org_id
- Validate plan exists/active (if plan_id provided)
- Recompute tier_code + snapshots + monthly_total when plan_id or contracted_units changes
- Recompute subscription.total_monthly for org after change
- Write audit events (project_billing_created/project_billing_updated/contracted_units_changed as appropriate)
- Idempotent no-op payload should not create noisy audit event (or create one only if you already standardize "no-op" audits)

2) Handoff request endpoint (PM + billing roles + SA)
- POST /api/billing/project/{project_id}/handoff-request
- Allowed: super_admin, project_manager, org owner, org_admin, billing_admin
- Blocked: management_team/contractor/viewer (unless you justify mgmt read/write change explicitly)
Payload:
- note (optional short text)
- target_user_id OR billing_email (optional, at least one if provided in UI)
Behavior:
- Set setup_state to pending_handoff (or pending_billing_setup if acknowledged by billing role)
- Create audit event billing_handoff_requested
- Must remain project/org scoped

3) Optional ack/complete endpoint (recommended for clearer workflow)
- POST /api/billing/project/{project_id}/handoff-ack
- POST /api/billing/project/{project_id}/setup-complete
(If you prefer to keep this inside PATCH setup_state transitions, okay — but enforce transition validation.)

Acceptance:
- Strict org/project scoping on every endpoint
- 400 on project with org_id=None
- 403 on unauthorized role
- PM cannot write org billing data
- subscription.total_monthly updates after project billing changes
