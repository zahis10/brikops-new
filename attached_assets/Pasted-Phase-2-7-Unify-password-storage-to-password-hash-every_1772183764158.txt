Phase 2.7 — Unify password storage to password_hash everywhere (Auth hardening)

Goal
We have a pre-existing inconsistency: some onboarding flows store the password under `password` instead of `password_hash`.
We must unify so ALL flows store ONLY `password_hash`, and legacy users with `password` remain able to login (with safe backfill + cleanup).

Scope (do not change unrelated flows)
- Fix ONLY password storage across: invite-accept + join-by-code (+ any other user-creation flows that currently write `password`)
- Do NOT change OTP flows, contractor flows, billing, RBAC, UI behavior (except where needed to pass the password field correctly).

Implementation requirements
1) Write path fix (primary)
- Find every place we set a password during onboarding (invite accept, join-by-code, any other creation/update path).
- Replace any writes to `password` with:
  - bcrypt hash -> `password_hash`
- Ensure new/updated users have:
  - `password_hash` present
  - `password` field NOT present

2) Backward compatibility (secondary, safe migration)
- In the email/password login endpoint:
  - If `password_hash` is missing AND `password` exists:
    - If `password` looks like a bcrypt hash (starts with $2a$/$2b$/$2y$): treat it as the hash
    - Else: treat it as plaintext and hash it
    - Set `password_hash` accordingly
    - Unset/remove `password` from the user doc
    - Log an audit event: action="auth_password_hash_backfilled" with {source:"login", had_plaintext: bool}
- This ensures existing accounts created via old flows can login and get cleaned automatically.

3) Optional (recommended) one-off cleanup script
- Provide a script backend/scripts/backfill_password_hash.py that scans for users with `password` but missing `password_hash`,
  backfills + unsets `password`.
- Read-only dry-run mode first, then apply mode.
- This is safer than startup writes.

Proof package required (no closure without proof)
A) Exact files changed + summary of changes
B) New user via invite flow:
   - created user doc contains `password_hash` only
   - login via email/password succeeds
C) New user via join-by-code flow:
   - created user doc contains `password_hash` only
   - login via email/password succeeds
D) Legacy user with only `password` field:
   - login succeeds
   - DB updated: `password_hash` created and `password` removed
   - audit event recorded
E) Negative tests:
   - wrong password still fails
   - no changes to OTP login
F) Regression:
   - demo logins still work
   - no RBAC/billing regressions

Release/quality gates for this change (auth-sensitive)
1) No closure without proof (root cause + fix + regression + API/UI proof)
2) Guardrails: strict scoping, no unexpected role/org/project access changes
3) No “user confusion” closure — show evidence
4) Release gate: smoke tests + negative tests before publish
