מאשר את התוכנית המעודכנת — היא מדויקת, מאובטחת יחסית, ושומרת על העיקרון החשוב: reset-password לא תלוי SMTP, ורק ה-Step-Up הוא השער.

כמה דרישות/שיפורים קטנים כדי שלא ניפול על “Break-glass” מסוכן מדי:

1) Redaction של smtp_error
- בלוגים וב-audit לשמור רק:
  smtp_error_class + smtp_error_code (אם קיים) + message קצר (ללא host/user/headers)
- לא להדפיס אימייל מלא, לא reply-to, לא from, לא env names ולא stack trace שמדליף פרטים.

2) Rate limit צריך להיות per-actor + per-target
- לא רק לפי IP.
- לדוגמה: key = actor_user_id + target_user_id (וגם IP כעוגן משני).
- עדיין 3 ניסיונות / 10 דקות, אבל חשוב שזה לא יאפשר brute force או “spam לוגים” על משתמשים אחרים.

3) Request ID אחיד מקצה לקצה
- לוודא שיש middleware שמייצר request_id אם לא הגיע (X-Request-ID), ומחזיר אותו גם ב-response header + ב-JSON.
- בלוג [STEPUP-FALLBACK] וב-audit לשמור את אותו request_id בדיוק.

4) Challenge חייב להישמר גם בכשל SMTP (כבר כתבתם) + שדה מקור
- להוסיף לשמירת ה-challenge שדה כמו delivery_channel = "email" / "log_fallback"
- כדי שבעתיד נוכל לנתח שימוש ב-break-glass ולהתריע.

5) התנהגות שגיאה כשפלאג כבוי
- כשה-SMTP נופל ו-STEPUP_LOG_FALLBACK_ENABLED=false:
  להחזיר status מתאים (מומלץ 503 Service Unavailable) עם error_code ברור:
  STEPUP_UNAVAILABLE_SMTP
- ה-frontend יציג “Step-Up לא זמין — SMTP” (כבר כתבתם) ולא יראה את זה ככשל reset-password.

6) הרשאות: fallback רק super_admin (כבר כתבתם) + Audit חובה
- לוודא שה-audit נכתב לפני/אחרי כל ניסיון (גם אם נכשל) כדי שיהיה trace מלא.
- audit event:
  stepup_fallback_log_used (on challenge create)
  + stepup_verify_failed / stepup_verify_success (כבר אמור להיות)
  + admin_password_reset (אחרי reset) / admin_password_reset_direct (T003)

7) T003 (אופרטיבי) — הערת בטיחות
- הסיסמה הזמנית חזקה: OK.
- לבקש מהמשתמש להחליף מיד אחרי התחברות (גם אם “חובה החלפה” מחוץ לטווח, לפחות נרשום ב-note וב-replit.md).

עם התוספות האלה — אפשר להתקדם בדיוק לפי T001–T004 כמו שרשמתם, כולל כל ה-Proof ו-Negative tests.
