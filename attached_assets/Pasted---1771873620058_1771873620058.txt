מצאתי באג קריטי בלוגיקה של דחיית סעיף בביקורת (מגובה בבדיקה בפועל):

המצב כרגע (שגוי):
- דחיתי סעיף ברמת item
- מופיע badge אדום "נדחה בביקורת" ✅
אבל:
1) הסעיף עדיין נשאר מסומן "תקין" (ירוק) ❌
2) הצלחתי לשלוח את השלב שוב לאישור למרות שיש סעיף דחוי ❌❌

זה חייב להיחסם.

מה צריך להיות (התנהגות נכונה):
A) דחיית סעיף = משנה את מצב הסעיף בפועל
- ברגע שמאשר דוחה סעיף, הסעיף צריך להפוך אוטומטית ל"לא תקין" (או effective_status = rejected/needs_fix שמוצג כלא תקין)
- אסור שיישאר ירוק "תקין" עם badge אדום בלבד (מבלבל למבצע)

B) חסימת שליחה לאישור מחדש כל עוד יש סעיף דחוי
- אם קיים item עם reviewer_rejection פעיל -> submit_for_review חייב להיחסם ב-backend (422)
- ה-frontend גם צריך לשקף זאת (disable + הודעה ברורה), אבל ההגנה הקריטית חייבת להיות בשרת

C) “תוקן” = רק אחרי שינוי בפועל ע"י מבצע
- reviewer_rejection צריך להישאר פעיל עד שהמבצע מתקן את הסעיף (סטטוס/תמונה/הערה)
- רק אז לנקות reviewer_rejection
- ורק אחרי שאין יותר סעיפים דחויים פעילים -> אפשר שוב לשלוח לאישור

יישור מומלץ (Backend first):
1. ב-endpoint של reject item:
   - לשמור reviewer_rejection (כבר קיים)
   - וגם לכפות item.status = "fail" (או שדה effective state ברור)
   - לעדכן audit event עם item_id + reason + actor
   - stage -> reopened (כמו עכשיו)

2. ב-submit_for_review validation:
   - אם יש items עם reviewer_rejection פעיל => להחזיר 422
   - הודעה ברורה:
     "לא ניתן לשלוח לאישור: קיימים סעיפים שנדחו בביקורת וטרם תוקנו"
   - להחזיר גם item_ids / count כדי שה-UI יוכל לקפוץ אליהם

3. ב-frontend (StageDetail):
   - סעיף עם reviewer_rejection יוצג כ"לא תקין / נדרש תיקון" (ויזואלית ברור)
   - submit button disabled אם יש rejected items
   - בתיבת החוסרים להוסיף שורה:
     "סעיפים שנדחו בביקורת: X"

Acceptance (חובה לפני סגירה):
1) סעיף מסומן תקין -> מאשר דוחה עם סיבה -> הסעיף מוצג מיידית כלא תקין
2) כל עוד הסעיף לא תוקן -> אי אפשר submit (UI + backend 422)
3) אחרי תיקון הסעיף -> badge דחייה מתנקה
4) רק אז אפשר submit מחדש
5) Timeline/Audit ממשיכים להציג מי דחה + סיבה + זמן (בלי למחוק מי סימן/מי צילם)

זה תיקון לוגי קריטי, לא רק UX.
