2) ניקוי כפילויות/דמו + הגדרת Viewer כ-Read-Only (RBAC מלא)
דרישה UI:
- להסיר/לנקות כפתורי Demo כפולים/מבלבלים (ובעיקר כל מה שגורם לבלבול בניווט).
- Viewer = קריאה בלבד: אין יצירה/עריכה/אישור/סגירה, ואין כפתורי פעולה ב-UI (הכול Hidden/Disabled).

דרישה שרת (חובה):
- כל פעולה אסורה ל-Viewer חייבת להיחסם בשרת עם 403.
- אין לסמוך רק על הסתרה ב-UI.

Acceptance:
- Viewer לא רואה כפתורי Create/Edit/Approve/Close.
- Viewer שמנסה לקרוא endpoints של כתיבה מקבל 403 (להוכיח ב-Proof).

Tests:
- בדיקות RBAC שמכסות לפחות:
  - Viewer -> create issue/task = 403
  - Viewer -> update issue/task = 403
  - Viewer -> approve/reject/close = 403

3) Super Admin — מודול /admin/users
דרישה:
- להוסיף עמוד /admin/users (גישה: super_admin בלבד).
- יכולות:
  - חיפוש משתמשים (שם/טלפון/אימייל אם קיים/ID).
  - שינוי סטטוס משתמש: active/disabled (או מה שקיים אצלנו).
  - שינוי תפקידים (לפי הרשאות) + שיוך לפרויקטים.
  - צפייה בפרטי משתמש בסיסיים + רשימת פרויקטים משויכים + תפקיד בכל פרויקט.

Audits (חובה):
- כל פעולה במודול נרשמת audit event עם:
  - actor_id, target_user_id, פעולה, old/new, timestamp, reason אם רלוונטי.

Acceptance:
- משתמש שאינו super_admin מקבל 403 ב-/admin/users ובכל endpoints תואמים.
- כל שינוי שנעשה דרך המסך מופיע ב-Audit.

Tests:
- Access tests (super_admin=200, others=403).
- Audit written tests על שינוי סטטוס/תפקיד/שיוך פרויקט.

4) שינוי / שחזור מספר טלפון (Phone change & recovery) + Audits + Tests
דרישה:
- להוסיף זרימה מסודרת לשינוי מספר טלפון למשתמש (לפי הרשאות שקיימות אצלנו).
- להוסיף זרימה לשחזור/החלפת מספר במקרי צורך (Recovery) לפי המדיניות שלכם.
- כל שינוי מספר:
  - חייב Audit מלא: actor_id, target_user_id, old_phone, new_phone, reason, timestamp.
  - ולידציות: פורמט, uniqueness, וכשנדרש OTP/אימות (בהתאם למה שכבר קיים).

Acceptance:
- שינוי מספר עובד מקצה לקצה, ללא שבירת login/OTP.
- audit נכתב תמיד, כולל reason.

Tests:
- שינוי מספר תקין -> 200 + audit.
- שינוי מספר עם פורמט שגוי/כבר תפוס -> 400.
- הרשאות: מי שאסור לו לשנות -> 403.
