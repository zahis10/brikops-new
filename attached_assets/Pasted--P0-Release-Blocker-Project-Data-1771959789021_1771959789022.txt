נדרש טיפול מיידי + בדיקה מחמירה (P0 / Release Blocker) בנושא בידוד נתונים בין פרויקטים (Project Data Isolation)

יש לי חשד חמור לזליגת נתונים בין פרויקטים במסך "בקרת ביצוע – בחירת קומה".
אם אכן בניינים מפרויקט אחד מופיעים בתוך פרויקט אחר — זה כשל קריטי של בידוד נתונים, וברגע שיש לקוחות משלמים זה פוגע באמון ובערך המוצר כולו.

זה לא "באג UI קטן". זה אירוע חמור שדורש חקירה מלאה, הוכחת שורש תקלה, ותיקון עם בדיקות רגרסיה.

========================
דרישת עבודה (חובה)
========================

אני מבקש לבצע בדיקה מחמירה מקצה לקצה ולחזור אליי רק עם:
1) ממצאי Root Cause ברורים
2) תיקון בפועל (backend + frontend אם נדרש)
3) הוכחות (Proof) שהבעיה לא קיימת יותר
4) בדיקות רגרסיה שמונעות חזרה של התקלה

========================
החשד המדויק
========================

במסך "בחירת קומה" מופיעים בניינים שלא שייכים לפרויקט שנבחר (Project Scope Leak).

חשדות אפשריים:
- חסר סינון קשיח לפי project_id ב-backend
- שימוש ב-project context ישן (state / localStorage / session)
- fallback שמחזיר "all accessible buildings" במקום "buildings for selected project"
- cache / SW / stale state
- endpoint תקין אבל UI משתמש ב-state לא נכון (merged list / stale list)

========================
דרישות בדיקה מחמירה (לא לדלג)
========================

A. Backend – חובה
1) לבדוק את כל ה-endpoints שמעורבים במסך "בחירת קומה" ולוודא שיש סינון קשיח לפי:
   - org_id
   - project_id (חובה)
2) לוודא שאין שום query path / fallback שבו project_id חסר ואז מוחזרים כל הבניינים שהמשתמש "רשאי" לראות
3) להוסיף logging זמני (או debug trace) ברמת request עבור המסך:
   - user_id
   - role
   - org_id
   - selected_project_id (מה-request/URL)
   - effective_project_id (מה שנכנס בפועל לשאילתה)
   - building_ids / building_names שהוחזרו
4) לבדוק soft-delete / archive flags שלא יוצרים התנהגות עקיפה
5) לבדוק גם הרשאות super_admin וגם PM (ולא להניח ש-super_admin "יכול לראות הכל" ברמת בניינים מחוץ לפרויקט הנבחר במסך זה)

B. Frontend – חובה
1) לבדוק מאיפה המסך לוקח selected project:
   - URL params
   - route state
   - context
   - localStorage / sessionStorage
   - zustand/redux (אם קיים)
2) לוודא שאין persisted state ישן ממעבר פרויקט / החלפת משתמש / logout-login
3) לוודא שבמעבר פרויקט נעשה reset מלא לרשימות:
   - buildings
   - floors
   - filters
   - search
4) לוודא שה-UI לא ממזג תוצאות מכמה קריאות API
5) לבדוק cache behavior (כולל SW אם קיים)

C. Repro / Verification – חובה
לבצע בדיקה יזומה עם לפחות שני פרויקטים שונים, שלכל אחד רשימת בניינים שונה, למשל:
- Project A -> Buildings: A, B
- Project B -> Buildings: D, Test-QS-D

ולוודא:
- כניסה ל-Project A מציגה רק A,B
- כניסה ל-Project B מציגה רק D,Test-QS-D
- אין overlap
- רענון דף לא משנה את הרשימה
- logout/login לא משאיר רשימה ישנה
- מעבר משתמש (PM -> super_admin ולהפך) לא משאיר state קודם

========================
רמת חומרה
========================

יש לסווג את זה כ-SECURITY / TENANT-ISOLATION INCIDENT (לפחות ברמת P0/P1 פנימית),
כי מדובר בחשש להצגת נתונים של פרויקט לא נכון בתוך הקשר עבודה של פרויקט אחר.

עד לסגירת הבדיקה והתיקון:
- לא מאשר להסתמך על המסך הזה בפרודקשן
- לא מאשר להחשיב את המודול "Production Ready"

========================
תוצר שאני מצפה לקבל מכם (Proof Package)
========================

1) Root Cause Summary (מה בדיוק נשבר ולמה)
2) רשימת קבצים ששונו + מה תוקן בכל קובץ
3) לפני/אחרי של ה-query/filter logic (במילים, לא חייב קוד מלא)
4) צילום/לוג של requests+responses עבור:
   - Project A
   - Project B
   - super_admin
   - PM
5) הוכחת UI אחרי תיקון (screenshots / screen recording)
6) Regression tests:
   - unit/integration אם יש
   - לפחות test case אחד שמבטיח project_id scoping
7) הצהרה מפורשת:
   "אין החזרת בניינים למסך בחירת קומה ללא סינון קשיח לפי project_id"

========================
הנחיה חשובה
========================

לא להחזיר תשובה כללית כמו "נראה שתוקן" / "כנראה cache".
אני צריך בדיקה מחמירה עם הוכחה חד משמעית, כי במקרה של לקוחות משלמים אין מקום לספק בנושא בידוד נתונים.

אנא עצרו פיתוח פיצ'רים לא קשורים עד שהנושא הזה נסגר עם Proof מלא.
