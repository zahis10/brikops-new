מעולה, ה-QC flow הבסיסי עובד מצוין. בוא נתקדם למיני-ספרינט הבא: QC Notifications + Reopen Flow + Audit Visibility (ערך תפעולי גבוה, סיכון נמוך).

מטרות הספרינט:
1) התראות (In-App) על אירועי QC
2) פתיחה מחדש (Reopen) לשלב שאושר/נדחה
3) Audit + Timeline תצוגתי בסיסי
4) שקיפות מלאה למנהל פרויקט: מי צילם / מי סימן / מי הגיש / מי אישר + תאריך/שעה
5) הרשאות תצוגה (מי רואה מה) — PM/מאשרים רואים מלא, צוות רגיל רואה מצומצם

-----------------------------------
A) Notifications (In-App V1)
-----------------------------------
להוסיף התראות על האירועים:
- qc_submitted_for_review
- qc_approved
- qc_rejected
- qc_reopened

מי מקבל:
- submit_for_review -> PM + approvers של הפרויקט (אם יש)
- approve/reject -> מי שהגיש את השלב / יוצר השלב + PM
- reopen -> מי שקשור לשלב + PM

Payload מומלץ:
- notification_id
- project_id
- stage_id
- floor_id
- building_id (אם קיים)
- stage_code
- stage_label_he
- actor_id
- actor_name
- action (submitted/approved/rejected/reopened)
- reason / note (אם קיים)
- created_at
- read_at (nullable)

UI V1 (Notifications):
1) אייקון פעמון (Bell) בהדר/טופ-בר
   - באדג' עם מספר התראות לא נקראו
2) Dropdown/Drawer של התראות (V1 פשוט)
   - כל שורה: אייקון + טקסט + זמן + סטטוס נקרא/לא נקרא
   - דוגמה טקסט:
     - "שלב 'טיח' נשלח לאישור – קומה 3, בניין 2"
     - "שלב 'הכנות חשמל' נדחה – סיבה: חסרות תמונות"
     - "שלב 'אקוסטיקה' אושר"
3) פעולות:
   - לחיצה על התראה -> ניווט ל-Stage Detail הרלוונטי
   - סימון נקרא אוטומטי בפתיחה
   - (אם קל) כפתור "סמן הכל כנקרא"

הערת UX:
- לא צריך מרכז התראות מורכב עכשיו. Drawer/Dropdown בסיסי מספיק.

-----------------------------------
B) Reopen Flow (PM בלבד כברירת מחדל)
-----------------------------------
להוסיף endpoint:
- POST /api/qc/stages/:stageId/reopen  body: { reason }

Rules:
- רק PM של הפרויקט (implicit approver) יכול reopen ב-V1
- reason חובה
- אפשר reopen מ:
  - approved
  - rejected
- סטטוס יעד מומלץ:
  - reopened (מומלץ לשקיפות)
- אחרי reopen: שלב ניתן לעריכה + הגשה מחדש

Audit:
- qc_reopened עם:
  actor_id, actor_name, stage_id, from_status, to_status, reason, timestamp

UI V1 (Reopen):
1) ב-Stage Detail, אם status = approved/rejected והמשתמש הוא PM:
   - להציג כפתור: "פתח מחדש"
2) לחיצה על "פתח מחדש" פותחת Modal:
   - כותרת: "פתיחה מחדש של שלב"
   - שדה חובה: "סיבת פתיחה מחדש"
   - כפתורים:
     - "ביטול"
     - "פתח מחדש"
3) אחרי הצלחה:
   - toast: "השלב נפתח מחדש בהצלחה"
   - הסטטוס מתעדכן ל-reopened
   - נרשם אירוע ב-timeline + נוצרת התראה

שגיאות UX:
- אם הסטטוס לא מתאים ל-reopen -> להציג הודעה ברורה בעברית (לא generic error)

-----------------------------------
C) Stage Timeline (UI קטן אבל חשוב)
-----------------------------------
במסך Stage Detail להוסיף כרטיס/סקשן:
- כותרת: "היסטוריית פעולות"

להציג לפחות 5–10 אירועים אחרונים (מה-audit), בסדר כרונולוגי יורד (חדש למעלה)

אירועים לתצוגה ב-V1:
- נשלח לאישור
- נדחה (עם סיבה)
- אושר
- נפתח מחדש (עם סיבה)

מבנה שורה ב-Timeline (UI):
- אייקון פעולה (submit/approve/reject/reopen)
- כותרת קצרה:
  - "נשלח לאישור"
  - "אושר"
  - "נדחה"
  - "נפתח מחדש"
- שורת משנה:
  - "על ידי: {שם משתמש}" (לפי הרשאה)
  - תאריך + שעה (פורמט ישראלי)
- שורת סיבה/הערה (אם קיימת), למשל:
  - "סיבה: חסרות תמונות בסעיפים 2,4"

דוגמאות טקסט:
- "נשלח לאישור • זאהי שמי • 12/03/2026 14:21"
- "נדחה • מהנדס ביצוע • 12/03/2026 15:03"
- "סיבה: חסרה תמונה בסעיף 3"

הערת UX:
- לא צריך timeline מורכב גרפית. רשימה נקייה בכרטיס מספיקה.

-----------------------------------
D) Audit Visibility / Chain of Responsibility (חשוב מאוד)
-----------------------------------
מבחינת שליטה ניהולית באתר, חשוב לתעד ולהציג:
- מי צילם את התמונה (uploader)
- מי סימן את הסעיף תקין/לא תקין (item status actor)
- מי שלח את השלב לאישור (submitted_by)
- מי אישר/דחה (approved_by / rejected_by)
- מי פתח מחדש (reopened_by)
- תאריך + שעה לכל פעולה
- (אם יש) סיבה/הערה

דגשים חשובים:
1) "מי צילם" != "מי סימן"
   - יכול להיות שאיש צוות אחד צילם, ואחר סימן/הגיש. חייבים לשמור בנפרד.
2) אם יש כמה תמונות לסעיף:
   לכל תמונה לשמור:
   - uploaded_by_user_id
   - uploaded_by_name
   - uploaded_at
   - file_id / url
   - optional: source (mobile/web) אם קל להוסיף
3) לכל שינוי סטטוס סעיף:
   - changed_by_user_id
   - changed_by_name
   - changed_at
   - from_status / to_status
4) לכל החלטת stage:
   - submitted_by / approved_by / rejected_by / reopened_by
   - timestamp
   - reason/note (אם קיים)

-----------------------------------
E) מי רואה את המידע (הרשאות תצוגה) – V1
-----------------------------------
לא לפתוח הכל לכולם.

המלצה V1:
- PM:
  רואה audit מלא:
  - שמות מלאים
  - זמנים מלאים
  - timeline מלא
  - מי צילם / מי סימן / מי הגיש / מי אישר
- Approvers:
  רואים audit מלא רק בשלבים שבסקופ שלהם
- מבצעים/צוות רגיל:
  רואים תצוגה מצומצמת:
  - סטטוס נוכחי
  - הפעולות של עצמם
  - בלי לחשוף שמות מלאים של כל שאר המשתמשים
  - אפשר להציג "אושר" / "נדחה" בלי שם המאשר

הערה:
- PM הוא implicit approver כברירת מחדל (כמו שסיכמנו)
- לא להשתמש במונח reviewer ב-UI בכלל — רק "מאשר"

-----------------------------------
F) UI מפורט למסך Stage Detail (מה להוסיף בפועל)
-----------------------------------
במסך Stage Detail להוסיף/לסדר לפי הסדר הבא:

1) Header עליון של השלב
- שם השלב (למשל: "בלוקים – שורה ראשונה")
- סטטוס (Badge): draft / ready / pending_review / approved / rejected / reopened
- שורת מידע:
  - בניין / קומה / פרויקט

2) Action Bar (כפתורים)
לפי סטטוס והרשאות:
- draft/ready/reopened:
  - "שלח לאישור" (אם עומד בולידציות)
- pending_review:
  - אם PM או Approver:
    - "אשר"
    - "דחה"
  - אם לא מורשה:
    - טקסט: "ממתין לאישור"
- approved/rejected:
  - אם PM:
    - "פתח מחדש"
  - אם לא PM:
    - ללא כפתור reopen

3) Block הסבר למה אי אפשר לשלוח (אם submit disabled)
כבר יושם בזרימה הקודמת — לשמור גם כאן:
- "לא ניתן לשלוח לאישור — חסרות תמונות ב-X סעיפים, חסרות הערות ב-Y סעיפים."
- כפתור: "קח אותי לסעיף הראשון שחסר"

4) כרטיס "היסטוריית פעולות" (Timeline)
- מתחת לאזור הכפתורים או בתחתית המסך
- PM/Approvers: תצוגה מלאה
- צוות רגיל: תצוגה מצומצמת

5) כרטיס "מי עשה מה" (Audit Summary קטן בראש המסך) — מומלץ V1
להציג שורות קצרות (אם המידע קיים):
- נשלח לאישור ע"י: {שם} • {תאריך שעה}
- אושר ע"י: {שם} • {תאריך שעה}
- נדחה ע"י: {שם} • {תאריך שעה}
- נפתח מחדש ע"י: {שם} • {תאריך שעה}
(להציג רק מה שקיים)

-----------------------------------
G) UI מפורט ברמת סעיף (Item) / תמונות
-----------------------------------
בכל סעיף QC (בתוך השלב), להציג metadata קטן ליד התמונות/שינוי סטטוס:

1) ליד כל תמונה (ב-UI של gallery/item photos):
- "צולם ע"י {שם}" + שעה
דוגמה:
- "צולם ע"י יוסי כהן • 12:43"

2) ליד סטטוס הסעיף (תקין/לא תקין):
- "סומן ע"י {שם}" + שעה (לפי הרשאה)
או בתצוגה מצומצמת:
- "עודכן" + שעה

3) הרשאות תצוגה:
- PM/Approvers: שמות מלאים
- צוות רגיל: בלי שמות של אחרים (או רק של עצמו)

-----------------------------------
H) Validation / UX
-----------------------------------
- Reject דורש סיבה (קיים) ✅
- Reopen דורש סיבה
- אם סטטוס לא מתאים ל-reopen -> 400 ברור + toast בעברית
- הודעות עברית ברורות בכל הפעולות
- לא לחשוף שגיאות raw מהשרת למשתמש

טקסטים מוצעים:
- "השלב נפתח מחדש בהצלחה"
- "לא ניתן לפתוח מחדש את השלב במצב הנוכחי"
- "אין לך הרשאה לפתוח מחדש שלב זה"
- "ההתראה סומנה כנקראה"

-----------------------------------
I) Acceptance Criteria
-----------------------------------
1) submit_for_review יוצר התראה למי שצריך (PM + מאשרים)
2) approve יוצר התראה למגיש/יוצר + PM
3) reject יוצר התראה עם סיבה
4) reopen יוצר התראה
5) PM יכול reopen לשלב approved/rejected עם סיבה
6) אחרי reopen ניתן לערוך ולהגיש שוב
7) ב-Stage Detail מופיע timeline עם submit/reject/approve/reopen + timestamps + actor (לפי הרשאה)
8) PM רואה ב-Stage Detail:
   - מי צילם כל תמונה
   - מי סימן כל סעיף
   - מי שלח/אישר/דחה/פתח מחדש
   - תאריכים/שעות
9) משתמש רגיל לא רואה audit מלא של כל הצוות (רק מידע מצומצם לפי הרשאה)

-----------------------------------
J) חשוב – בלי לסבך עכשיו
-----------------------------------
לא צריך כרגע:
- מדדי KPI/Analytics מלאים
- תיעוד ידני של תחילת עבודה (טיח/ריצוף וכו')
- דוחות ביצוע מורכבים

כרגע המיקוד:
התראות + Reopen + Audit Visibility + UI ברור למנהל פרויקט
