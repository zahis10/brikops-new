==================================================
A) BUGFIX CRITICAL — "לא תקין" חייב הערה (באמת)
==================================================

הבעיה:
כרגע אפשר לסמן "לא תקין" גם בלי הערה בפועל (זה מתנהג כהמלצה, לא כחובה).

הדרישה:
- אם סעיף מסומן "לא תקין" (fail) -> הערה חובה
- לא רק ב-submit הסופי, אלא גם בשמירה של הסעיף עצמו (save item / autosave / patch)
- גם frontend וגם backend (defense in depth)

מה אני רוצה:
1. Frontend:
   - כשמסמנים "לא תקין" בלי הערה:
     - להציג inline error ברור ליד שדה הערה
     - disable לשמירה של הסעיף (אם יש save נפרד)
     - אם אין disable — לפחות חסימה עם הודעה ברורה
   - ניסוח ברור:
     "סומן לא תקין — חובה להוסיף הערה"

2. Backend:
   - כל endpoint ששומר/מעדכן item חייב לאכוף:
     - status=fail ללא note -> 422 ברור
   - error payload ברור ואחיד (כדי שה-frontend יציג הודעה טובה)

3. Regression:
   - pass = הערה אופציונלית (כמו V1 שהגדרנו)
   - fail = הערה חובה תמיד

Acceptance:
- fail בלי הערה נחסם ב-UI
- גם אם עוקפים UI — server מחזיר 422
- fail + הערה נשמר תקין

==================================================
B) BUGFIX CRITICAL — שמירת הערה לא מוחקת סטטוס/תמונה
==================================================

הבעיה:
המשתמש סימן "לא תקין" + הוסיף תמונה + שמר.
חזר אחר כך להוסיף הערה ושמר שוב — והסטטוס "לא תקין" נמחק.
זה באג אינטואיטיביות/merge חמור.

הדרישה:
- שמירת עדכון חלקי (למשל רק note) חייבת להיות PATCH merge אמיתי
- אסור ששמירה של שדה אחד תדרוס status/photos הקיימים
- אותו דבר גם להפך: שמירת סטטוס לא תמחוק note/photo

מה לבדוק/לתקן:
1. Frontend payload building:
   - לוודא שלא נשלח אובייקט item "חלקי" שגורם ל-replace
   - אם יש localChanges merge -> לוודא merge עמוק/מדויק
2. Backend update logic:
   - PATCH semantics אמיתי (set fields only), לא replace document/item
   - אם שדה לא נשלח -> לא מאפסים אותו
3. אחרי save:
   - ה-refresh מהשרת לא מוחק local state
   - אין race condition בין optimistic update ל-response reload

Acceptance:
- סטטוס fail נשאר fail אחרי שמירת הערה
- תמונה נשארת אחרי שמירת הערה
- הערה נשארת אחרי שינוי סטטוס
- אין "קפיצות" UI שמחזירות item למצב קודם

==================================================
C) PRODUCTIVITY SPRINT — יעילות עבודה למאשרים/PM (בלי שינוי לוגיקה)
==================================================

אחרי תיקון 2 הבאגים, נמשיך לשיפורי יעילות עבודה במסך ה-stage (מובייל-first):

1) Review Workbar (למאשר/PM במסך pending_review)
מטרות:
- לצמצם קליקים
- להבין מהר מה נשאר לבדוק
- ניווט מהיר בין סעיפים בעייתיים

להוסיף bar/section קומפקטי (רק למאשרים/PM) עם:
- כמה סעיפים ממתינים
- כמה "לא תקין"
- כמה נדחו ע"י מאשר
- כפתור "לסעיף הבא לבדיקה"
- כפתור "לסעיף הבא שנדחה"
- toggle "הצג רק סעיפים עם בעיה"

(Visual/UX + client-side navigation בלבד; בלי שינוי סטטוסים)

2) Quick navigation chips (בתוך stage)
chips מהירים:
- הכל
- לא סומן
- לא תקין
- נדחו
- חסרה תמונה
- חסרה הערה

הערה:
- הספירות חייבות להתעדכן בזמן אמת (כמו שכבר תיקנו קודם)
- ללא ריענון מסך

3) Rejection follow-up UX
כשמאשר דוחה סעיף:
- אחרי הצלחה, להציג CTA קטן:
  - "עבור לסעיף הבא"
  - "סנן רק נדחו"
- בלי לשבור את הזרימה ובלי רענון אגרסיבי

4) Submit/Approve action clarity
למאשר/PM:
- אם stage חסום בגלל סעיפים נדחים / חסרים:
  - להציג blocker ברור וקצר בראש אזור הפעולות
  - עם jump ישיר לסעיף הראשון שחוסם
- בלי טקסט עמוס

D) WhatsApp on Item Rejection (V1 קטן, אופציונלי/ידני)
==================================================

רעיון מעולה — אני רוצה להתחיל ממנו בזהירות, לא אוטומטי מלא עדיין.

הצורך:
כאשר מאשר דוחה סעיף, לפעמים צריך לעדכן מיידית את מי שסימן "תקין" / מי שהעלה את התמונה, עם התמונה וסיבת הדחייה.

המלצה שלי ל-V1:
Manual / opt-in action אחרי דחיית סעיף (לא אוטומטי)

כפתור קטן אחרי reject item:
"שלח עדכון ב-WhatsApp"

התנהגות V1:
- זמין רק למאשר/PM
- מופיע רק אחרי שיש rejection reason
- מציע לשלוח ל:
  1) מי שסימן את הסעיף (אם קיים טלפון)
  2) מי שהעלה את התמונה האחרונה (אם קיים טלפון)
  3) אם זה אותו אדם -> לא לשכפל
- תצוגה מקדימה של ההודעה לפני שליחה
- checkbox לבחירה למי לשלוח

תוכן ההודעה (V1):
- שם פרויקט
- בניין/קומה/שלב
- שם סעיף
- "הסעיף נדחה בבקרת QC"
- סיבת הדחייה
- קישור ישיר ל-stage/task
- (אם אפשר בקלות ובתשתית הקיימת) תמונה ראשית של הסעיף

חשוב:
- Audit event על שליחת הוואטסאפ:
  - מי שלח
  - למי נשלח
  - על איזה item
  - timestamp
  - message_type=qc_item_rejection_notice
- Rate-limit בסיסי + idempotency כדי למנוע שליחה כפולה מלחיצות כפולות
- אם אין טלפון / אין הרשאה / שליחה נכשלה -> הודעת שגיאה ברורה, בלי לשבור את פעולת הדחייה עצמה

לא לחסום את הספרינט על זה:
אם V1 ווטסאפ גדול מדי כרגע — לסיים קודם A+B+C ואז לפתוח אותו כספרינטון נפרד.
