ספרינט הבא — תיעוד + תקשורת (BrikOps) | עדיפות לפי הסדר

מטרה:
לשפר מהירות עבודה בשטח + תקשורת חוזרת אחרי דחייה + תיעוד מסודר לפני/אחרי עבודה,
בלי לשבור את היציבות שכבר יש (optimistic upload, batch upload, scroll anchor, draft preservation).

========================
P1 — WhatsApp ידני לדחייה (V1)
========================
דרישה:
להוסיף אפשרות ידנית לשליחת הודעת WhatsApp כאשר סעיף/שלב נדחה, אבל:
- בחירת נמען רק מתוך אנשי צוות בפרויקט (members)
- לא להקליד מספר טלפון ידנית
- לא לחשוף raw phone input למשתמש

התנהגות UX:
- CTA קטן/ברור באזור דחייה (stage/item rejection context), למשל:
  "שלח הודעת דחייה ב-WhatsApp"
- בלחיצה: modal / bottom sheet לבחירת נמען מתוך חברי הצוות בלבד
- להציג שם + תפקיד + (אם יש) אינדיקציה שיש מספר WhatsApp זמין
- אם אין לאף אחד מספר זמין -> הודעה ברורה + CTA לסגור
- אפשרות לערוך/לאשר נוסח לפני שליחה (V1 אפשר עם טקסט ברירת מחדל ו-edit קטן)

תוכן הודעה (V1):
- שם פרויקט / קומה / שלב / סעיף
- סטטוס: נדחה
- סיבת דחייה (אם קיימת)
- קישור ישיר לפריט/שלב במערכת
- תמונה ראשונה (אם קיימת) — אם כבר יש תשתית media template / payload, להשתמש בה
- אם אין תמונה, לשלוח טקסט בלבד

כללי הרשאות:
- רק משתמש מורשה (PM / approver / management לפי ההרשאות הקיימות למסך) יראה CTA
- קבלנים לא שולחים הודעת דחייה מהמסך הזה (אלא אם נחליט אחרת בעתיד)

Audit / Tracking:
- audit event חדש לדוגמה: qc_rejection_whatsapp_sent_manual
- payload: project_id, run_id/stage_id/item_id, recipient_user_id, recipient_role, has_image, result_status
- תיעוד גם במצב failure (שליחה נכשלה)

הערות טכניות:
- להשתמש בתשתית WhatsApp הקיימת אם אפשר (dry-run / real mode)
- לא להוסיף תלות חדשה אם לא חייב
- אם אין מספר טלפון לנמען -> לחסום שליחה לפני submit עם הודעה ברורה

Acceptance:
- בחירה מתוך צוות בלבד (אין phone input)
- שליחה מצליחה עם recipient valid
- חסימה כשאין מספר זמין
- audit נרשם
- אין שינוי בהרשאות עסקיות קיימות

========================
P2 — Media Guardian / Polish (הקשחת חוויית העלאות)
========================
דרישה:
פוליש + שמירה על אמינות בהעלאות מרובות (צילום/גלריה), בלי לשנות API עסקי.

לביצוע:
1) Upload queue indicator קטן ברמת הסעיף
   - "מעלה 2/5..." + spinner
   - מצב success/failure לכל קובץ (פשוט ועדין)

2) חסימת פעולות מסוכנות בזמן העלאה
   - אם יש upload פעיל בסעיף:
     - לא לאפס draft
     - אזהרה לפני יציאה/רענון (אם יש pending uploads / unsaved changes)
   - אם יש upload פעיל בכל המסך:
     - Disable עדין ל"שלח לאישור" או הודעת blocker: "ממתין לסיום העלאות"

3) Retry לקבצים שנכשלו (פר קובץ)
   - בלי למחוק את שאר הקבצים שהצליחו
   - שמירה על anchor/scroll באותו סעיף

4) שיפור מיקרו-קופי
   - "צלם עכשיו" / "צלם עוד" / "בחר תמונות"
   - שגיאות ברורות בעברית (גודל קובץ / פורמט לא נתמך / כשל רשת)

5) שמירה על יציבות קיימת
   - no scroll jump
   - no localChanges reset
   - optimistic preview + rollback חלקי נשמרים

Acceptance:
- העלאה מרובת תמונות נשארת יציבה
- אין קפיצה לסעיף אחר
- "שלח לאישור" נחסם כשיש upload פעיל
- retry עובד לקובץ ספציפי בלבד

========================
P3 — "פתיחת מלאכה" (תיעוד לפני כניסת קבלן)
========================
מטרה:
לתעד מצב קיים לפני עבודה כדי לאפשר השוואה ולדרוש החזרה לקדמותו.

MVP (מומלץ):
לא לבנות מודול חדש גדול — להתחיל כתבנית/סעיף מובנה:
- סעיף/שלב ראשון בשם "פתיחת מלאכה"
- חובה לצרף תמונות (לפחות תמונה אחת)
- הערה אופציונלית
- חותמת זמן + מי העלה
- אפשר להוסיף כמה תמונות (כבר נתמך)
- מוצג בראש ה-flow של הקומה/שלב הרלוונטי

אם אפשר בקלות:
- tag/flag לסעיף: pre_work_documentation = true
- UI badge קטן: "תיעוד לפני עבודה"

בעתיד (לא עכשיו):
- השוואת לפני/אחרי
- אישור קבלן על קבלת מצב
- דרישת סגירה מול "החזרה לקדמותו"

Acceptance:
- יש סעיף פתיחת מלאכה ברור
- תיעוד נשמר עם תמונות מרובות + timestamp + user
- מופיע בראש הרשימה/בתחילת ה-flow (לפי ההגדרה)

========================
P4 — וידאו קצר לתיעוד (אופציונלי, רק אם נשאר זמן)
========================
היקף זהיר (V0/V1):
- תמיכה בהעלאת וידאו קצר רק בסעיפי תיעוד מסוימים (לא בכל מקום)
- max duration / max size ברורים
- אם מורכב מדי כרגע — לדחות, לא לפגוע ביציבות התמונות

הערה:
עדיף לא להתחיל בזה לפני P1+P2+P3 כי זה יותר כבד (storage, preview, upload time, mobile UX).

========================
כללי זהב לספרינט הזה
========================
- לא לשבור את היציבות של upload הקיים
- לא לשנות הרשאות עסקיות בלי אישור מפורש
- לא להכניס raw role names או raw status codes ל-UI
- כל שינוי עם screenshots + הוכחת flow על מובייל (iPhone)

========================
Proof / Deliverables בסוף הספרינט
========================
1) Screenshots:
- בחירת נמען WhatsApp מתוך צוות
- שליחה מוצלחת / חסימה בלי מספר
- Upload queue / retry / blocker
- סעיף "פתיחת מלאכה" עם תמונות

2) Flow proof:
- reject item -> send manual WhatsApp
- batch upload photos -> no jump / no draft reset
- submit blocked while uploads running -> unlock after finish

3) Audit proof:
- דוגמת event של שליחת WhatsApp ידנית
- דוגמת event failure (אם יש)
