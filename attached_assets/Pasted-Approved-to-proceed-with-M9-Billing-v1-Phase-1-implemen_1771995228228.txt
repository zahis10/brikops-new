Mandatory Corrections / Guardrails (Phase 1)
1) No automatic backfill/seed on startup (critical)
- Do NOT run `backfill_project_org_ids()` automatically on app startup in production.
- Do NOT mutate billing data automatically on startup.
- Replace with:
  - dry-run report (counts + orphan/ambiguous list)
  - explicit apply migration (idempotent)
  - optional admin-only/manual execution path
- Startup may log health/readiness only, but must not silently rewrite ownership/billing mappings.

2) Remove/rename `subscriptions.plan_id` (org-level ambiguity)
- Billing v1 pricing is per-project, so `subscriptions.plan_id` at org level is semantically misleading.
- Preferred:
  - keep `project_billing.plan_id`
  - remove `subscriptions.plan_id`
- If an org-level plan selector is needed later, use a clearly named field like `pricing_profile_id` (not `plan_id`).

3) Strict endpoint scope split (required)
- `GET /api/billing/org/{org_id}`:
  - allowed: super_admin, org owner, org_admin, billing_admin
  - NOT allowed: PM/management_team unless they also hold org billing role
- `GET /api/billing/project/{project_id}`:
  - read-only allowed: project_manager, management_team, org billing roles, org owner, super_admin
  - blocked: contractor, viewer
- PM of one project must NOT get org-wide billing visibility by default.

4) Safe `projects.org_id` backfill rules (no unsafe auto-assignment)
- Do NOT auto-assign org_id using a “first org” lookup for multi-org users.
- If backfill source is ambiguous, mark project as orphan/ambiguous and report it (no silent assignment).
- Deliver dry-run evidence before apply.

5) Audit logging for all billing/org-role write paths (required)
Add audit events for:
- org role changes (`org_admin`, `billing_admin`)
- plan CRUD (super_admin)
- project_billing create/update
- contracted_units changes
- super_admin billing overrides affecting billing state
Logs must be concise and include actor, org_id, project_id (if applicable), action, before/after summary.

6) `observed_units` is monitoring only in Phase 1
- No billing charge recalculation on every apartment add/remove in Phase 1.
- Phase 1 = foundation + read-only UI + monitoring surfaces only.

7) Pricing snapshots in `project_billing` (recommended/strongly preferred)
Store snapshot fields so totals remain stable even if billing plan definitions change later:
- `project_fee_snapshot`
- `tier_fee_snapshot`
- `tier_code`
- optional `pricing_version`

8) Feature-flag discipline
- `BILLING_V1_ENABLED` default false
- New endpoints and new UI surfaces fully gated
- No behavior change to existing trial/paywall flows when flag is off
