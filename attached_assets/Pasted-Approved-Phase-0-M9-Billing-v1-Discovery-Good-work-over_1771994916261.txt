Phase 1 Corrections (Required Before / As Part of Implementation)
1) Pricing model correction (critical)
- Billing v1 is NOT per-unit usage pricing.
- Per-project pricing must be:
  Project Fee (fixed monthly per project) + Units Tier Fee (flat tier-based monthly component)
- Do NOT model `unit_tiers` as `price_per_unit` for v1.
- Replace with a flat tier fee model (e.g. tier code + monthly fee snapshot).

2) Separate contracted vs observed units (required)
Add distinct fields/concepts:
- `contracted_units` = billing basis (commercially agreed)
- `observed_units` = system-detected project size (monitoring/enforcement)
Do not use a single ambiguous `unit_count`.

3) Role model hygiene (important)
- Do not deepen the mixing between org-level and project-level roles.
- PM is a project-scoped role.
- Billing authority should be org-scoped (`owner`, `org_admin`, `billing_admin`).
- Backward compatibility is fine, but new billing logic must use explicit org-scoped billing permissions.

4) Multi-org safety fix (blocker for billing UI flows)
- `get_user_org()` returning “first org” is unsafe for multi-org users.
- Billing context resolution must be based on:
  - selected project -> project.org_id, OR
  - explicit selected organization context
- Do not rely on first-org lookup for billing screens/actions.

5) `projects.org_id` hardening (required early)
- Phase 1 must include a safe backfill + orphan-project report before any billing enforcement.
- Deliver:
  - count of projects missing org_id
  - backfill strategy/results
  - rollback-safe migration plan
- Do not enforce plan logic on projects lacking org_id.

6) Phase 2 wording/behavior correction
- “Auto-recalculate on unit changes” should mean observed_units monitoring + upgrade-needed detection,
  NOT direct usage-based billing changes on every apartment add/remove event.

7) Keep Organization vs Company separation strict
- Company remains contractor-firm catalog entity only.
- Billing payer and subscription ownership remain Organization-only.
