מעולה, בוא נמשיך באותו קו התקדמות (UX + הרשאות + שקיפות), בלי לשבור לוגיקה קיימת.

אני רוצה לדחוף עכשיו מיני-ספרינט ממוקד:
1) שקיפות אישורים לפי שלב (מי יכול לאשר ולמה)
2) קשיחות הרשאות למאשרי QC (קבלנים לא יכולים להיות מאשרים – לעולם)
3) ניקוי UI משמות תפקידים טכניים (management_team וכו׳)

חשוב:
- לשמור על PM כברירת מחדל כמאשר (implicit approver)
- בלי rewrite
- mobile-first
- backend permission enforcement קודם, UI אחר כך
- לא רק להסתיר UI — גם לחסום בשרת

==================================================
A) קשיחות הרשאות למאשרי QC — חסימת קבלנים (קריטי)
==================================================

דרישה עסקית:
- משתמשים בתפקיד contractor לא יכולים להיות מאשרי QC בכלל (לא להוספה, לא להשארה פעילה)
- זה לעולם לא אמור לקרות

מה לבצע:
1. Backend (הכרחי)
   - ב-POST של הוספת מאשר QC:
     - לחסום הוספה של משתמש עם role=contractor
     - להחזיר 400/422 ברור עם הודעה בעברית:
       "לא ניתן להגדיר קבלן כמאשר QC"
   - בשרת לא להסתמך על frontend בכלל

2. Backend Validation / Data hygiene (מומלץ מאוד)
   - אם קיימים כבר מאשרים היסטוריים שהם contractor (legacy data):
     אחת מהאפשרויות (מהיר + בטוח):
     - לא להציג אותם כמאשרים פעילים + להחזיר flag שהם invalid_role
     או
     - לסמן אותם כלא פעילים
   - לא למחוק אוטומטית בלי audit
   - אם מבוצע revoke/disable אוטומטי/ידני → audit event ברור

3. Frontend (בחירה/Modal הוספת מאשר)
   - ברשימת המשתמשים לבחירה:
     - לא להציג contractors בכלל
     או להציג disabled עם טקסט "לא ניתן לבחור קבלן כמאשר"
   - אם איכשהו מתקבלת שגיאת שרת — להציג toast ברור

Acceptance:
- Contractor לא מופיע כאופציה תקינה להוספה
- ניסיון API ישיר להוסיף contractor נחסם בשרת
- אין מצב שבו contractor יכול לאשר stage בפועל

==================================================
B) ניקוי UI משמות תפקידים טכניים (management_team וכו׳)
==================================================

בעיה:
- כרגע מוצגים למשתמש שמות פנימיים/טכניים כמו:
  management_team, contractor וכו׳
- זה מסגיר מבנה מערכת ומבלבל משתמשים

דרישה:
- לא להציג role codes פנימיים ב-UI
- להציג רק שמות ידידותיים בעברית

מה לבצע:
1. ליצור מיפוי תצוגה אחיד (frontend, ואם צריך גם backend response helper):
   - owner / pm / project_manager -> "מנהל פרויקט"
   - management_team -> "מנהל עבודה" / "צוות ניהול" (להחליט על ניסוח אחד קבוע)
   - contractor -> "קבלן"
   - super_admin -> "מנהל מערכת" (אם בכלל מוצג)
   - viewer -> "צפייה בלבד" (אם רלוונטי)

2. להחיל את המיפוי בכל המקומות הרלוונטיים:
   - Modal הוספת מאשר
   - רשימת מאשרי QC
   - כרטיסים/טיימליין/מטא-דאטה (אם role מוצג שם)
   - כל empty states / helper texts

3. fallback
   - אם role לא מוכר: להציג "משתמש" ולא את הקוד הגולמי

Acceptance:
- אין יותר הופעה של management_team / contractor / role codes ב-UI למשתמש
- כל התפקידים מוצגים בעברית אחידה וברורה

==================================================
C) שקיפות אישורים לפי שלב (Visibility) — מה שהצעתי, עכשיו למימוש
==================================================

מטרה:
למנוע בלבול של "מי אמור לאשר?", "למה אין לי כפתור אישור?", "מי יכול לאשר את השלב הזה?"

מה לבצע (V1):
1. Stage Detail / Stage Header — כרטיס/אזור שקיפות אישור
   להציג בצורה ברורה:
   - מי יכול לאשר את השלב כרגע:
     - מנהל פרויקט (ברירת מחדל)
     - מאשרים נוספים (אם קיימים)
   - האם האישור הוא:
     - לכל השלבים
     - או לשלבים ספציפיים (scope)
   - סטטוס משתמש נוכחי:
     - "יש לך הרשאת אישור לשלב זה"
     - או "אין לך הרשאת אישור לשלב זה"

2. כאשר stage ב-pending_review:
   - Approver/PM:
     - באנר ברור "ממתין לאישור שלך"
   - משתמש ללא הרשאת אישור:
     - באנר מידע "ממתין לאישור מנהל הפרויקט / מאשרי QC"
     - ואם הוא PM אבל אין setup נוסף — עדיין PM יכול לאשר (כי PM implicit)

3. הסבר "למה אין לי כפתור אישור?" (חשוב UX)
   - אם המשתמש אינו מורשה:
     - הודעה קצרה:
       "אין לך הרשאת אישור לשלב זה. רק מנהל הפרויקט או מאשרי QC שהוגדרו לשלב זה יכולים לאשר."
   - אם יש scope mismatch:
     - לציין שזה לא בתוך הסקופ שלו

4. קישור מהיר לניהול מאשרים (למורשים בלבד)
   - PM/owner/admin/super_admin רואים CTA:
     "ניהול מאשרי QC"
   - משתמשים אחרים לא רואים CTA ניהולי

5. Source of truth
   - לא לחשב רק ב-frontend
   - להשתמש ב-endpoint / my-approver-status (או endpoint משופר) שמחזיר:
     - is_approver
     - is_pm_implicit
     - mode (all/specific/none)
     - allowed_stage_ids / scopes (אם קיים)
     - reason_code (optional, לשימוש UI: not_configured / out_of_scope / role_not_allowed)

Acceptance:
- PM תמיד רואה שהוא מאשר ברירת מחדל
- משתמש רגיל מבין למה אין לו כפתור אישור
- מאשר QC רואה בבירור שיש לו הרשאה לשלב
- אין בלבול בין "ממתין לאישור" לבין "מי מאשר"

2. למנהל פרויקט:
   - להוסיף אפשרות הסרה ברורה (כבר קיימת) — לוודא שעובד חלק במובייל ✔
   - אם אין מאשרים נוספים:
     - להציג מסר ברור:
       "מנהל הפרויקט הוא המאשר כברירת מחדל"
       "ניתן להוסיף מאשרים נוספים לפי שלבים ספציפיים או לכל השלבים"

3. במודל הוספת מאשר:
   - לסנן contractors (מסעיף A)
   - להציג תפקידים בעברית בלבד (מסעיף B)

E) Audit / Security / Consistency (חובה)
==================================================

1. Audit events:
   - approver_added
   - approver_revoked
   - approver_add_blocked_invalid_role (optional but useful)
   - כל שינוי עם actor_id + actor_name + timestamp

2. Backend-first enforcement:
   - גם אם מישהו עוקף UI, קבלן לא יכול להיות approver ולא יכול לאשר stage

3. No logic regressions:
   - PM implicit approver נשאר עובד
   - management_team read-only נשאר עובד
   - non-authorized users לא יכולים לנהל מאשרים
