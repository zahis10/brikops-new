מעולה, ה-Proof נראה מצוין והספרינט הזה סגור ✅

בוא נעבור לספרינט UX הבא — בלי לגעת בלוגיקה / הרשאות / API contracts.
המטרה: להפוך את חוויית העבודה של PM/מאשרים/צוות ליותר ברורה, מהירה ונעימה במובייל.

ספרינט UX הבא (QC + Approvers) — Visual/UX Only

1) QC Approvers Tab / Modal polish (מובייל)
- לשפר את מודל "הוסף מאשר" במובייל:
  - גובה מקסימלי + גלילה פנימית נוחה
  - כפתור סגירה ברור וקבוע
  - שדה חיפוש דביק (sticky בתוך המודל אם אפשר)
  - CTA תחתון קבוע שלא נעלם
- ברשימת המשתמשים:
  - שם גדול וברור
  - role בעברית בשורה משנית (קטן יותר)
  - badge ברור אם המשתמש כבר מאשר
  - disabled ברור למי שלא ניתן לבחור
- Empty / no results states טובים יותר:
  - "לא נמצאו משתמשים"
  - "אין משתמשים מתאימים להוספה"

2) Approver list readability (בטאב מאשרי QC)
- לשפר קריאות של רשימת המאשרים:
  - היררכיה: שם > סוג הרשאה (כל השלבים/שלבים ספציפיים) > פירוט שלבים
  - spacing אחיד
  - אייקונים עקביים
- legacy invalid contractor warning:
  - להשאיר ברור, אבל פחות “צועק” אם יש כמה פריטים
  - להוסיף microcopy קצר שמסביר שזה record ישן ולא פעיל
- כפתור הסרה:
  - touch target גדול יותר
  - loading state בזמן הסרה
  - disable זמני למניעת double tap

3) Stage ApproverInfoCard polish (במסך stage)
- כרטיס "מי מאשר?" יהיה ברור יותר וקריא במובייל:
  - חלוקה ויזואלית בין:
    - מנהל פרויקט (ברירת מחדל)
    - מאשרים נוספים
  - reason/message יותר קצר ונקי
  - CTA לניהול מאשרים בולט יותר רק ל-PM
- למשתמש לא מורשה:
  - הודעה קצרה וברורה בלי עומס טקסט
  - לשמור counts בלי שמות (כמו עכשיו), אבל בתצוגה יותר ברורה

4) Pending Review approver banner polish
- הבאנר "ממתין לאישור" / "ממתין לאישור שלך":
  - להפוך ליותר קומפקטי במובייל
  - CTA approve/reject (אם יש) שלא שובר שורות בצורה מכוערת
  - helper text קצר יותר
- לוודא שאין overlap עם sticky/header/bottom actions בכל המצבים

5) Timeline visual polish (ללא שינוי נתונים)
- לשפר קריאות timeline:
  - ריווח בין אירועים
  - כותרת אירוע ברורה יותר
  - metadata (שם/זמן) בקונטרסט טוב
  - הפרדה יותר ברורה בין אירועי שלב לאירועי סעיף
- collapse/expand:
  - CTA ברור יותר ("הצג היסטוריה" / "הסתר היסטוריה")
  - אינדיקציה כמה אירועים יש

6) Microcopy & consistency sweep (QC screens)
- סריקה רוחבית של כל מסכי QC:
  - ניסוחים אחידים (אישור/דחייה/פתיחה מחדש/דחה סעיף)
  - אותו פורמט תאריך/שעה בכל מקום
  - אותה שפה לכפתורים (למשל "הוסף מאשר", "הסר מאשר", "לסעיף הראשון")
- לוודא שאין טקסטים טכניים/פנימיים גלויים בכלל

Acceptance (UX בלבד)
1. אין שינוי בלוגיקה/הרשאות/API
2. כל המסכים עובדים טוב במובייל (iPhone רוחב קטן)
3. אין חפיפות בין header / sticky / bottom actions / modals
4. מודל "הוסף מאשר" נוח לשימוש עם רשימות ארוכות
5. הקריאות של "מי מאשר?" ו-timeline טובה יותר משמעותית
6. אין raw role codes ואין microcopy לא עקבי
