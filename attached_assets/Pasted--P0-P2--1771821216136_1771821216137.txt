יש כמה בעיות/שיפורים דחופים אחרי העדכון האחרון. בבקשה לטפל לפי סדר עדיפויות (P0→P2). צירפתי צילומי מסך.

================
P0 (חוסם שימוש)
================

1) “מרכז ניהול” (דשבורד) קורס מיד
שגיאה: undefined is not an object (evaluating 'kpis.open_total') ב-ProjectDashboardPage.
נדרש:
- safe defaults + loading state:
  const kpis = data?.kpis ?? { open_total:0, in_progress_total:0, closed_total:0, pending_approval_total:0, overdue_total:0, ... }
- חוזה יציב FE/BE: ה-API תמיד מחזיר {kpis:{...}, pending_approvals:[], stuck_contractors:[], load_by_building:[], contractor_quality:[]} גם אם אין נתונים.
- אם API נכשל → UI מסודר (“לא הצלחנו לטעון נתונים, נסה לרענן”) ולא crash.
- להוסיף ErrorBoundary לדף הזה כדי שלא יפיל את כל האפליקציה.

2) תרגומים/חשיפת role codes (אבטחה + UX)
עדיין מופיע work_manager באנגלית לפעמים (וזה גם חושף DB codes).
נדרש:
- אין בשום מקום UI שמציג role code גולמי.
- כל role מוצג אך ורק דרך tRole().
- לבצע global search בתצוגה ל- work_manager / execution_engineer / וכו’ ולחסום הדפסה בסוגריים/טקסט.
- להשלים he.json לכל roles קיימים.

3) Role “מנהל עבודה” לא צריך להיות בחירה נפרדת ב-UI
מבחינת מוצר: “מנהל עבודה” הוא חלק מצוות ניהול ואין צורך בתפקיד נפרד למשתמש.
נדרש:
- להסיר “מנהל עבודה” מהבחירות ב-UI (גם מה-button-group).
- אם רוצים להשאיר בקוד/DB לצרכים פנימיים – להפוך ל-alias של צוות ניהול:
  * Display: תמיד להציג “צוות ניהול”
  * Permissions: זהה ל-management_team
  * Filtering/logic: treat work_manager as management_team
  * Migration: אין חובה עכשיו, אבל לא לחשוף את code בכלל.

================
P1 (שליטה + ניהול)
================

4) יש משתמשים שלא ניתן לערוך/Drawer לא נפתח / אין מידע
- ב-/admin/users: UserDrawer צריך להיפתח תמיד על “User”, גם אם אין project_memberships.
- אם אין memberships:
  UI מצב “משתמש לא משויך לפרויקטים” + פעולות Super Admin:
  a) שיוך לארגון (בחר ארגון)
  b) שיוך לפרויקט + role
  c) שינוי טלפון/אימייל אם זה קיים כבר
מטרה: שלא יהיו “משתמשים יתומים” במערכת.

5) שינוי תפקידים בין קבלן ↔ צוות ניהול
צריך לאפשר Owner/PM/SuperAdmin לשנות גם contractor → management_team ולהפך (גם אם לא נפוץ).
נדרש:
- להסיר פילטר שמונע מעברים “לא נפוצים”.
- להשאיר רק RBAC + Guards:
  * לא להסיר Owner
  * לא להשאיר פרויקט בלי PM
  * PM לא יכול להוריד Owner/להעיף Owner

6) העברת “חברה” (Organization) למשתמש
צריך יכולת לתקן טעויות שיוך ארגון (למשל “חשמלאי בדיקות” שויך לחברה לא נכונה).
נדרש (מינימום):
- Super Admin: פעולה “העבר לארגון אחר”
  * update organization_membership
  * cascade: להסיר/לנטרל memberships של המשתמש בפרויקטים של הארגון הישן (או לבחור להעביר רק פרויקט ספציפי)
  * audit מלא
(אם זה רגיש מדי ל-Owner – אז רק SA בשלב ראשון.)

7) RBAC לא ברור לצוות ניהול
נדרש:
- Badge קטן ב”מרכז ניהול” ובטאב צוות:
  “גישה מלאה (PM/Owner)” / “גישה חלקית (צוות ניהול)”
- ולוודא בפועל:
  * “ממתין לאישור שלי” רק PM/Owner
  * לצוות ניהול: KPI בסיסי + תקועים אצל קבלנים + עומס לפי מבנה (ללא פעולות אישור)

================
ספציפיקציה ברורה למודל Roles (כדי שלא יהיו ויכוחים)
================
Project Roles (אלו התפקידים היחידים שצריכים להיות ב-UI):
- project_manager (PM)
- management_team (כולל מנהל עבודה/מהנדס ביצוע – תת-כינוי בלבד)
- contractor
- viewer

Not in UI:
- work_manager (אם קיים ב-DB → alias של management_team, לא מוצג ולא ניתן לבחירה)
- execution_engineer / safety_* / site_manager וכו’:
  * אם צריך בעתיד – יהיה “Sub-role” בתוך management_team (שדה optional), לא project_role.
  * בשלב הזה: לא להציג כ-role עיקרי ולא להדליף codes.

Acceptance / Proof:
- “מרכז ניהול” נפתח בלי crash בפרויקט ריק ובפרויקט עם נתונים.
- אין role codes באנגלית בשום מקום.
- Owner/PM מצליח לשנות contractor↔management_team.
- ב-/admin/users ניתן לפתוח Drawer גם למשתמש בלי memberships ולהקצות אותו לארגון/פרויקט.
- SLA 24/48/72 עובד + “שלח תזכורת” עובד + “סגירות השבוע” מוצג.
