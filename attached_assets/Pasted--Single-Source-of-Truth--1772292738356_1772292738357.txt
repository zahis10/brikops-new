בוא נעשה כמו שצריך — איחוד בילינג לדף אחד (Single Source of Truth) בצורה בטוחה בלי לשבור קוד.

מטרה:
- דף חיוב מרכזי יחיד שבו עושים את כל פעולות הבילינג.
- כל שאר המקומות באפליקציה (כרטיס בפרויקט / באנר אדום וכו’) יהיו “קריאה בלבד” + לינק לדף המרכזי.
- אפס כפילויות, אפס drift בהרשאות, ואפס בלבול למשתמש.

החלטה מוצרית (חד-משמעית):
✅ מקור האמת היחיד = /billing/org/{org_id}
שם נמצאות *כל* הפעולות: בקשת תשלום, חידוש מראש, סימון שולם, ביטול, העלאת קבלה/צפייה.
בכל מקום אחר: סטטוס בלבד + כפתור ניווט אחד “מעבר לניהול חיוב”.

עיקרון בטיחות:
- לא מוחקים שום route.
- לא משנים API (אלא אם חייבים להוסיף שדה org_id/project->org).
- משנים UI בזהירות: הסתרת פעולות במקום הלא נכון + הוספת לינק למקום הנכון.
- בדיקות והרשאות נשארות קשוחות בבקאנד (billing_admin/org_admin/SA בלבד ל-mutations).

שינויים נדרשים (Frontend):

0) Helper אחד ליצירת URL אחיד (כדי שלא יהיו טעויות בעתיד):
- ליצור פונקציה אחת: getBillingHubUrl({ org_id, project_id? })
  שמחזירה:
  /billing/org/{org_id}#billing                     (בלי project_id)
  /billing/org/{org_id}?project_id={project_id}#billing  (עם project_id)
- להשתמש בה בכל מקום: ProjectBillingCard, הבאנר האדום, תפריטים/קישורים עתידיים.

1) ProjectBillingCard (טאב חיוב בתוך פרויקט):
- להסיר/להסתיר את כל כפתורי הפעולה (למשל "חידוש מראש", שליחת בקשה, העלאת קבלה וכו’).
- להשאיר:
  א) סטטוס בלבד: trial/paid_until/expired + read_only_reason (אם קיים)
  ב) CTA יחיד וברור: "מעבר לניהול חיוב" -> getBillingHubUrl(org_id, project_id)
- חשוב: אם חסר org_id בקונטקסט של הפרויקט, להוסיף אותו ל-payload של project (או להביא אותו מ-endpoint קיים) בצורה מינימלית בלי לשנות לוגיקה עסקית.

2) הבאנר האדום למעלה (נגמר רישיון / שדרוג):
- בלחיצה להעביר תמיד לדף המרכזי:
  getBillingHubUrl(org_id, current_project_id)
- אם אין project_id (המשתמש לא בתוך פרויקט): getBillingHubUrl(org_id)

3) OrgBillingPage (/billing/org/{org_id}):
- להוסיף תמיכה ב-query param project_id:
  - אם קיים project_id: לגלול/למקד את המשתמש לסקשן הרלוונטי (focus/scroll) + להציג בהקשר “פרויקט נוכחי”.
  - לא חובה לבנות UI מורכב של בחירה, אבל כן לתת חווית “הגעת למקום הנכון”.
- להשאיר את ההגיון שכבר תוקן:
  - "בקש שדרוג מבעלים" רק ל-PM שאין לו סמכות חיוב.
  - Contractors/Site managers/Viewers לא רואים UI של בקשת תשלום.

שינויים נדרשים (Backend) — רק אם חסר, אחרת לא נוגעים:
- להשאיר את כל ה-mutation endpoints עם בדיקת הרשאות מפורשת:
  org_admin / billing_admin / SA בלבד.
- owners בלי role מפורש נשארים חסומים (403) — זה נכון.
- אם צריך: להוסיף בפרויקט שדה org_id בתגובה של project-get (מינימום שינוי, בלי שינוי הרשאות).

תוכנית ביצוע בטוחה (בלי לשבור):
Phase 1 (Safe UI Unification):
- להוסיף helper URL + להחליף את כפתורי הפעולה בכרטיס הפרויקט לסטטוס+לינק בלבד.
- לעדכן את הבאנר האדום שיפנה לדף המרכזי.
- לא לשנות API ולא לשנות DB.

Phase 2 (Polish + Guardrails):
- OrgBillingPage: focus/scroll לפי project_id.
- להוסיף בדיקות יחידה/helper כדי למנוע החזרת URL שגוי.
- לוודא שאין כפתורי פעולה בשום מקום נוסף (חיפוש רכיבים/טקסטים: "חידוש מראש", "שלח בקשת תשלום").

בדיקות חובה (High/Sensitive – Billing):
A) Proof (לפני/אחרי) + Regression:
- צילום מסך לפני/אחרי של ProjectBillingCard:
  - לפני: הופיע "חידוש מראש"
  - אחרי: אין פעולות, רק סטטוס + "מעבר לניהול חיוב"
- צילום מסך של הבאנר האדום שמוביל ל-/billing/org/... (כולל URL בפועל).

B) RBAC / Guardrails:
- FE: אין פעולות בילינג בטאב הפרויקט לשום תפקיד.
- BE: כל mutation endpoints מחזירים 403 למי שאין org_admin/billing_admin/SA.

C) Smoke tests (מינימום):
1) כניסה לפרויקט -> טאב חיוב -> אין פעולות -> לינק לדף מרכזי עובד.
2) מהבאנר האדום -> מגיע לדף מרכזי (עם project_id אם קיים).
3) billing_admin מבצע פעולה אחת בדף המרכזי (mark-paid או upload receipt) -> 200.
4) owner ללא billing role מנסה אותה פעולה -> 403.

D) Deliverables (לא סוגרים בלי זה):
- 6 צילומי מסך:
  1) Contractor/Viewer בפרויקט: סטטוס+לינק בלבד
  2) Site manager בפרויקט: סטטוס+לינק בלבד
  3) PM ללא billing: בפרויקט סטטוס+לינק; בדף המרכזי רואה “בקש שדרוג מבעלים”
  4) Org owner ללא billing role: בפרויקט סטטוס+לינק; mutations 403
  5) Billing admin: בדף המרכזי פעולות עובדות
  6) SA: שליטה מלאה
- צירוף תוצאות E2E/console clean, ותקציר “Root cause + Fix + Regression”.

הערה חשובה:
המטרה היא לא “לתקן עוד מקום”, אלא למנוע בכלל מצב שיש עוד מקום. אחרי האיחוד הזה, כל שינוי בבילינג נעשה במקום אחד בלבד.
