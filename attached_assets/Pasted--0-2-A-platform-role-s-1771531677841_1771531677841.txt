מעולה. הנה “חוקי תפקידים” מסודרים כדי שלא נעשה סלט.

0) עקרון על
- יש 2 שכבות:
  A) platform_role (super_admin / none) — זה רק שלי, נקבע רק דרך SUPER_ADMIN_PHONE. לא ניתן לשינוי מה-UI ולא דרך API רגיל.
  B) project_role — תפקידי פרויקט בלבד. פה כן ניתן לשינוי ע"י super_admin.

1) סט תפקידי Project (הסופיים)
- project_manager (PM) — תפקיד לקוח מקסימלי. מנהל את הפרויקט.
- management_team — צוות ניהולי (מנהל עבודה/מפקח וכו’). יכול לעבוד מול ליקויים, לא מנהל מערכת.
- contractor — קבלן/ביצוע. רואה רק את המשימות שלו/הטרייד שלו, מעלה הוכחות, לא מנהל.
- viewer — צפייה בלבד. אין כתיבה בכלל (רק קריאה).

2) הרשאות כלליות לכל תפקיד (בקצרה)
- PM:
  - יכול ליצור/לערוך ליקויים, להגדיר דחיפות, לשייך קבלנים, לאשר/לדחות הוכחות, לראות הכל.
  - יכול לראות תוכניות, לראות timeline/attachments מלאים.
- management_team:
  - יכול לראות הכל בפרויקט.
  - יכול לעדכן סטטוסים/הערות/שדות “תפעוליים” (אם החלטנו), אבל לא משנה חיוב/משתמשים.
  - לא יכול לגשת ל-/admin/*.
- contractor:
  - רואה רק את המשימות שהוקצו לו + רק ברובריקה של ה-trade שלו.
  - יכול להעלות הוכחות (תמונות) רק כשהוא assignee ורק כשהקטגוריה תואמת trade.
  - לא רואה הוכחות של קבלן קודם.
- viewer:
  - רואה פרויקטים/ליקויים/תוכניות/טיימליין לקריאה בלבד.
  - כל פעולה שמשנה נתונים: חסומה (POST/PUT/PATCH/DELETE => 402/403 לפי מנגנון הבילינג/הרשאות).

3) חוקי שינוי תפקידים (Role Transitions) — מה מותר בסופר אדמין
כל שינוי role מחייב:
- note חובה (סיבה), נשמר ב-audit
- שמירת before/after מלא (project_id, user_id, old_role, new_role, actor)

Allowed transitions (למניעת טעויות):
A) viewer ↔ management_team ↔ project_manager
- מותר לשדרג/להוריד בין שלושתם.
B) contractor ↔ viewer
- מותר “להוריד” contractor ל-viewer (למשל סיום עבודה/רק צפייה).
- שדרוג viewer ל-contractor מותר רק אם יש company_id + trade_key תקין (אחרת 400 עם הודעה).
C) contractor → management_team / project_manager
- ברירת מחדל: חסום (409) כדי לא להפוך קבלן לצוות/PM בטעות.
- אם אתה רוצה לאפשר בעתיד: יהיה toggle “Override” רק ל-super_admin עם אזהרה + note חובה (אבל כרגע עדיף לחסום).

4) כללי “בטיחות” בשינוי תפקיד
- לא ניתן להשאיר פרויקט בלי אף PM:
  - אם מנסים להוריד PM האחרון בפרויקט — לחסום 409 “חייב להיות לפחות מנהל פרויקט אחד”.
- שינוי role שמבטל גישה לכתיבה (למשל PM→viewer):
  - עדיין מאפשר login וקריאה, אבל חוסם כתיבה מיידית.
- אם מורידים משתמש ל-viewer/contractor:
  - UI צריך לרענן routing כדי שלא יישאר על /control אם אין הרשאה.

5) מה צריך לבנות בפועל (סעיף 2 אצלך)
Backend:
- Endpoint לשינוי role ברמת project_membership:
  PUT /admin/users/{user_id}/projects/{project_id}/role
  body: { "new_role": "...", "note": "..." }
  - super_admin בלבד
  - ולידציות לפי החוקים למעלה
  - audit event: admin_change_role (כולל old/new)

Frontend:
- ב-AdminUsers → בפרטי משתמש:
  - טבלה/רשימה “חברויות בפרויקטים”
  - ליד כל פרויקט: dropdown role + כפתור “שמור” + שדה “סיבה” חובה
  - להציג הודעות שגיאה ברורות (למשל “אי אפשר להוריד את ה-PM האחרון”).

6) לגבי הצופה (viewer) — למה צריך אותו בכלל
- שימוש אמיתי בשטח: דייר/יזם/מפקח/בודק/לקוח שרוצה לראות סטטוסים ותמונות בלי להרוס נתונים.
- אם נחליט לבטל אותו: אפשר — אבל כרגע הוא עוזר המון למכירה כי זה “גישה לצפייה” בלי עוד רישוי.

אישור ממני:
- תתחיל ב-1,3,4 (כרטיסי אדמין + תרגום + RTL טלפון)
- ואז תעבור ל-2 לפי החוקים למעלה בדיוק.
